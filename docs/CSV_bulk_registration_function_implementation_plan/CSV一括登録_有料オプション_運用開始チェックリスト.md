## 環境・設定（申し込みフォームを使う場合は最優先で実施）

**CSV代行申し込みフォーム**で客から申し込みを受け付ける場合、**運営（あなた）にメールが届く設定**をしていないと、客が申し込んでも依頼内容を把握できず仕事ができない。以下を**運用開始前に必ず**設定すること。

| 項目 | 説明 | 未設定時の影響 |
|------|------|----------------|
| **`ADMIN_NOTIFICATION_EMAIL`** | 申し込みメールの**送信先**（届けて確認したいメールアドレス）。 | 申し込みAPIが 503 を返す。メールは1通も届かない。 |
| **`BREVO_API_KEY`** | Brevo でメール送信するためのAPIキー。 | 送信処理でエラーになり、メールは届かない。 |

- **`ADMIN_NOTIFICATION_EMAIL`**: あなたが受け取れるメールアドレスなら**何でもよい**（Gmail・会社用・フリーメール等）。Brevo の管理画面で「受信先」を登録する必要は**ない**。
- **代行履行のみ**（申し込みフォームを使わない・メールで直接受注する）場合は上記は不要。フォームを有効にするなら**未設定のまま運用開始しないこと**。

---

### ADMIN_NOTIFICATION_EMAIL について

- **あなたが受け取れるメールアドレスなら何でもよい。** Gmail、Outlook、会社用アドレス、フリーメールなど、届け先として使えるアドレスを1つ指定するだけ。
- Brevo の管理画面で「受信先」を登録する必要は**ない**。アプリ側の環境変数に書いたアドレスあてに、Brevo が送信する。

---

### Brevo で必要な設定（運営が手動で行う）

**Brevo 側で必要なのは次の2つだけ。**

1. **API キーを取得する**
   - [Brevo にログイン](https://app.brevo.com/) → 右上のプロフィールアイコン → **Settings** → 左メニュー **SMTP & API** → **API Keys**（または [API Keys ページ直リンク](https://app.brevo.com/settings/keys/api)）。
   - **Generate a new API key** でキーを発行する（権限は「Send transactional emails」等、送信できればよい）。
   - 表示されたキー（例: `xkeysib-xxxxxxxx...`）をコピーし、下記「どこで設定するか」のとおり **`BREVO_API_KEY`** に貼り付ける。**Brevo の画面では「受信先」や「申し込み用の設定」は作らなくてよい。**

2. **送信元メール（ドメイン）の認証**
   - アプリは **送信元** として `BREVO_SENDER_EMAIL`（既定値: `noreply@yadopera.com`）を使う。Brevo では「このドメインから送ってよい」と認証しないと送信できない場合がある。
   - Brevo 管理画面: **Senders & IP**（または **Domains**）で、送信に使うメールアドレスまたはドメインを**認証**する。認証方法は Brevo の案内（DNS の TXT レコード追加や、送信テスト用の確認メールなど）に従う。
   - 既に認証済みの送信元（例: 本番で他機能のメール送信に使っている `noreply@yadopera.com`）があれば、**追加の設定は不要**。初めて Brevo を使う場合や、送信元を変えた場合だけ上記を実施する。

**Brevo で「申し込みフォーム用」のテンプレートや受信先リストを作る必要はない。** 申し込み内容はアプリが API で送信し、届け先は `ADMIN_NOTIFICATION_EMAIL` の値だけ。

---

### どこで設定するか（設定する場所）

**設定する変数は同じ2つ（`BREVO_API_KEY` と `ADMIN_NOTIFICATION_EMAIL`）だが、書く場所は「どの環境で動かすか」で変わる。**

| どこでアプリを動かすか | 設定する場所 | 備考 |
|------------------------|--------------|------|
| ローカル（自分のPC・Docker）だけ | **backend/.env** のみ | Render には設定不要 |
| ステージング・本番（Render）だけ | **Render の Environment** のみ | 手元の .env は不要 |
| ローカルと Render の両方で使う | **両方** | 各環境ごとに同じ変数名で値を入れる |

**アプリの設定は、上記のとおり「使う環境」に応じて次のどちらか（または両方）で行う。Brevo の画面では「受信先」や「申し込み用」の設定はしない。**

#### ローカル（Docker でテストする場合）

| 設定するもの | 設定する場所（ファイルパス） |
|--------------|------------------------------|
| すべて | **`backend/.env`**（プロジェクトルートの `backend` フォルダの中の `.env`） |

- **`backend/.env` が無い場合**: 同じ `backend` フォルダにある `backend/.env.example` をコピーし、ファイル名を **`.env`** にしたものを `backend` フォルダに置く。その後、その `.env` を開いて編集する。
- **編集する行の例**（`=` の右に値を書く。引用符は不要）:
  - `BREVO_API_KEY=xkeysib-xxxxxxxxxxxxxxxx`
  - `ADMIN_NOTIFICATION_EMAIL=your@example.com`
- 保存したら、**バックエンドを再起動**する（例: プロジェクトルートで `docker compose down` のあと `docker compose up -d --build`）。再起動後に申し込みフォームから送ると、指定したメールアドレスに届く。

#### ステージング・本番（Render.com で運用する場合）

| 設定するもの | 設定する場所（画面の場所） |
|--------------|----------------------------|
| すべて | **Render のバックエンドサービスの「Environment」** |

- [Render Dashboard](https://dashboard.render.com/) にログインする。
- **Your services** の一覧から、**バックエンド**（API サーバー）のサービス名をクリックして開く。
- 左側のメニューで **Environment** をクリックする。
- **Environment Variables** の一覧が表示される。**Add Environment Variable** または **Key** / **Value** の入力欄で、次の2つを追加する。
  - **Key**: `BREVO_API_KEY`  
    **Value**: Brevo で取得した API キー（例: `xkeysib-xxxxxxxx...`）
  - **Key**: `ADMIN_NOTIFICATION_EMAIL`  
    **Value**: 申し込みメールを受け取りたいメールアドレス（例: `your@example.com`）
- **Save Changes** を押す。多くの場合、保存すると自動で再デプロイされる。されない場合は **Manual Deploy** で再デプロイする。デプロイが終われば、申し込みフォームから送った内容が指定したアドレスに届く。

変数名の一覧は **`backend/.env.example`** を参照。ローカルでの送信テスト手順は [LOCAL_BROWSER_TEST.md](../LOCAL_BROWSER_TEST.md) の「5a. CSV一括登録 代行申し込み」を参照。

## 1. 受注〜履行の流れ（チェックリスト）

### 1.1 受付・確認

- [ ] 施設からの申し込み（メール／フォーム等）を受信した
- [ ] **施設名・プラン（Standard/Premium）・希望登録件数・希望言語・FAQ内容（原稿またはファイル）**を確認した
- [ ] 料金表に照らして**見積もり**を出し、施設に送付した（§4.1 または plan2.txt の料金表を参照）
- [ ] 支払い方法・請求の手続きを案内した（現時点は手動見積・請求。Stripe連携は別フェーズ）

### 1.2 原稿・データの準備

- [ ] 施設から受け取ったFAQ内容を、[FAQ_CSV_format_guide.md](files/FAQ_CSV_format_guide.md) に準拠した**CSV**に整えた
- [ ] 必須カラム（`category`, `language_ja_question`, `language_ja_answer`）および必要に応じて他言語カラムを用意した
- [ ] ファイルは **UTF-8（BOM付き推奨）**、**10MB以内**であることを確認した
- [ ] 対象施設の**施設ID（facility_id）**を開発者管理ページまたはDBで確認した

### 1.3 履行（方法C: スクリプトで一括登録）

- [ ] **方法C**（推奨）: スクリプトで履行する場合  
  - [ ] CSVをサーバー（またはDocker内）から読めるパスに配置した  
  - [ ] **Docker で実行する場合**（推奨）:
    ```bash
    cd backend
    docker compose cp ./faq.csv backend:/tmp/faq.csv
    docker compose exec backend python scripts/bulk_upload_faq_csv_for_facility.py --facility-id <施設ID> --csv /tmp/faq.csv
    ```
  - [ ] **ドライラン**で事前確認する場合: 上記の末尾に `--dry-run` を付与  
  - [ ] 実行後、成功メッセージ（登録件数・処理時間）を確認した  
- [ ] **方法B**（代替）: 施設側のJWTでAPIを叩く場合は、[開発者によるCSV一括登録代行_手順と方法_20260212.md](開発者によるCSV一括登録代行_手順と方法_20260212.md) の「方法B」に従った

### 1.4 納品・フォロー

- [ ] 施設に「登録が完了したこと」をメールで通知した
- [ ] 初回は**表示・動作チェック**（料金に含まれる）の実施を案内した
- [ ] 請求が未済の場合は、請求・入金確認を行った

## 3. スクリプト実行時のエラー例と対処

| 現象 | 想定原因 | 対処 |
|------|----------|------|
| 「施設が見つかりません」 | `--facility-id` が誤り、または未登録 | 施設IDをDBまたは管理画面で再確認 |
| 「Standard/Premiumプランのみ利用可能」 | 対象施設が Free/Mini/Small | プラン変更を案内するか、代行対象外と説明 |
| 「施設に紐づくユーザーが1人もいません」 | 施設にユーザーが未作成 | 施設にユーザー登録を依頼してから再実行 |
| 「CSVに有効なデータ行がありません」 | 空またはフォーマット不正 | [FAQ_CSV_format_guide.md](files/FAQ_CSV_format_guide.md) に沿ってCSVを修正 |
| 「〜件までしか登録できません」 | 既存FAQ数＋CSV行数がプラン上限超過 | 件数を減らすか、プラン変更を案内 |
| 言語数制限超過 | Standard等の言語数制限を超える言語がCSVに含まれる | 言語を減らすか、Premiumを案内 |

## 4. 運用開始の判定

- **申し込みフォーム**を利用する場合: 上記「環境・設定」の **`ADMIN_NOTIFICATION_EMAIL` と `BREVO_API_KEY` を設定し、実際に申し込みメールが届くことを確認したうえで**運用開始とする。未設定のままでは運用開始不可（客の申し込みが届かず意味がない）。
- 上記チェックリストに従い、**受注→見積→CSV準備→スクリプト（または方法B）で履行→納品通知**まで一連の流れを一度実施できる状態になっていれば、運用開始可能とする。
- 請求は Stripe 実装後に連携する前提。現時点では手動見積・請求で運用開始可能。
