# 開発者によるCSV一括登録代行 — 調査・手順（有料代行サービス履行のための運用）

**作成日**: 2026年2月12日  
**最終更新日**: 2026年3月21日  
**目的**: **有料のFAQ作成・アップロード代行**を契約したユーザーに対して、運営・開発者がCSV作成**と**アップロード（管理画面への登録作業）を**履行する**ための手順と方法をまとめる。

---

## 0. 本ドキュメントの位置づけ — plan2.txt からの引き継ぎ（ビジネス上重要）

**本ドキュメントは、CSV一括登録 実装計画書 v2.0（`CSV_bulk_registration_function_implementation_plan2.txt`）の §1 概要・目的（ビジネス要件・料金体系・料金に含まれる／含まれない内容）および §3 機能仕様（対象ユーザー・対象プラン・登録件数制限）を正として引き継ぎ、代行履行の運用手順をまとめたものである。plan2.txt に詳細に記載して保存している内容をスルーせず、すべて運用に活かす。**

---

### 0.1 引き継ぎ元：plan2.txt §1 概要・目的（そのまま記載）

**【目的】**（plan2.txt より）  
宿泊施設管理者**または開発者**がCSVファイルをアップロードすることにより、FAQを自動で一括登録（上書き/追加）できる機能を実装する。  
→ **代行**は「開発者（運営）が依頼を受けて登録する」ケースである。

**【ビジネス要件】**（plan2.txt より）
- 宿泊施設管理者が**自身で実行する場合: 無料**
- **開発者へFAQ登録依頼する場合: 有料**
- **StandardプランおよびPremiumプランのみの機能**（Free/Mini/Smallでは利用不可）
- **プラン料金とは別料金が発生**（Stripe実装後に継承）

**【料金体系】**（plan2.txt より — オプション料金・代行料金の正）
| プラン | 件数 | 料金 |
|--------|------|------|
| **Standard**（最大100件） | 〜30件 | ¥12,000 |
| | 〜60件 | ¥22,000 |
| | 〜100件 | ¥32,000 |
| **Premium**（件数無制限） | 〜50件 | ¥25,000 |
| | 〜100件 | ¥45,000 |
| | 〜200件 | ¥68,000 |
| | 200件超 | 個別見積（¥90,000〜） |

**【料金に含まれる内容】**（plan2.txt より — そのまま）
- 日本語FAQの質問文・回答文の自然さ／文法確認
- 軽微な表現修正（意味変更なし）
- AI翻訳＋人による品質確認
- **管理画面への登録作業**
- 表示・動作チェック（初回）

**【料金に含まれない内容】**（plan2.txt より — そのまま）
- FAQの新規企画・構成設計
- 内容の事実確認（運営保証なし）
- 大幅な書き直し・再編集
- 法的表現監修・翻訳証明

**【3.2 対象ユーザー】**（plan2.txt §3 より）
- 宿泊施設管理者（自身で登録: 無料）
- **開発者（依頼を受けて登録: 有料）** ← 本ドキュメントはこのケースの履行手順をまとめる

**【3.3 対象プラン】**（plan2.txt §3 より）  
Standard、Premium。Free/Mini/Smallプランでは利用不可（制限メッセージを表示）。

**【3.5 登録件数制限】**（plan2.txt §3 より）  
Standard: 最大100件まで（既存FAQ数＋CSV行数が100を超えるとエラー）。Premium: 無制限。

### 0.2 代行契約の履行＝運営がアップロードまで行う（ビジネスチャンスを捨てない）

ユーザーが**代行を契約した**場合、運営側が「CSV作成」**と**「アップロード（管理画面への登録作業）」の**両方**を行うのが契約履行である。  
**「CSVだけ渡してユーザーに自分でアップロードしてもらう」は代行契約を履行しておらず、有料サービスを提供していない＝ビジネスチャンスを放棄する行為である。**  
本ドキュメントは、代行契約を結んだユーザーに対して、運営・開発者が**アップロードを確実に履行するための手段**をまとめたものである。

### 0.3 代行を利用しない場合（ユーザー自身で登録する場合）

- ユーザーが**代行を契約していない**場合: ユーザー自身が管理画面からCSVをアップロードする（無料）。その場合は、運営側はCSVテンプレートの提供やマニュアル（4.8 CSV一括登録）の案内で支援する。  
- その「ユーザー自身でアップロード」の案内は、**代行契約時には使わない**。

## 1. 現状の技術仕様（API・認証）

- **CSV一括登録API**: `POST /api/v1/admin/faqs/bulk-upload`
  - **認証**: JWT必須（`get_current_user`）。**ログイン中のユーザーが所属する施設**にのみFAQが登録される。
  - **対象プラン**: Standard または Premium のみ（それ以外は 403）。
  - **モード**: 追加モード（add）のみ。既存の intent_key と重複する行があると 400 で全体ロールバック。
- **管理画面**: 施設管理者がログインし、FAQ管理画面から「CSV一括登録」でファイルをアップロードする想定。
- **開発者向け**: **`POST /api/v1/developer/facilities/{facility_id}/faqs/bulk-upload`**（`type=developer` の JWT 必須）。施設 ID をパスで指定し、multipart で CSV を送る。**代行履行の第一選択（方法D）**。

したがって、代行履行には **方法D（開発者管理画面／開発者 API）** を標準とし、必要に応じて **方法C（スクリプト）・方法B（施設管理者 JWT）** を用いる。

---

## 2. 代行契約を履行するための手段

### 方法D: 開発者管理画面から CSV をアップロードする（代行履行の主推奨・2026-03-21〜）

**位置づけ**: **有料代行を履行する場合の第一選択**。ブラウザで **ステージング／本番のフロント URL** を確認したうえで、開発者パスワードでログインし、施設一覧から対象施設を特定して **「CSV一括登録（FAQ）」** を実行する。multipart でサーバが受け取るため、**Render Shell やコンテナ内パスを扱わなくてよい**。

**手順（概要）**

1. **環境の確認** — ブラウザのアドレスバーでステージング（例: `yadopera-frontend-staging.onrender.com`）か本番かを確認する。
2. **開発者ログイン** — `DEVELOPER_PASSWORD`（各環境のシークレット）。
3. **施設の特定** — 申し込みメールのログインメール・施設名と、一覧の **施設ID・プラン**を突合。Standard / Premium のみ（Small 等は拒否）。
4. **CSV 投入** — 「CSV一括登録（FAQ）」→ ファイル選択 → 確認チェック → 実行。
5. **（推奨）** `GET /__debug_env` 等で **デプロイコミット**が想定と一致するか確認してから本番で同様に実行。

**API（curl・自動化の参照）**: `POST /api/v1/developer/facilities/{facility_id}/faqs/bulk-upload`（`multipart/form-data`: `file`, `mode=add`）。フロントの Axios は **60 秒タイムアウト**。

**監査**: `admin_activity_logs` の `action_type=developer_faq_bulk_upload`。

**詳細・用語（シェル上のパス等）**: `docs/FAQ_CSV一括登録_翻訳生成_必須ルール_AIと運営向け.md` §5.3.0、実装の記録 `docs/20260321_CSV一括登録_開発者管理画面からの実行_実装計画.md`。

**メリット**: パスワード共有なし。Shell・Docker 不要で履行しやすい。誤環境・誤施設の再発防止に UI が寄与。  
**デメリット**: 開発者パスワードを知る者に限定（現状どおり）。

---

### 方法C: バックエンドスクリプトでサービスを直接呼ぶ（バックアップ）

**位置づけ**: **方法D が使えない場合・障害時・緊急時・オフライン検証**の補助。認証情報をユーザーと共有せず、運営側だけでアップロードを完結できる。

**手順**

1. **CSVを用意**（ヒアリング・作成は別プロセス。フォーマットは [FAQ_CSV_format_guide.md](files/FAQ_CSV_format_guide.md) に準拠。必須: `category`, `language_ja_question`, `language_ja_answer`。UTF-8（BOM付き推奨）で保存）。サーバーやDocker内から読めるパスに配置。
2. **対象施設ID**を確認する（開発者管理ページの施設一覧やDBで確認）。
3. **backend** でスクリプトを実行する（Docker利用を推奨。要約定義書の大原則「Docker環境必須」に合わせる）。
   - **Docker で実行する場合**（コンテナ内のパスを指定。ホストのCSVを渡すには `docker compose cp` でコンテナにコピー）:
     ```bash
     cd backend
     docker compose cp ./faq.csv backend:/tmp/faq.csv
     docker compose exec backend python scripts/bulk_upload_faq_csv_for_facility.py --facility-id 51 --csv /tmp/faq.csv
     ```
   - **ローカルでDB接続する場合**（`DATABASE_URL` 等が設定されていること）:
     ```bash
     cd backend
     python scripts/bulk_upload_faq_csv_for_facility.py --facility-id 51 --csv /path/to/faq.csv
     ```
   - ドライラン（実際には登録しない）: `--dry-run` を付ける。
4. スクリプトは「その施設に紐づく1人のユーザー」を `uploaded_by` として利用する（監査用）。対象施設にユーザーが1人もいない場合、Standard/Premium 以外、FAQ件数上限超過・言語制限超過の場合はエラーで終了する。

**メリット**: ユーザーにパスワードを渡さない。運営側だけで確実に登録作業を履行できる。  
**デメリット**: バックエンド（またはDocker）へのアクセスと、施設IDの特定が必要。

**スクリプトの配置**: `backend/scripts/bulk_upload_faq_csv_for_facility.py`

### 方法B: ユーザーのJWTでAPIを直接叩く（curl / Postman）

**位置づけ**: スクリプト実行環境が使えない場合の**補助手段**。代行履行は可能だが、認証情報の受け渡しが発生する。

**手順**

1. **CSVを用意**（上記と同様）。
2. **依頼元ユーザーのメール・パスワードでログイン**し、JWTを取得する。
   ```bash
   curl -s -X POST "https://yadopera-backend-staging.onrender.com/api/v1/auth/login" \
     -H "Content-Type: application/json" \
     -d '{"email":"施設管理者のメール","password":"パスワード"}' \
     | jq -r '.access_token'
   ```
3. 取得した `access_token` を使って、**bulk-upload** を呼ぶ。
   ```bash
   curl -X POST "https://yadopera-backend-staging.onrender.com/api/v1/admin/faqs/bulk-upload" \
     -H "Authorization: Bearer ここにaccess_token" \
     -F "file=@/path/to/faq.csv" \
     -F "mode=add"
   ```
4. 成功時は 201、重複等で失敗時は 400 が返る。レスポンスの `detail` で理由を確認する。

**注意**: パスワードを運営側で扱うため、依頼に限定し、作業後はパスワードの扱いをユーザーに再周知する。本番では本番バックエンドURLに読み替える。

**メリット**: スクリプトやDockerが不要な環境でも代行履行が可能。  
**デメリット**: 認証情報の受け渡しが発生する。

---

### 方法A: ユーザーにCSVを渡し、ユーザー自身にアップロードしてもらう

**位置づけ**: **代行契約を結んでいない場合**（ユーザーが自分で登録する・無料）の案内。**代行契約を結んだ場合はこの方法は使わない。** 代行契約時に「CSVを渡して自分でアップロードしてください」と案内するのは、有料サービスを提供していないことになり、ビジネスとして不適切である。

**代行を利用しない場合の手順（参考）**

1. ユーザーがテンプレートまたはマニュアルに従ってCSVを用意する（または運営がテンプレートのみ提供）。
2. ユーザーがログインし、管理画面の「FAQ」→「CSV一括登録」からアップロードする。
3. 操作が難しい場合は画面共有で案内する。

**代行契約時**: 方法Aは使用しない。**方法D（第一）**、必要に応じて **方法C または方法B** で運営側がアップロードを実行する。

## 3. 推奨の使い分け（修正版）

| 状況 | 推奨 | 備考 |
|------|------|------|
| **有料代行契約を結んだユーザー**（CSV作成＋アップロードを運営が履行する） | **方法D（開発者管理画面）を第一に使用**。方法D が使えない場合のみ **方法C**、さらに必要なら **方法B（JWT）**。 | 契約履行＝運営がアップロードまで行う。方法Aは使わない。 |
| 開発者画面が使えない・バックエンド Shell でのみ実行できる場合 | **方法C（スクリプト）** | バックアップ。`docs/FAQ_CSV一括登録_翻訳生成_必須ルール_AIと運営向け.md` §5.2.1・§5.3.1。 |
| スクリプト・Dockerが使えない環境で代行を履行する場合 | **方法B（JWT）** | 補助手段。認証情報の取り扱いに注意。 |
| **代行を利用しないユーザー**（自分で登録する・無料） | **方法A**（CSVテンプレート・マニュアルを渡し、ユーザーに管理画面からアップロードしてもらう）。 | 代行契約時には適用しない。 |

**まとめ**: 代行契約時には **方法D を標準**とし、やむを得ない場合に方法C・方法B。方法Aは「代行なし」の場合のみ案内する。

## 4. CSVフォーマット・注意（共通）

- **必須カラム**: `category`, `language_ja_question`, `language_ja_answer`
- **カテゴリ**: `basic`, `facilities`, `location`, `trouble` のいずれか
- **文字コード**: UTF-8（BOM付き推奨）。Excelで保存する場合は「CSV UTF-8（カンマ区切り）」を選択
- **重複**: 既に同じ `intent_key` が存在する行があると、**一括が 400 でロールバック**する。追加のみのため、既存FAQと被らない内容にするか、事前に施設のFAQ一覧で intent_key を確認する
- **プラン制限**: Standard は 100件、Premium は無制限（件数はインテント単位）。言語数制限も施設プランに従う

詳細: [FAQ_CSV_format_guide.md](files/FAQ_CSV_format_guide.md)
