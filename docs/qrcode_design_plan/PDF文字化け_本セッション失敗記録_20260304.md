# PDF文字化け問題 — 本セッション失敗記録（2026-03-04）

**目的**: 同一過ちを絶対に繰り返さないため、事実に基づく経過と結果を記録する。  
**重要度**: 最高。QRコード・PDFは本ビジネスの根幹であり、根幹が破壊されたまま修復できていない。開発中に起きたどのトラブルよりも最大級のビジネス影響が発生している。

## 5. 失敗の要因整理（再発防止のため）

1. **ユーザーの「元に戻せ」の意味の取り違え**  
   「セッション開始時」＝フォント同梱済みの `ffe7c04` であるのに、一度「フォント同梱を取り消す revert」を実行してしまった。

2. **原因切り分けの不足**  
   ステージングで文字化けする原因が「コンテナ内にフォントが無い」のか「コードの参照先・登録ロジックの誤り」なのかを、Render 上のビルド/ランタイムで検証せずに、複数のコード変更を続けた。

3. **一度に複数の変更**  
   フォントリストの変更・RUN test 追加・COPY 順序変更・ヘルスエンドポイント追加などを、検証を挟まず連続で行い、どの変更が有効でどの変更が無効か切り分けにくくした。

4. **Render 環境の検証不足**  
   「COPY . . に含めればキャッシュでスキップされない」という前提を、Render の実際のビルドログや、デプロイ後のコンテナ内での `ipagp.ttf` の有無で確認していない。その結果、549c13c デプロイ後も PDF に IPA が反映されていない事実を、事前に予測・防止できなかった。

5. **ビジネス重要度の扱い**  
   QR/PDF が根幹であることを文書では認識していたが、変更の影響範囲とロールバック手順を明確にせず、PoC 顧客影響を最小化する手順（例: 事前のローカル/Docker での PDF 生成確認、Render でのフォント存在確認）を徹底しなかった。

---

## 6. 二度と繰り返さないための措置（記録として残す）

- **「元に戻せ」「セッション開始時」と言われた場合**: 必ず、セッション開始時点のコミットハッシュ（本セッションでは `ffe7c04`）を確認し、そのコミットの内容と完全に一致するよう復元する。解釈で「revert」してよいか判断しない。
- **根本原因が不明な場合**: コードを複数同時に変更しない。まず「コンテナ内にフォントがあるか」「どのフォントが登録されているか」を、デプロイ後の環境（ヘルスエンドポイント・ログ・Shell）で確認する。そのうえで、変更は 1 つずつにし、それぞれデプロイと PDF 取得で効果を確認する。
- **Render でのビルド**: Dockerfile を変更したら、可能な限り「Clear build cache & deploy」を行うか、ビルドログで `COPY . .` および `RUN test -f /app/fonts/ipagp.ttf` が実行されていることを確認する。デプロイ後に、コンテナ内に `/app/fonts/ipagp.ttf` が存在することを確認してから「修正完了」と評価する。
- **根幹機能の変更時**: 変更内容・ロールバック手順・検証方法を短いチェックリストにし、PoC 顧客影響を最小化する順序で実施する。ユーザーの時間を奪わないよう、見通しの甘い「とりあえずの修正」を連発しない。

## 8. ステージングで文字化けした原因の調査分析（2026-03-04）

本セッションでステージングにデプロイした結果、PDF が文字化けした原因を事実に基づいて調査し、以下に記録する。

### 8.1 確認した事実

| 項目 | 結果 |
|------|------|
| リポジトリ内のフォント | `backend/fonts/ipagp.ttf` は `git ls-files` に含まれる（追跡済み）。約 6.2MB。 |
| Git LFS | 未使用。実体の TrueType がコミットされている。 |
| .gitignore | `font` / `.ttf` / `fonts` の除外なし。 |
| backend/.dockerignore | `font` / `.ttf` / `fonts` の除外なし。 |
| render.yaml | `dockerfilePath: ./backend/Dockerfile`, `dockerContext: ./backend`。Docker ビルドを使用。 |
| デプロイ後の PDF | 埋め込みフォントは `DejaVuSans`, `Helvetica`, `Helvetica-Bold` のみ。IPA ゴシックは含まれていない。 |
| コードの参照順 | `qr_code_service.py` は `/app/fonts/ipagp.ttf` を最優先で参照し、存在すれば `pdfmetrics.registerFont(TTFont(...))` で登録する。 |

### 8.2 想定される原因（優先度順）

**原因1: Render の Docker ビルドキャッシュにより、フォントがイメージに含まれていない（最有力）**

- Render は Docker ビルドで **全中間レイヤーをキャッシュ** する（[Render Docker ドキュメント](https://docs.render.com/docker)）。
- 本セッションで一度、フォント同梱を取り消す revert（7077638）をデプロイした。その時点では `backend/fonts/ipagp.ttf` がリポジトリから削除され、Dockerfile の `COPY fonts/ /app/fonts/` は空の `fonts/` をコピーするか、レイヤーがキャッシュされた状態になっていた。
- その後、フォントを再びコミットし、Dockerfile を「COPY . . でフォントを含める」形（549c13c）に変更しても、**COPY . . のレイヤーが「フォントなし」の古いコンテキストでキャッシュされ、そのキャッシュが再利用されている**可能性がある。
- その場合、イメージの `/app/fonts/ipagp.ttf` は存在しないか空となり、`os.path.exists("/app/fonts/ipagp.ttf")` が False になり、リスト末尾の DejaVuSans が採用され、日本語が文字化けする。
- **矛盾点**: 549c13c の Dockerfile には `RUN test -f /app/fonts/ipagp.ttf || exit 1` がある。フォントが無ければビルドが失敗するはず。したがって、(a) RUN test のレイヤーもキャッシュから復元され「成功」として扱われている、(b) 実際にはビルドが失敗しているが Deploy live 表示だけが先行している、(c) ビルドは成功しているが実行時のみ `/app` が別の内容に差し替わっている、のいずれかが必要。

**原因2: ビルドは成功しているが、実行コンテナが別イメージまたは古いイメージである**

- 549c13c の「Deploy live」は、そのコミットでビルドしたイメージではなく、**以前のビルドのイメージ**で起動している可能性（キャッシュやデプロイ順序の不整合）。
- その場合、RUN test は通過した別ビルドのイメージが動いており、実際に動いているイメージにはフォントが入っていない、という状態になり得る。

**原因3: コンテナ内ではフォントがあるが、reportlab の登録に失敗している**

- `/app/fonts/ipagp.ttf` は存在するが、Render のランタイム（ファイルシステム・権限・パス）の違いで `TTFont(font_name, font_path)` が例外を出し、`except` で続行して次のパス（最終的に DejaVuSans）が使われている可能性。
- 現状、この場合のログ（`Failed to register font ...`）をステージングで確認していないため、断定はできない。

**原因4: buildCommand による /app の上書き（可能性は低い）**

- render.yaml の `buildCommand: alembic upgrade head` が、ビルド後や起動時に `/app` を別内容でマウントしたり上書きしたりしている場合、ビルド時にコピーしたフォントが実行時には無い、という状態になり得る。
- 一般的な Render の挙動としては、Docker ビルドで作ったイメージをそのまま起動するため、可能性は低いが要因として列挙する。

### 8.3 結論（調査時点）

- **直接の事実**: ステージングで生成された PDF には IPA ゴシックが埋め込まれておらず、DejaVuSans 等のみ。そのため日本語が正しく表示されず文字化けしている。
- **原因**: **未確定**。以下「10. 調査継続・原因判明のための必須検証」を実施し、事実で確認するまで断定しない。

### 8.4 再発防止のための検証手順（記録）

1. **デプロイ前**: ローカルで `docker build -f backend/Dockerfile backend/` を実行し、`RUN test -f /app/fonts/ipagp.ttf` が通ること、および `docker run --rm <image> ls -la /app/fonts/` で ipagp.ttf が存在することを確認する。
2. **Render デプロイ時**: 「Clear build cache & deploy」を一度実行し、キャッシュの影響を外す。
3. **デプロイ後**: 可能であれば Render Shell で `ls -la /app/fonts/ipagp.ttf` を実行し、ファイルの有無を確認する。または、開発者用ヘルス等で `/app/fonts/ipagp.ttf` の存在を返すエンドポイントを再度用意し、ステージングから確認する。
4. **PDF 生成後**: アプリログで `Japanese font registered: ipagp from /app/fonts/ipagp.ttf` が出力されているか確認する。出ていなければ、別のパス（DejaVu）が使われている。

**補足**: ユーザーは **「Clear build cache & deploy」を毎回手動で実施している**。したがって今回の文字化けは「キャッシュを消していなかった」ことが原因ではなく、別要因を特定する必要がある。

## 10. 調査継続・原因判明のための必須検証

**方針**: 原因を把握できるまで調査分析を継続する。**原因が判明するまで報告しない。** 以下は、原因を確定するために Render 側で実施すべき検証のみを記載する。

### 10.1 すでに否定した要因（コード・設定上の事実）

| 要因 | 結果 |
|------|------|
| フォントが Git に含まれていない | `git ls-files backend/fonts/` に `backend/fonts/ipagp.ttf` あり。否定。 |
| .gitignore で除外されている | `font` / `.ttf` / `fonts` の除外なし。否定。 |
| backend/.dockerignore で除外されている | 同様に除外なし。否定。 |
| render.yaml で Backend に rootDir が設定されコンテキストがずれている | render.yaml の web サービスに rootDir はない。dockerContext: ./backend はリポジトリルート基準で backend/。否定。 |
| Dockerfile の COPY 順・パス誤り | COPY . . でコンテキスト（backend/）が /app/ にコピーされる。dockerContext: ./backend のため backend/fonts/ → /app/fonts/。否定。 |

### 10.2 原因特定に必要な事実（Render 側でないと確認できないもの）

次のいずれかが成り立っていれば、原因を特定できる。

1. **ビルド結果**: 該当デプロイの Render ビルドログで、`COPY . .` および `RUN test -f /app/fonts/ipagp.ttf` が **キャッシュではなく実行**されているか。実行されていれば、その時点のイメージにはフォントがある。
2. **実行時ファイルシステム**: デプロイ後のコンテナ（Render Shell または起動時スクリプト）で **`ls -la /app/fonts/ipagp.ttf`** を実行し、存在・サイズが期待どおりか。
3. **アプリケーションログ**: ステージングで PDF を 1 回生成した直後のログに **`Japanese font registered: ipagp from /app/fonts/ipagp.ttf`** が出ているか。出ていなければ、そのリクエストでは `/app/fonts/ipagp.ttf` が使われておらず、別パス（例: DejaVu）が使われている。

### 10.3 原因別の解釈（検証結果に応じて確定する）

| 検証結果 | 解釈 |
|----------|------|
| ビルドログで COPY / RUN test がキャッシュ表示 | 原因: Render のキャッシュが、コンテキスト内容の変更を反映せずレイヤーを再利用している。対策: キャッシュを無効化する ARG の導入、または Render サポートへの問い合わせ。 |
| ビルドでは実行されているが、実行時 `ls` で /app/fonts/ipagp.ttf が無い | 原因: 実行コンテナが、そのビルドのイメージではない（古いイメージ・別ビルドのイメージが動いている）。対策: デプロイ手順・サービス設定の見直し。 |
| 実行時には /app/fonts/ipagp.ttf が存在するが、ログに「Japanese font registered: ipagp」が出ない | 原因: ファイルはあるが reportlab の TTFont 登録が失敗している（例外でスキップ）。対策: 該当例外ログの確認と、reportlab のバージョン・環境依存の要因調査。 |
| 上記のいずれでも説明がつかない | 追加で、同一リクエストで「どのフォントパスで exists が True になったか」「どのフォント名が japanese_font_name に設定されたか」をログに出すようにし、再現デプロイ・再取得 PDF で確認する。 |

### 10.4 脆弱性の認識（再発防止の前提）

- 本件程度の修正ミスでビジネスが終わり得ることを**深刻に認識**する。
- 原因が判明するまで、**「修正完了」「解消した」と報告しない。**
- 原因が判明した時点で、本記録に「確定した原因」と「再発防止のための恒久対策」を追記する。

## 11. ログによる原因確定と修正（2026-03-04 追記）

ユーザーから共有された **Render デプロイログ** により、原因を特定し、修正を実施した。

### 11.1 ログで確定した事実

ログに以下の行が複数回出力されていた。

```
Japanese font registered: DejaVuSans from /usr/share/fonts/truetype/dejavu/DejaVuSans.ttf
```

- **意味**: `qr_code_service.py` の日本語フォント候補リストは、先頭が `/app/fonts/ipagp.ttf`、続いて IPA 系システムパス、**最後**が DejaVuSans。このログは **リストの最後のパスだけが使われた** ことを示す。
- **結論**: 実行時コンテナ内に **`/app/fonts/ipagp.ttf` が存在しない**。同様に IPA のシステムパスも存在せず、6 番目の DejaVuSans のみが登録されている。したがって **COPY でフォントがイメージに入っていない**（キャッシュで古いレイヤーが使われている、またはフォントのみをコピーするレイヤーが実行されていない）。

### 11.2 実施した修正（Dockerfile）

1. **フォントを単体で先にコピー**  
   `COPY fonts/ /app/fonts/` を独立したレイヤーとして実行し、直後に `RUN test -f /app/fonts/ipagp.ttf || exit 1` で欠落時はビルド失敗とする。
2. **キャッシュ無効化用 ARG**  
   `ARG RENDER_GIT_COMMIT=` と `ARG CACHEBUST=1` を追加。Render がビルド時に `RENDER_GIT_COMMIT` を注入する場合はそれで、未設定時は Dashboard で `CACHEBUST` を変更可能にし、**COPY の直前レイヤーが変わることで COPY が毎回実行される**ようにした。

### 11.3 ユーザーが実施すること

1. **「Clear build cache & deploy」で再デプロイ**  
   キャッシュを消したうえでデプロイし、新しい Dockerfile でビルドが最初から走るようにする。
2. **（任意）Render の Environment に `CACHEBUST` を追加**  
   今後のデプロイで COPY がキャッシュされないようにするため、`CACHEBUST` を追加し、デプロイのたびに変わる値（例: タイムスタンプやコミット SHA）に設定する。
3. **デプロイ後の確認**  
   PDF を 1 回生成し、ログに **`Japanese font registered: ipagp from /app/fonts/ipagp.ttf`**（または IPA 系パス）が出ることを確認する。DejaVuSans のみの場合は、まだフォントがイメージに入っていない。

### 11.4 疑義・追加確認が必要な場合

- Render のビルドログで `COPY fonts/` および `RUN test -f /app/fonts/ipagp.ttf` が **キャッシュではなく実行**されているか確認したい。
- デプロイ後のコンテナ内で `ls -la /app/fonts/ipagp.ttf` の結果を共有できる場合、原因の裏付けに使える。

上記について、ログや結果を共有いただければ、必要に応じて追加の修正案を提示する。
