## 1. 外部分析の要約

第三者が Phase1・Phase2 の文書を精読したうえで整理した分析（要約）は以下のとおりである。

### 1.1 Phase1 の結論について

- **「Docker イメージは実行されていない」** という判定は **完全に正しい**。
- 根拠: `cwd` が `/opt/render/project/src/backend`、`/app` が NOT_FOUND、`build_marker` が NOT_FOUND であること。Docker であれば `cwd = /app` になるため、**Native runtime（ソース実行）** の典型パターンである。
- よって **ケース A（Docker イメージ未使用）** は 100% 正しい分析と結論づけている。

### 1.2 Phase2 の分析について

- **「Build Command があるためネイティブ Python 実行」** は正しい可能性が高い。
- Render では「Build Command あり ⇒ Native build」「Dockerfile 指定 ⇒ Docker build」であり、**既存サービスでは runtime が固定される**。したがって render.yaml を変更しても既存サービス単体では runtime は変わらない。
- **「runtime 変更は Blueprint 同期でのみ可能」** は Render 公式仕様と一致する、としている。

### 1.3 提示した「次の行動」と注意点

- **New → Blueprint → repo connect → Deploy Blueprint** は基本的に正しい行動である。
- **重要**: Blueprint は「既存サービス更新」ではなく **「新規サービス作成」** することが多い。そのため、既存の `yadopera-backend-staging` とは別に `yadopera-backend-staging-1` などが作られる可能性がある。
- **安全手順**: Blueprint 作成時に **Preview** で「Services to create」を確認し、`yadopera-backend-staging` が **create** ではなく **update** になっているか確認する。**create** の場合は一旦キャンセルし、既存サービスを削除してから Blueprint で作り直す方が安全、としている。

### 1.4 推奨される進め方

- **一番安全な方法**: 今の staging を削除し、Blueprint で作り直す。
- 理由: runtime が確実に Docker になる、render.yaml が完全適用される、環境ズレが消える。
- 手順: (1) staging service を delete → (2) New → Blueprint → (3) repo connect → (4) deploy。
- PoC 段階であれば、staging DB に重要データがなく、本番と分離され、コードが GitHub に存在することを確認したうえであれば、この方法は問題ないとしている。

### 1.5 文字化けの真因

- **フォントが悪いのではなく、Render の runtime の誤認識が本質** である。
- 構造: Render runtime mismatch → Dockerfile 無視 → フォント未導入 → PDF 文字化け。
- なぜ以前は文字化けしなかったか: **ネイティブ Render 環境の OS に DejaVu が入っており、偶然日本語が表示できていた**。Docker 化するとベースイメージにフォントがほぼ無く、文字化けするという典型パターンである。

### 1.6 調査手法の評価

- **`/__debug_env` で環境を観測したこと** は、SRE が行うレベルの切り分けとして **非常に良いデバッグ方法** であり、分析プロセスはかなり正確である、と評価している。

## 3. 調査分析の完全な整理

### 3.1 問題の構造（確定している事実）

1. **現状の実行環境**
   - ステージングバックエンドは **Native Python**（ソース実行）で動作している。
   - `cwd`: `/opt/render/project/src/backend`、`/app`: NOT_FOUND、`build_marker`: NOT_FOUND。
   - デプロイ対象のコミット（例: 5bbbc6b）のコードは反映されているが、**Dockerfile でビルドされたイメージは使われていない**。

2. **原因**
   - サービスは **Dashboard で手動作成** され、**Blueprint に接続されていない**。
   - そのため **render.yaml は無視** され、Dashboard の Build Command（`pip install ...`）と Start Command に従い、ネイティブ Python でビルド・実行されている。
   - Runtime は **Dashboard の UI では変更できない**（Build & Deploy に「Language」「Environment」を選ぶ項目は存在しない）。

3. **文字化けの本質**
   - **フォント設定の誤りではなく、実行環境の不一致** である。
   - ネイティブ環境にはシステムの DejaVu があり、以前は偶然日本語が表示できていた。Docker 化されていないため Dockerfile 内のフォント設定は効いておらず、かつ現行のネイティブ環境でも IPA 等は入っていない、という状態が続いている。

### 3.2 Blueprint 作成が 2 回失敗した原因（確定）

- **エラー**: `unknown type "static"`（Render からの表示）。
- **原因**: `render.yaml` のフロントエンドサービスで **`type: static`** と定義しているため。
- **Render Blueprint 仕様**（[Blueprint Spec](https://docs.render.com/blueprint-spec)）では、静的サイトは **`type: web`** に **`runtime: static`** を指定する形で定義する。**`type: static` という type は存在しない**。
- そのため Blueprint パーサーが `render.yaml` を解釈できず、Blueprint の作成・同期が失敗している。

**該当箇所（現行 render.yaml 34–36 行目付近）**:

```yaml
  - type: static
    name: yadopera-frontend-staging
```

**仕様に合わせる場合の形（修正は指示があるまで行わない）**:

- `type: static` をやめ、**`type: web`** と **`runtime: static`** に変更する。

### 3.3 大原則との対応

| 大原則 | 本件での適用 |
|--------|--------------|
| **根本解決 > 暫定解決** | フォントだけいじるのではなく、実行環境（runtime）を Docker に揃えることが根本。 |
| **シンプル > 複雑** | まず render.yaml を仕様どおりにし、Blueprint で 1 回で正しくデプロイする。 |
| **具体的 > 抽象的** | __debug_env の cwd・build_marker で事実を確認する。 |
| **安全・確実 > 慎重さ** | Preview で create/update を確認する。必要なら「staging 削除 → Blueprint で作り直し」で二重作成を避ける。 |

## 4. 大原則に準拠した次の進め方（Render 画面に即した詳細）

- **方針**: Blueprint を利用する。そのために (1) `render.yaml` を Blueprint 仕様に合わせて修正し、(2) Blueprint を作成・デプロイする。既存サービスを「更新」するか「削除して作り直す」かは、Preview の結果で判断する。

### 4.2 Step 0: render.yaml の修正（実施は指示があるまで待つ）

**内容**  
Blueprint がパースできるように、フロントエンドサービスを仕様どおりに直す。

## 7. Step 4 完了後の結果説明・評価（2026-03-06）

### 7.1 スクショから分かったこと

- **実施済み**: §4.6 Step 4 まで（New Blueprint 画面でリポジトリ接続・Branch: develop・Blueprint Path: render.yaml の設定、Review Blueprint configurations の表示まで）。
- **Blueprint の対象**: `render.yaml` には **バックエンド**（yadopera-backend-staging, runtime: docker）と **フロントエンド**（yadopera-frontend-staging, runtime: static）の **2 サービス** が含まれており、Review では両方の設定が読み込まれる。表示がフロントエンドの一部（rootDir: frontend, routes 等）になっている画面は、検出された設定の一覧の一部であり、「フロントだけデプロイする」状態ではない。
- **エラー**: Render が **Blueprint の検証でエラー** を出しており、**Deploy Blueprint には進めない**。
  - 表示メッセージ: **「A Blueprint file was found, but there was an issue.」**
  - 詳細（4 件）:
    - `services[1].routes[1] source cannot be the same as destination`
    - `services[1].routes[2] source cannot be the same as destination`
    - `services[1].routes[3] source cannot be the same as destination`
    - `services[1].routes[4] source cannot be the same as destination`

### 7.2 原因の特定

- **services[1]** = `render.yaml` の 2 番目のサービス = **yadopera-frontend-staging**（フロントエンド）。
- **routes** は静的サイト用の rewrite ルール。現行の定義は以下のとおり。

| インデックス | source            | destination         | 判定 |
|-------------|-------------------|---------------------|------|
| routes[0]   | /                 | /index.html         | 異なる → OK |
| routes[1]   | /assets/*         | /assets/*           | **同一** → エラー |
| routes[2]   | /registerSW.js    | /registerSW.js      | **同一** → エラー |
| routes[3]   | /manifest.webmanifest | /manifest.webmanifest | **同一** → エラー |
| routes[4]   | /sw.js            | /sw.js              | **同一** → エラー |
| routes[5]   | /*                | /index.html         | 異なる → OK |

- Render の Blueprint 検証では **rewrite の source と destination が同一のルールは禁止** されている。  
  「通過」目的（例: `/assets/*` → `/assets/*`）でも、同一だとエラーになる。

### 7.3 評価

- **Step 4 まで**: リポジトリ接続・Branch・Blueprint Path の設定は妥当。`render.yaml` も読み込まれ、バックエンド（runtime: docker）・フロントエンド（runtime: static）の 2 サービスが検出されている。
- **現状**: **検証エラーのため次のステップ（Deploy Blueprint / Preview での create・update 確認）には進めない。** このまま「Deploy Blueprint」を押しても失敗するか、実行できない。

### 7.4 次のステップに進めてよいか

**いいえ。** 上記の **routes の検証エラーを解消するまで、Step 5（Preview の確認）や Deploy Blueprint には進まない。**

### 7.5 大原則に準拠した修正案（指示があるまで適用しない）

- **根本解決**: Blueprint がパース・検証を通るように、**禁止されている「source と destination が同一」の rewrite をやめる**。
- **シンプル**: 同一の 4 本（routes[1]〜[4]）を **削除** する。静的サイトでは、多くの場合 `/assets/*` や `/registerSW.js` 等は **静的ファイルとしてそのまま配信** されるため、同じパスへの rewrite がなくても動作する。SPA 用に必要なのは **`/` → `/index.html`** と **`/*` → `/index.html`** の 2 ルールである。
- **具体的な変更案**（実施は指示があるまで行わない）:
  - **削除するルール**:  
    `source` と `destination` が同一の 4 件  
    （`/assets/*`→`/assets/*`、`/registerSW.js`→`/registerSW.js`、`/manifest.webmanifest`→`/manifest.webmanifest`、`/sw.js`→`/sw.js`）。
  - **残すルール**:  
    `source: /`, `destination: /index.html` と `source: /*`, `destination: /index.html` の 2 件のみ。

上記を反映したうえで **コミット・プッシュ** し、Render の New Blueprint 画面で **再度 Review**（またはページを読み直し）すると、検証エラーが解消され、Step 5（Preview で create/update 確認）→ Step 6（Deploy Blueprint）に進める想定である。

## 8. routes 修正後の実行結果 — region エラー（2026-03-06）

### 8.1 結果の説明

- **実施**: routes の source=destination 4 件削除をコミット・プッシュしたあと、Render の New Blueprint 画面で再度 Review を実行した。
- **表示**: 「Review Blueprint configurations from kurinobu/yadopera」で、**新たな検証エラー** が表示された。
- **エラー内容**: **「A Blueprint file was found, but there was an issue.」**  
  **`services[0].region invalid region: tokyo`**

### 8.2 評価

- **routes の修正**: 前回の「source cannot be the same as destination」エラーは解消されており、**routes まわりの修正は有効** である。
- **現状**: **region 指定が不正** なため、Blueprint の検証が通らず、Step 5・Step 6 には進めない。

### 8.3 調査分析

- **services[0]** = バックエンド（yadopera-backend-staging）。現行 `render.yaml` で **`region: tokyo`** を指定している。
- **Render の公式仕様**（[Regions - Render Docs](https://render.com/docs/regions)、[Blueprint Spec](https://docs.render.com/blueprint-spec)）では、有効な region は次の **5 つのみ** である。
  - **singapore**
  - **frankfurt**
  - **virginia**
  - **ohio**
  - **oregon**（デフォルト）
- **tokyo** は **有効な値ではない**。Render は東京リージョンを提供しておらず、Blueprint は上記 5 いずれか（または未指定で oregon）でなければならない。

### 8.4 大原則に準拠した修正案

- **根本解決**: Blueprint が検証を通るように、**有効な region に変更する**。
- **シンプル**: `region: tokyo` を、**アジアで最も近い有効リージョン** である **`region: singapore`** に 1 箇所だけ変更する。既存のステージングサービスが別のリージョンで作成済みの場合は、既存に合わせるか、デフォルトの **`oregon`** にすることも選択肢である。
- **具体的な変更**: `render.yaml` のバックエンドサービス（services[0]）の **`region: tokyo`** を **`region: singapore`** に変更する。

**質問**: 「yadopera-backend-staging が「create」と表示される場合」→「推奨: 一旦キャンセル」→「既存の yadopera-backend-staging（および必要に応じて yadopera-frontend-staging）を削除」→「New > Blueprint から同じ render.yaml で Blueprint を作成・デプロイする」でよいか。

### 9.4 PoC 実行中における DB などのバックアップ

**質問**: PoC 実行中です。DB などのバックアップは必要ないですか。

**回答**: **利用している DB の種類によって異なります。**

- **現在の構成（render.yaml より）**: バックエンドの `DATABASE_URL` は **`sync: false`** で、コメントに「Railway Hobbyから取得」とある。つまり **DB は Render ではなく Railway 側でホストされている** と解釈できる。
- **その場合**:
  - **Render 上で削除するのは「Web サービス（バックエンド）」と「Static Site（フロント）」だけ** であり、**Railway 上の Postgres は削除されない**。データは Railway に残る。
  - 新しく作成するバックエンドサービスに、**同じ `DATABASE_URL`（および `REDIS_URL`、`OPENAI_API_KEY`）を再度設定** すれば、これまでと同じ DB・Redis に接続できる。
  - したがって **Render のサービス削除だけなら、DB のバックアップは必須ではない**（Railway 側のデータはそのまま）。
- **もしステージング用 DB が Render Postgres の場合**:
  - Render の「サービス削除」が DB 削除を伴うかは、そのサービスの種類・設定次第。**Render Postgres を個別に削除しない限り、DB は残る** ことが多いが、仕様は [Render のドキュメント](https://render.com/docs) で要確認。
  - **残したいデータがある場合** は、PoC であっても **pg_dump などでバックアップを取得しておく** ことを推奨する。
- **まとめ（PoC の場合）**:
  - **DB が Railway 等の外部** → サービス削除では消えないので、バックアップは必須ではない。必要に応じて Railway 側のバックアップ方針に従う。
  - **DB が Render 上で、かつデータを残したい** → pg_dump 等のバックアップを推奨。
  - **ステージングに重要なデータが無い** → 運用方針としてバックアップを省略することは可能だが、**可能な範囲で取得しておく** と安全。

### 9.5 補足: なぜ「削除したのに同じ URL で立ち上がるか」と、環境設定の扱い

**「削除したのに同じ URL で立ち上がる」について**

- **削除するもの**: 既存の **Render 上のサービス**（Web サービス・Static Site）そのもの。削除すると、そのサービスとその URL は一度なくなります。
- **その後にやること**: **New > Blueprint** で同じ `render.yaml` を指定して **新規に** サービスを作成・デプロイする。
- **同じ URL になる理由**: `render.yaml` には **サービス名** が `name: yadopera-backend-staging` / `name: yadopera-frontend-staging` と書かれている。既存サービスを削除したあとなので、名前の衝突がなく、Render は **この名前どおりの新サービス** を 1 組だけ作る。Render のデフォルトでは **サービス名.onrender.com** が URL になるため、**新しく作られたサービス** が再び `yadopera-backend-staging.onrender.com` / `yadopera-frontend-staging.onrender.com` という **同じ URL** で使えるようになる、という意味です（「削除したサービスが復活する」のではなく、「同じ名前の新しいサービスができて、同じ URL になる」）。

**環境設定（Stripe など）が消えるか**

- **はい。削除したサービスにだけ紐づいていた設定はすべて消えます。**
- Render のサービスを削除すると、そのサービスの **Environment** にあった **すべての環境変数**（Stripe 系、その他 Dashboard で手動追加したもの）は **残りません**。Blueprint で **新規** に作るサービスは、**render.yaml に書かれている env だけ**が初期状態です。
- **render.yaml に書いてあるもの**:  
  `DATABASE_URL` / `REDIS_URL` / `OPENAI_API_KEY`（`sync: false` のため **値は YAML に無く、デプロイ時または後に手動入力**）、`SECRET_KEY`（`generateValue: true` で自動生成）、`CORS_ORIGINS` / `ENVIRONMENT` / `FRONTEND_URL` / `DEBUG` / `LOG_LEVEL`（値が YAML にあり、そのまま適用される）。
- **render.yaml に無いもの**:  
  **STRIPE_WEBHOOK_SECRET** や **STRIPE_API_KEY** など、Dashboard でだけ設定していた環境変数は **YAML に含まれていません**。そのため、**削除後に Blueprint で作り直したサービスには入らず、消えたまま**です。
- **必要な対応**:  
  **削除する前に**、既存バックエンドの **Environment** 画面で、**すべての環境変数名と値**（Stripe 含む）を控えるかスクショを保存する。Blueprint でデプロイした **あと**、新しいバックエンドの **Environment** で、**render.yaml に無いもの**（Stripe 系など）を **もう一度追加** する。  
  （DATABASE_URL / REDIS_URL / OPENAI_API_KEY は Deploy 時の入力または作成後の Environment で設定。）

### 9.6.1 削除の操作方法（Render ダッシュボードでの手順）

**Suspend ボタンでは駄目です。** Suspend は「一時停止」であり、サービスは残ったままです。Blueprint で同じ名前のサービスを新規作成するには、**サービスを完全に削除（Delete）** する必要があります。

**正しい削除手順（各サービスごとに実施）:**

1. **削除対象のサービス名をクリック**  
   （例: `yadopera-backend-staging`）をクリックし、そのサービスの詳細画面を開く。

2. **左メニューまたは画面上で「Settings」を開く**  
   サービスの **Settings** ページに移動する。

3. **下までスクロールし「Delete」のセクションを探す**  
   Render では **Settings** のいちばん下に **「Delete or suspend」** または **「Delete Service」** のようなセクションがある。**Suspend** ではなく **Delete（削除）** のほうを選ぶ。

4. **削除の確認**  
   多くの場合、**サービス名を入力して確認** するよう求められる。表示の指示に従い、該当サービス名（例: `yadopera-backend-staging`）を入力してから削除を実行する。

5. **もう 1 つのサービスも同様に削除**  
   **yadopera-backend-staging** を削除したあと、**yadopera-frontend-staging** についても上記 1〜4 を繰り返す。

**まとめ:**

- **やってはいけないこと**: サービス一覧の **⋯（三点メニュー）→ Suspend** だけを押す。これではサービスは残り、Blueprint で「既存に紐づいている」状態が解消されない。
- **やること**: 各サービスを開く → **Settings** → 下の方の **Delete Service**（削除）を実行し、確認でサービス名を入力して削除する。

### 9.8 デプロイ・環境設定後の点検（設定が正しいか・抜け落ちがないか）

デプロイと環境設定が完了したら、以下で**設定の正しさ**と**抜け落ち**を点検する。

#### A. 実行環境の確認（Docker で動いているか）

| 確認項目 | 方法 | 期待値 |
|----------|------|--------|
| **cwd** | `GET https://yadopera-backend-staging.onrender.com/__debug_env` の JSON の `cwd` | `"/app"`（Docker の WORKDIR） |
| **build_marker** | 上記レスポンスの `build_marker` | `"BUILD_MARKER_20260305_001"` |
| **app_fonts_list** | 上記レスポンスの `app_fonts_list` | IPA 等が含まれる（例: `["ipagp.ttf"]`）。PDF 日本語用。 |

上記が満たされていれば、**想定どおり Docker イメージで動作しており、PDF 用フォントもイメージ内に存在している**。

#### B. 接続・必須環境変数の確認

| 確認項目 | 方法 | 期待値 |
|----------|------|--------|
| **ヘルス** | `GET https://yadopera-backend-staging.onrender.com/api/v1/health` | `{"status":"healthy","database":"connected","redis":"connected"}` |
| **DB・Redis** | 上記で `database` / `redis` が `connected` | 接続成功。未設定や誤設定なら `error` になる。 |

#### C. 環境変数の抜け落ちチェック（Render Dashboard で確認）

バックエンドの **Environment** に、以下が**キー名も含めて**設定されているか確認する。**typo に注意**（例: `FRONTEND_UR` ではなく `FRONTEND_URL`、`CORS_ORIGIN` ではなく `CORS_ORIGINS`）。

**C.1 どこを見ればよいか（Render ダッシュボードでの手順）**

1. ブラウザで [https://dashboard.render.com](https://dashboard.render.com) にログインする。
2. ダッシュボードの **Dashboard** または **Services** 一覧から、**yadopera-backend-staging** をクリックする（バックエンドのサービス名）。
3. サービス詳細画面で、**左サイドメニュー** または画面上部タブのうち **「Environment」** をクリックする。
4. **Environment** 画面に、環境変数の一覧（Key と Value の一覧）が表示される。**Key** の列で次を確認する。
   - **CORS_ORIGINS** というキーが**正確に**あるか（`CORS_ORIGIN` のように末尾の `S` が無い typo になっていないか）。
   - **FRONTEND_URL** というキーが**正確に**あるか（`FRONTEND_UR` のように末尾の `L` が無い typo になっていないか）。
5. 値は「Secret」でマスクされていてもよい。**キー名のスペル**が正しいことが重要。typo があれば、正しいキー名で変数を追加し、誤ったキーは削除する。

**render.yaml で自動設定されるもの（Blueprint で入る想定）:**

| キー | 備考 |
|------|------|
| DATABASE_URL | sync:false のため Deploy 時または手動で設定 |
| REDIS_URL | 同上 |
| OPENAI_API_KEY | 同上 |
| SECRET_KEY | generateValue で自動 |
| CORS_ORIGINS | 値: フロント URL 等 |
| ENVIRONMENT | staging |
| FRONTEND_URL | ステージング用フロント URL |
| DEBUG | False |
| LOG_LEVEL | INFO |

**手動で追加が必要なもの（render.yaml に無い）:**

| キー | 用途 | 未設定時の影響 |
|------|------|----------------|
| STRIPE_SECRET_KEY | Stripe 秘密鍵 | 決済・サブスク不可 |
| STRIPE_WEBHOOK_SECRET | Webhook 署名 | Webhook 失敗 |
| STRIPE_PUBLISHABLE_KEY | 公開キー | フロントで Stripe 利用不可 |

| DEVELOPER_PASSWORD, DEVELOPER_SESSION_EXPIRE_HOURS | 開発者管理画面 | 開発者ページにログイン不可 |

**点検の結論:** 上記 A・B が期待どおりであり、C の必須項目（利用している機能分）が Dashboard に存在し、キー名の typo がなければ、設定の抜け落ちはないと判断できる。必要に応じて、ログイン・PDF 生成・決済・メール送信を 1 回ずつ動作確認するとより安全。

#### 動作確認の実施状況（何を誰が確認するか）

| 確認項目 | 実施者 | 実施方法・備考 |
|----------|--------|----------------|
| **Docker で動いているか**（cwd=/app, build_marker） | 文書作成側で実施済み | `GET /__debug_env` で確認。結果: 想定どおり。 |
| **DB・Redis 接続** | 文書作成側で実施済み | `GET /api/v1/health` で確認。結果: healthy / connected。 |
| **CORS_ORIGINS・FRONTEND_URL のキー名** | **利用者** | 上記 **C.1** の手順で Render Dashboard → yadopera-backend-staging → **Environment** を開き、一覧の Key を目視確認。 |
| **ログイン** | **利用者** | ブラウザでステージングのフロントにアクセスし、施設・管理者のログインができるか操作して確認。 |
| **PDF 生成** | **利用者** | ステージングで QR コード PDF を 1 回生成し、日本語が正しく表示されるか確認。 |

| **メール送信** | **利用者** | 認証メールや通知メールが送信・受信できるか 1 回確認。 |

**まとめ:** 「動作確認」のうち、**API で確認できる項目（Docker 稼働・ヘルス）は実施済み**。**ログイン・PDF 生成・決済・メール**は、**利用者がブラウザでステージングにアクセスし、手動で 1 回ずつ操作して確認する必要がある**（AI 側からは代行できない）。

### 9.9 PDF 文字化けが解消されていない件（2026-03-06 確認結果）

**確認結果（利用者報告）**

- 環境変数: CORS_ORIGINS・FRONTEND_URL は正しく設定済み。
- ブラウザテスト完了、ログイン可能。
- **PDF 出力: 文字化けは解消されていない**（例: 設置場所・生成日時などの日本語が表示されない／化ける）。

**想定内か**

- **想定内ではない。** Blueprint デプロイで Docker 実行に切り替えたあとも、PDF の日本語表示が復旧していることが期待されていた。

**原因の説明**

1. **実行環境**は想定どおり **Docker**（`cwd=/app`、`build_marker` 確認済み）。コンテナ内に IPA フォント（`/app/fonts/ipagp.ttf`）が存在するビルドもあった。
2. **アプリ側のコード**は、当該セッションで「DejaVu のみ」に戻していた（`qr_code_service.py` の `japanese_font_paths` が DejaVuSans のみ）。
3. **DejaVu Sans は日本語グリフを持たない**ため、「設置場所」「生成日時」等の日本語が描画されず、空白や文字化けになる。
4. その結果、**イメージに IPA が入っていても、コードが参照していない**状態だった（コードとイメージの不整合）。

**評価**

- Docker 化と Blueprint 適用は成功しており、文字化けの直接の原因は **PDF 用に使用するフォントがコード上で DejaVu のみになっていたこと**。
- 日本語表示を復旧するには、**PDF 用に IPA フォントを参照するようにコードを戻す**必要がある。

**実施した対応**

1. **Dockerfile**: `fonts-ipafont` を apt で追加し、コンテナ内の `/usr/share/fonts/` に IPA フォントが入るようにした。
2. **qr_code_service.py**: PDF 用フォントの候補に、`/app/fonts/ipagp.ttf` およびシステムの IPA パスを追加し、**IPA を優先・DejaVu をフォールバック**するように変更した。

**次の確認**

- 上記をコミット・プッシュし、ステージングを再デプロイしたうえで、**QR コード PDF を再度出力**し、日本語（設置場所・生成日時など）が正しく表示されるか確認する。
