## 0. この文書の位置づけ

親計画の A-4 は **「エスカレーション記録の作成だけでなく、施設担当者がメール等で確実に気づけること」**、および **「メールに載せる受付番号が、ゲスト画面・管理画面と同じ `escalations.id` であること」**を求めている。

親計画の **大原則（開発）**（Docker で検証、`feature/*` → `develop`、Runbook 参照）に従うこと。**受付番号に別採番・別フィールドを増やさない。**

**用語（本書）**: ビジネス説明・運用に触れる箇所は **スタッフ不在時間帯対応キュー** と統一する。コード上の `OvernightQueue` 等は実装名。正本: `docs/開発規約/文書用語_スタッフ不在時間帯対応キュー.md`。

## 1. 親計画から外れない受け入れ条件（必須）

| 項目 | 内容 |
|------|------|
| 受付番号 | メールの件名・本文に含める数値は **必ず DB の `escalations.id`**（ゲストの成功表示・管理画面の受付番号と同一）。 |
| 件名 | 可能なら件名にも受付番号を含める（運用で検索しやすくするため）。 |
| 本文 | **必ず**受付番号を含める。施設名、会話を開くための手がかり（例: 管理画面の会話へ遷移するための URL またはセッション ID）、作成時刻など。**ゲストのメッセージ本文の扱いは個人情報に配慮し、要約または末尾に抜粋するか方針を決める。** |
| 送信経路 | 既存の Brevo（`sib-api-v3-sdk`）・`EmailService` / 設定（`settings.brevo_api_key` 等）に合わせる。新規の別メール基盤を増やさない（親計画の「既存通知チャネル」）。 |
| 失敗時 | **ログに追えること**（`facility_id`, `escalation_id`, 例外内容）。本番では Brevo 未設定を黙って飲み込まない（開発では警告ログで可という親計画の趣旨に合わせ、**環境で挙動を切り替える**）。 |
| 通知済みフラグ | 送信成功後に **`escalations.notified_at`** を更新し、管理画面や API が「通知済み」を正当に表示できるようにする（現状フィールドは存在するが、送信処理と結びついていない）。 |

## 2. 調査結果（事実）

### 2.1 エスカレーション作成後にメールが送られていない

| 場所 | 内容 |
|------|------|
| `EscalationService.create_escalation` | DB にレコード作成、会話の `is_escalated` 更新、ダッシュボード用キャッシュ無効化まで。**メール送信の呼び出しはない。** |
| `POST /api/v1/chat/escalate`（ゲストの「スタッフに連絡」） | 上記 `create_escalation` のあと **200 返却まで。メール・スタッフ不在時間帯対応キューどちらも呼んでいない。** |
| `ChatService.process_chat_message`（AI 応答に伴う自動エスカレーション） | `create_escalation` のあと、**スタッフ不在時間帯なら** `OvernightQueueService.add_to_overnight_queue` と自動返信のみ。**不在時間外でもメール送信はない。** |

### 2.2 スタッフ不在時間帯対応キュー側の「通知」は実送信ではない

| 場所 | 内容 |
|------|------|
| `OvernightQueueService.process_scheduled_notifications` | コメントで **`send_escalation_notification` は TODO**。現状は **`queue.notified_at` を立てるだけ**で、スタッフへの実メールは送っていない。 |

### 2.3 既存のメール基盤（流用可能）

| 場所 | 内容 |
|------|------|
| `EmailService` | 認証・クーポン・CSV 申込など Brevo 経由の送信が実装済み。 |
| `notification_service.notify_admin_email_failure` | **運営向け**の失敗通知。施設担当への「新規お問い合わせ」通知とは別。 |
| `Facility` | **`email`** 列あり（施設の連絡先として通知先の第一候補）。 |
| `settings` | `brevo_api_key`, `brevo_sender_email`, `frontend_url` 等。 |

### 2.4 まとめ（ギャップ）

- **A-4 で求める「実送」は、いずれの経路でも未接続。**
- ゲスト手動の `POST /escalate` は **スタッフ不在時間帯対応キューとも無関係**（チャットの AI 起因エスカレーションとの挙動差あり）。

### 2.5 実装反映後の状態（2026-03-26 追記）

上記 §2.1〜2.4 は **2026-03-25 時点の調査メモ**として残す。以下はコード反映後の事実。

- `backend/app/services/escalation_notification_service.py` の `send_staff_escalation_notification` が Brevo 実送と `notified_at` 更新（成功時のみ）を担う。
- `backend/app/services/escalation_absence_routing.py` の `queue_escalation_if_staff_absence` で不在時はスタッフ不在時間帯対応キューへ入れる処理＋自動返信、不在外は呼び出し側が即時送信。
- `ChatService.process_chat_message`・`POST /api/v1/chat/escalate`・`OvernightQueueService.process_scheduled_notifications` から上記が呼ばれる。ゲスト手動経路もスタッフ不在時間帯対応キューと整合。

## 3. 提案する実装方針（大原則に準拠）

### 3.1 単一の「スタッフ向けエスカレーション通知」関数

- **入力**: `escalation_id`（＝受付番号）、DB セッション（施設・会話・必要なら最後のゲストメッセージを読むため）。
- **処理**:
  1. `Escalation` を取得。`facility_id`, `conversation_id`, `created_at` を確認。
  2. 宛先: **`Facility.email`** を基本とする（複数ユーザへ送る要件が出たら別タスクで `User` 一覧を検討）。
  3. 件名・HTML/テキスト本文を組み立てる。**差し込む番号は `escalation.id` のみ。**
  4. Brevo で送信。成功時 **`escalation.notified_at = now`** を commit。
  5. 失敗時 **エラーログ**（上記必須フィールド）。`BREVO_API_KEY` 未設定は **本番では例外または明示的なログレベル error**、開発は警告でスキップなど、親計画・Runbook と整合させる。

**二重送信防止**: 送信成功後にのみ `notified_at` を立てる。再実行・バッチから呼ぶ場合は **`notified_at is None` のときだけ送る**などの冪等性を設計に含める。

### 3.2 呼び出しタイミング（即時と遅延の統一）

| 経路 | 提案 |
|------|------|
| **チャット応答に伴うエスカレーション** | 現状どおり不在時間内はスタッフ不在時間帯対応キューへ。**キューに入れた場合は即時メールを送らない。** 不在時間外は **レコード作成直後に 3.1 を呼ぶ。** |
| **ゲスト手動 `POST /escalate`** | **チャットと同じ不在判定**を共通化し、不在内ならスタッフ不在時間帯対応キューへ追加（現状は未実装のため **ギャップ解消**）、不在外なら **即 3.1**。 |
| **スタッフ不在時間帯対応キュー `process_scheduled_notifications`** | TODO を撤去し、**3.1 と同じ送信処理**を呼ぶ。成功後にキューとエスカレーション双方の通知済み時刻を更新。 |

これにより **「どの入口から作られても、最終的に同じテンプレ・同じ受付番号で届く」**ようにする（親計画の三点一致）。

### 3.3 メール本文に含める URL

- `settings.frontend_url` と `/admin/conversations/{session_id}` など、**既存の管理画面ルート**に合わせる（ハードコードする場合はフロントの `router` 定義と一致確認）。

### 3.4 テスト

- Brevo をモックし、**`notified_at` が成功時のみ更新される**こと、**本文に `escalation.id` が含まれる**ことを pytest で検証。
- スタッフ不在時間帯対応キュー処理からの呼び出し経路のテストを 1 本追加すると安全。

### 3.5 文書・運用

- `docs/サービス開始までの手順_Runbook.md` に **本番の Brevo・施設メールアドレス**の確認項目を追記するか、既存節にリンク。

## 6. 実装順序（要約・この順がおすすめ）

後ろに回すと手戻りが増える順です。

1. **送信の芯だけ先に作る**  
   Brevo で送る関数（仮称でも可）と、**成功時だけ** `escalations.notified_at` を更新する処理、失敗時ログ。  
   **二重送信防止**（例: 既に `notified_at` があるときは送らない）もここで決める。  
   **実装メモ（2026-03-25）**: `backend/app/services/escalation_notification_service.py` の `send_staff_escalation_notification(db, escalation_id)` がこれに相当（チャット等からの呼び出しは未接続）。

2. **チャット応答で自動で記録が付いた直後**（`ChatService.process_chat_message`）  
   スタッフ不在時間外 → **すぐ 1. の送信**。  
   不在時間内 → 今までどおりスタッフ不在時間帯対応キューに入れ、**この時点では送らない**。  
   **実装済み（2026-03-25）**: `queued_for_overnight` が False のときのみ `send_staff_escalation_notification` を呼ぶ。

3. **ゲストの「スタッフに連絡」API**（`POST /chat/escalate`）  
   **2 と同じ不在判定**を共通化し、不在外は即送信、在内はキューへ（いまはどちらも無いので両方足す）。

4. **スタッフ不在時間帯対応キュー処理**（`process_scheduled_notifications`）  
   TODO をやめて **1. と同じ送信**を呼ぶ。キュー側の通知済み時刻も整合させる。  
   **実装済み（2026-03-25）**: `send_staff_escalation_notification` 成功時のみ `overnight_queue.notified_at` を更新。管理 API `POST .../overnight-queue/process` は `processed_count`＝送信成功件数、`total_count`＝時間窓内の未通知候補総数。

5. **テスト・Docker・本番前確認**  
   pytest（モック）、実機で「ゲストの受付番号＝メールに書いてある番号」。  
   **実装済み（2026-03-25）**: `test_escalation_notification_service.py` に受付番号・管理画面 URL を Brevo モックへのアサートを追加。Docker PostgreSQL 上で A-4 関連 pytest を流すスクリプト `backend/scripts/run_a4_tests_with_docker_postgres.sh` と `backend/scripts/README.md` 記載。本番・ステージングの手動確認項目は `docs/サービス開始までの手順_Runbook.md` **§3.3.1**。

6. **Runbook**  
   §3.3.1 に本番前確認を記載済み（Brevo・施設メール・受付番号・FAQ 未解決リスト同定）。

## 7. メールに書く／書かない（要約）

| 書く（必須に近い） | 内容 |
|-------------------|------|
| 受付番号 | **必ず** DB の **`escalations.id`**（ゲスト・管理画面の受付番号と同じ数値だけ）。 |
| 件名 | **推奨**: 件名にも同じ受付番号（受信トレイで探しやすくするため）。 |
| 施設名 | どの施設の連絡か分かるように。 |
| 会話を開く手がかり | **管理画面の会話への URL**（`frontend_url` ＋ 既存の会話詳細パス）。必要なら **セッション ID** を補足で。 |
| 作成日時 | エスカレーション（または記録）の作成時刻。 |

| 書く（運用で決める） | 内容 |
|---------------------|------|
| ゲストの発言 | **方針未固定**。全文転記は個人情報の扱いと相談。**短文の要約**や **管理画面で全文を見る誘導のみ**でも、親計画の「通知で気づける」目的は満たせる。 |

| 書かない（避ける） | 内容 |
|-------------------|------|
| 別の番号 | `session_id` や会話 ID を「受付番号」の代わりに見せない。別採番しない。 |
| 内部向けエラー原文 | スタッフが混乱する生の例外メッセージは本文に直書きしない（ログへ）。 |
| ゲスト向け画面のコピーだけ | 施設担当が何をすればよいか（開く URL・受付番号）が分からない文だけにしない。 |
