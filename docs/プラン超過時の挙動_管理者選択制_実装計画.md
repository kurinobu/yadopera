# プラン超過時の挙動（管理者選択制）実装計画

**作成日**: 2026年3月13日  
**目的**: 要約定義書・アーキテクチャ設計書に記載の「プラン超過時の挙動（管理者選択制）」を、大原則に準拠して実装するための計画書。  
**基準文書**: `docs/Summary/yadopera-v03-summary.md`、`docs/Architecture/やどぺら_v0.3_アーキテクチャ設計書.md`、`docs/20260307_プロジェクト現況と今後の計画_総括.md`

## 0. 目的・全体像・経緯・大原則

### 0.1 目的

月間質問数がプラン上限を超過した場合の挙動を**施設ごとに管理者が選択**できるようにする。

- **通常継続（従量課金）**: 超過後もAI応答を継続し、超過分を従量課金（¥30/質問）で請求する。
- **AI停止・FAQ限定モード**: 超過後はAI応答を停止し、FAQ検索結果のみで応答する（OpenAI 呼び出しなし・Stripe メーター報告なし）。

### 0.2 全体像

| 項目 | 内容 |
|------|------|
| 対象プラン | 月間質問数に上限があるプラン（Free: 30件、Small: 200件、Standard: 500件、Premium: 1,000件）。Mini は従量のみのため**選択対象外**（常に「通常継続」相当）。 |
| 選択肢 | ① 通常継続（従量課金） ② AI停止・FAQ限定モード |
| 保存場所 | 施設（facilities）ごとに 1 件。DB 新規カラムで保持。 |
| 反映箇所 | ゲストチャット API（質問受付時）、管理画面（設定表示・変更）、ダッシュボード（利用状況のステータス表示）。 |

### 0.3 経緯

- 要約定義書 v0.3.4（2025-12-21）で「プラン超過時の挙動の実装：管理者選択制」が Phase 3 残存課題として追加。
- Phase 4 Stripe 実装では「従量課金連携を優先」し、超過時の「AI停止・FAQ限定モード」は**未実装**のまま。
- 現状は**超過後も常にAI応答＋超過分をメーター報告**する動きのみ実装済み。ダッシュボードの `status: faq_only` / `overage` は**表示用**で、チャット側の分岐はない。

### 0.4 大原則（準拠必須）

1. **根本解決 > 暫定解決** … 超過判定・請求期間・施設設定を正に一本化する。
2. **シンプル構造 > 複雑構造** … 既存の `plan_limits`・請求期間・チャットフローを拡張するだけに留める。
3. **統一・同一化 > 特殊独自** … 既存の `dashboard_service` の超過判定・`chat_service` のメーター報告ロジックを流用する。
4. **具体的 > 一般** … エンドポイント・カラム名・判定順を本計画で具体化する。
5. **拙速 < 安全確実** … Docker 環境でテストし、ステージングで検証してから本番反映。
6. **Docker環境必須** … 修正・テストはすべて docker-compose 上で実行する。

## 1. 仕様詳細

### 1.1 用語

| 用語 | 意味 |
|------|------|
| プラン上限 | 請求期間内の「月間質問数」上限。Free: 30、Small: 200、Standard: 500、Premium: 1,000。Mini は NULL（上限なし）。 |
| 超過 | 請求期間内の質問数がプラン上限を超えた状態（今回の質問を含めて判定）。 |
| 通常継続（従量課金） | 超過後も RAG（OpenAI）で応答し、超過 1 問あたり Stripe メーターに 1 件報告する。 |
| AI停止・FAQ限定モード | 超過後は埋め込み＋FAQ 検索のみで応答し、OpenAI は呼ばず、Stripe メーターにも報告しない。 |

### 1.2 管理者選択の扱い

- **Free**: 30 件超過後に「FAQのみ」にするか「AI継続（課金なし）」にするか選択可能。デフォルトは **faq_only**（現行の表示仕様に合わせる）。
- **Small / Standard / Premium**: 超過後に「従量課金でAI継続」か「FAQ限定」を選択可能。デフォルトは **continue_billing**（現行どおり）。
- **Mini**: 上限がないため選択 UI は出さない。実装上は `overage_behavior` は未使用（または常に continue_billing とみなす）。

### 1.3 チャット処理の分岐（概要）

1. 施設取得・言語検証（現行どおり）。
2. セッション取得または新規作成（現行どおり）。
3. ユーザーメッセージを保存（現行どおり）。
4. **【新規】超過判定**: 請求期間内の質問数（今回を含む）がプラン上限を超えており、かつ `overage_behavior == 'faq_only'` の場合 → **FAQ限定パス**へ。
5. **FAQ限定パス**: 埋め込み生成 → `search_similar_faqs`（既存）→ 最良一致の FAQ の該当言語の answer を返す。一致がなければ「該当するFAQが見つかりませんでした。スタッフまでお問い合わせください。」等の固定文。この場合 **Stripe メーター報告は行わない**。応答メッセージを保存して返却。
6. **通常パス**: 現行どおり RAG 実行 → メッセージ保存 → 必要なら Stripe メーター報告（Mini: 全件、Small/Standard/Premium: 超過時のみ）。

### 1.4 ダッシュボードのステータス

- `status` は現行どおり `normal` / `warning` / `overage` / `faq_only` を返す。
- **変更点**: 施設の `overage_behavior` を考慮する。
  - 超過かつ `overage_behavior == 'faq_only'` → `status = 'faq_only'`
  - 超過かつ `overage_behavior == 'continue_billing'` → `status = 'overage'`
- これにより「FAQのみ対応中」バッジ等は、管理者が「FAQ限定」を選んだ場合にのみ表示される。

## 2. コードベース現況

### 2.1 施設（facilities）

- **モデル**: `backend/app/models/facility.py`
- **既存**: `plan_type`, `monthly_question_limit`, `stripe_customer_id` 等。**overage_behavior は未存在**。
- **請求期間**: `plan_started_at` と `backend/app/utils/billing_period.py` の `calculate_billing_period` で算出。

### 2.2 チャット処理

- **入口**: `backend/app/api/v1/chat.py` → `ChatService.process_chat_message`
- **流れ**: 施設取得 → 言語検証 → 会話取得/作成 → ユーザーメッセージ保存 → **`_report_usage_to_stripe_if_needed`**（Mini: 全件、Small/Standard/Premium: 請求期間内件数が上限超過時のみ 1 件報告）→ **RAG 実行**（常に実行、超過時も変更なし）。
- **結論**: 超過時に「FAQのみ」に切り替える分岐は**未実装**。メーター報告の有無のみ、プランと超過で分岐している。

### 2.3 ダッシュボード（月次利用）

- **取得**: `DashboardService.get_monthly_usage`（`backend/app/services/dashboard_service.py`）
- **ロジック**: Free で 30 件超過 → `status = 'faq_only'`、Small/Standard/Premium で上限超過 → `status = 'overage'`。**施設の「選択」は未参照**（表示専用の固定ルール）。
- **スキーマ**: `MonthlyUsageResponse` に `status`, `overage_questions` あり（`backend/app/schemas/dashboard.py`）。

### 2.4 管理画面

- **プラン・請求**: `frontend/src/views/admin/PlanBilling.vue`。現在プラン一覧・変更・解約・請求履歴・領収書のみ。**超過時挙動の設定はなし**。
- **利用状況**: `MonthlyUsageCard.vue` で `status === 'faq_only'` のとき「FAQのみ対応中」等を表示。`overage_behavior` は未使用。

### 2.5 Stripe

- **メーター報告**: `backend/app/services/stripe_service.py` の `report_usage_to_meter`。`chat_service._report_usage_to_stripe_if_needed` から呼ばれる。
- **重要**: Stripe ダッシュボードに「プラン超過時にサービスを止める／継続する」のような**挙動設定は存在しない**。課金は「メーターに送ったイベント」に基づくのみ。したがって「超過時の挙動」は**アプリ側の制御のみ**で実現する。
  - **通常継続**: 超過分もメーターに送る（現行どおり）。
  - **FAQ限定**: 超過後はその質問についてメーターに送らない（OpenAI も呼ばない）。

## 3. Stripe 手動設定について

### 3.1 結論：追加の手動設定は不要

- 「プラン超過時の挙動」は**Stripe の画面や設定では制御しない**。
- Stripe 側で必要なのは、既に Phase 4 で用意済みの以下のみ。
  - 商品・価格（月額・従量 ¥30/質問）
  - メーター（イベント名は環境変数 `STRIPE_METER_EVENT_NAME` と一致）
  - Webhook（subscription / invoice 等）

### 3.2 Stripe で「超過時」にできること・できないこと

- **できること**: 使用量に応じた請求（メーターイベントの回数で課金）、請求書・領収書、アラート（しきい値超えの通知）など。
- **できないこと**: 「〇〇件を超えたらアプリ側でAIを止める」といった**業務ロジックの切り替え**。これはすべてアプリ側の「メーターを送る／送らない」と「RAG を呼ぶ／FAQ のみ返す」で実装する。

### 3.3 実装との対応

| 管理者の選択 | アプリの動作 | Stripe |
|--------------|--------------|--------|
| 通常継続（従量課金） | 超過後も RAG 実行し、超過 1 問ごとにメーターに 1 件送る | 超過分が請求に反映される（既存のとおり） |
| AI停止・FAQ限定 | 超過後は RAG を呼ばず FAQ のみ返し、その質問ではメーターに送らない | 超過分は請求されない |

Stripe ダッシュボードで新たに設定する項目は**なし**。

## 4. 実装ステップ

### Step 0: 共通定義

- **overage_behavior の値**: `continue_billing` | `faq_only`。
- **デフォルト**: Free は `faq_only`、Small/Standard/Premium は `continue_billing`。Mini は設定不要（常に従量＝実質 continue_billing）。

### Step 1: DB 拡張

- **Alembic マイグレーション**で `facilities` に `overage_behavior` を追加。
  - 型: `VARCHAR(20)` または `String(20)`。値は `'continue_billing'`, `'faq_only'`。
  - デフォルト: `'continue_billing'`（一括で後から Free のみ `faq_only` に更新してもよい）。
  - NULL 許容: 非NULL 推奨（既存レコードは `'continue_billing'` で埋める）。
- **Model**: `backend/app/models/facility.py` に `overage_behavior = Column(String(20), default='continue_billing', nullable=False)` を追加。
- **既存データ**: マイグレーションで `plan_type = 'Free'` の施設は `overage_behavior = 'faq_only'` に更新。

### Step 2: バックエンド API

- **施設取得・更新で overage_behavior を扱う**  
  - 管理用の施設取得（例: `GET /api/v1/admin/facility` やダッシュボード用の施設情報）のレスポンスに `overage_behavior` を含める。
  - 施設設定更新 API（例: `PUT /api/v1/admin/facility` や専用の設定 API）で `overage_behavior` を更新できるようにする。バリデーション: `continue_billing` | `faq_only` のみ許可。
- **プラン一覧・現在プラン**  
  - 必要に応じて「現在の overage_behavior」をプラン・請求 API のレスポンスに含め、管理画面で表示する。

### Step 3: チャットサービス（中核）

- **請求期間内の質問数**  
  - 既存と同様に `plan_started_at` と `calculate_billing_period` で請求期間を算出し、`Message` の件数をカウントする（**今回保存したユーザーメッセージを含めた件数**で判定）。
- **上限の取得**  
  - Free: 30、Small: 200、Standard: 500、Premium: 1,000。`facility.monthly_question_limit` が NULL や不整合の場合はプラン種別から上記の値に補正（既存の `_report_usage_to_stripe_if_needed` と同様）。
- **分岐**  
  - `(請求期間内の質問数 > プラン上限)` かつ `facility.overage_behavior == 'faq_only'` の場合:
    1. **Stripe メーター報告は行わない**（`_report_usage_to_stripe_if_needed` を呼ばない、または FAQ 限定パスではスキップ）。
    2. **FAQ 限定応答**:
       - 既存の `generate_embedding` と `search_similar_faqs` を使用（`app.ai.embeddings` / `app.ai.vector_search`）。
       - 最良一致 FAQ の `request.language` に相当する translation の `answer` を返す。複数ある場合は 1 件（例: 類似度最高）を採用。
       - 該当する FAQ が無い、または閾値未満の場合は、固定文言（例: 「該当するFAQが見つかりませんでした。スタッフまでお問い合わせください。」）を多言語対応で返す（既存の `get_fallback_message` や専用メッセージ）。
    3. 返却用の `RAGEngineResponse` 相当のオブジェクトを組み立て（`response`, `ai_confidence`, `matched_faq_ids`, `response_time_ms`, `escalation.needed=False`）、メッセージ保存してから返す。
  - 上記以外は**現行どおり** RAG 実行 → メッセージ保存 → `_report_usage_to_stripe_if_needed`。

### Step 4: ダッシュボード（月次利用）

- **get_monthly_usage** で施設の `overage_behavior` を参照する。
  - 請求期間内質問数がプラン上限を超過している場合:
    - `overage_behavior == 'faq_only'` → `status = 'faq_only'`
    - `overage_behavior == 'continue_billing'` → `status = 'overage'`
  - Free の 30 件超過も同様に、`overage_behavior` で `faq_only` / `overage` を切り替える（現行は一律 `faq_only` だったが、選択制に合わせる）。

### Step 5: 管理画面（フロント）

- **プラン・請求ページ**（`PlanBilling.vue`）または**施設設定**に「プラン超過時の挙動」を追加。
  - 表示条件: 現在のプランが **Free / Small / Standard / Premium** のいずれか（Mini は非表示）。
  - UI: ラジオまたはセレクトで「通常継続（従量課金）」／「AI停止・FAQ限定モード」を選択。保存時に施設更新 API で `overage_behavior` を送る。
  - 文言例:
    - 「プラン超過時: 通常継続（超過分は ¥30/質問で請求）」
    - 「プラン超過時: AI停止・FAQのみ対応（超過分は課金されません）」
- **利用状況カード**（`MonthlyUsageCard.vue`）  
  - 既存の `status === 'faq_only'` 表示でよい。バックエンドが `overage_behavior` を考慮して `status` を返すため、フロントは変更最小でよい（必要なら「設定に従い FAQ のみ対応中」等の説明を追加可能）。

### Step 6: テスト・検証

- **Docker 環境**で以下を実施。
  - 施設の `overage_behavior` を `faq_only` にし、プラン上限を超えた状態でチャット送信 → OpenAI が呼ばれず、FAQ のみの応答になること、かつメーターに送信されないこと。
  - 同様に `continue_billing` のときは超過後も RAG が実行され、メーターに 1 件報告されること。
  - Free で 30 件超過時、`faq_only` のときは FAQ のみ、`continue_billing` のときは AI 応答になること（Stripe は Free では使わないため、メーターは送らないままでよい）。
- **ダッシュボード**: 超過時に `overage_behavior` に応じて `status` が `faq_only` / `overage` になることを確認。
- **管理画面**: 設定変更が保存され、再読み込みで反映されることを確認。

## 5. 注意事項・依存

- **請求期間**: 既存の `plan_started_at` と `calculate_billing_period` に依存。変更しない。
- **Mini プラン**: 上限がないため `overage_behavior` の選択 UI は出さない。実装では `monthly_question_limit is None` のときは「超過」にならないので、分岐に入らない。
- **新規登録時**: 施設作成時に `overage_behavior` をプランに応じてデフォルト設定（Free: `faq_only`、それ以外: `continue_billing`）。
- **既存テスト**: `test_dashboard_service_billing_period.py` 等で `overage_behavior` を考慮した期待値に更新する。
