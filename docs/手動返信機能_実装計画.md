## 3. 原因分析（今回）

1. **検証環境不安定**: ブラウザ初期化タイムアウト/DOM例外で目視確認が崩れた。  
2. **検証導線の混在**: 施設・管理者・セッションの対応固定が遅れ、検証が迂回した。  
3. **報告不足**: API成功とUI成功を明確に分離した報告が遅れた。

## 4. 再実装する場合の推奨計画（主目的優先）

### 4.1 方針（目的に合わせて順序を分離）

- **Phase 1（主目的）**: エスカレーション質問への管理者直接返信を成立させる。  
- **Phase 2（付帯目標）**: 管理者返信をゲストへ確実に届ける導線を追加で成立させる。  
- 返信と解決操作は分離（既存運用に合わせる）。

### 4.2 段階

**Phase 1: 主目的を先に完了（管理者直接返信）**  
- backend: `messages.role=staff` と管理者返信 API（`POST /admin/conversations/{session_id}/reply`）  
- admin UI: 会話詳細に「ゲストに返信」欄を追加  
- 検証: 管理者から返信できること、DB/APIで `role=staff` 保存されること

**Phase 2: 付帯目標（ゲスト到達保証）**  
- guest UI: `staff` 表示、ポーリング、再読み込み導線、注意文  
- 検証: 自動反映または再読み込みで staff 返信を取得できること

**ローカル検証順**  
1. Phase 1の確認（管理者直接返信）  
2. Phase 2の確認（ゲスト到達保証）

**ステージング検証順**  
- `__debug_env` で実行コミット一致確認  
- 対象施設を1つに固定  
- Phase 1 -> Phase 2 の順で再実施

### 4.3 Go / No-Go（段階ごと）

**Phase 1 Go**  
- 管理者直接返信が成立（API/UI/DB一致）

**Phase 2 Go**  
- 保存・取得・表示が一致  
- 自動反映または再読み込みで必ず表示  
- 白画面なし

**Phase 1 No-Go**  
- 管理者返信 API が失敗（4xx/5xx）  
- 会話詳細 UI から返信不能  
- `messages(role=staff)` が保存されない

**Phase 2 No-Go**  
- 白画面/初期化失敗  
- APIにある staff がUI表示されない  
- 既存導線に回帰不具合

### 4.4 判定フロー（固定）

1. まず Phase 1 を完了判定  
2. Phase 1 が Go の場合のみ Phase 2 へ進む  
3. Phase 2 で No-Go が1件でも出たら即取り下げ（revert）  
4. 復旧確認証跡（HEAD/origin/staging）一致でクローズ

## 5. 再発防止ルール

- 「完了」は目視確認まで含めて確定する。  
- API成功のみは「UI未確定」と明示する。  
- PoC中は安全優先で即復旧可能性を最上位に置く。  
- 進行報告は「結論 -> 現状 -> 次アクション」で簡潔に行う。
