# 開発者管理ページ — サービス開始後運営用統計の追加 調査と提案

**作成日**: 2026年3月15日  
**前提**: PoC KPI（目標値）の計測ではなく、**サービス開始後の運営**に活かす統計・数値の取得・閲覧を開発者管理ページに追加する。  
**方針**: 大原則（根本解決・シンプル・統一・具体的・安全確実）に準拠した提案とする。  
**※** 施設一覧への**施設ID**表示追加は本提案に含める。

---

## 1. 視点の整理

| 項目 | PoC期（従来） | サービス開始後運営（本提案） |
|------|----------------|------------------------------|
| 目的 | 検証KPIの達成確認（総質問数100件/月・自動応答率70%等） | 日々の運営・サポート・収益・健全性の把握 |
| 利用者 | PoC評価者 | 開発者（運営者）本人 |
| 主な用途 | 成功基準の測定・報告 | 問い合わせ対応、障害切り分け、請求・解約判断、トレンド把握 |

本提案では、**運営に必要な最小限の統計を既存構造に乗せ、複雑化を避ける**（シンプル構造・統一・同一化）。

---

## 2. 現状の開発者管理ページ

### 2.1 システム概要（overview）

- 総施設数 / アクティブ施設数 / 総FAQ数
- 過去7日のチャット数 / 過去7日のエスカレーション数
- 過去24時間のエラー数（Critical / Error / Warning）

### 2.2 施設一覧（facilities）

- **API**: `id`, name, email, is_active, plan_type, faq_count, chats_7d, errors_7d, last_admin_login を返却済み。
- **画面**: 施設名・メール・状態・料金プラン・FAQ数・チャット（7日）・エラー（7日）・最終ログインを表示。**施設IDは未表示。**

### 2.3 利用可能なデータ（コードベース）

- **Facility**: id, name, email, plan_type, is_active, created_at, subscription_status 等
- **Message**: role='user' の件数 → 質問数。response_time_ms あり（施設別集計可能）
- **Escalation**: 施設別・期間別集計可能
- **GuestFeedback**: 施設別の positive/negative 集計可能（施設管理者ダッシュボードで既に利用）
- **FAQSuggestion**: 施設別・承認数等の集計可能
- **ErrorLog**: 施設別・期間別で既に集計済み

## 3. 大原則との対応

| 大原則 | 本提案での適用 |
|--------|----------------|
| **根本解決 > 暫定解決** | 運営に必要な数値を「開発者ページで一覧できる」形で揃え、都度DBやログを掘らないで済むようにする。 |
| **シンプル構造 > 複雑構造** | 既存の overview / 施設一覧の拡張に留め、新画面・新概念を増やさない。集計は既存モデル・既存APIパターンで行う。 |
| **統一・同一化 > 特殊独自** | 期間は「7日」「30日」「当月」など既存（ダッシュボード・請求期間）に合わせる。施設管理者ダッシュボードで使っている指標と名前・意味を揃える。 |
| **具体的 > 一般** | 追加する項目を一覧で明示し、API・フロントの変更箇所を特定する。 |
| **拙速 < 安全確実** | まず表示系と軽い集計から入れ、重い集計や新テーブルは必要になった段階で検討する。 |

## 4. 提案内容

### 4.1 施設一覧への「施設ID」追加（必須）

- **目的**: 問い合わせ・障害対応・ログ・DB照会時に施設を一意に特定するため。
- **現状**: APIは既に `FacilitySummaryResponse.id` を返している。フロントの表で表示していないだけ。
- **変更**: 施設一覧テーブルの**先頭列に「施設ID」を追加**し、`facility.id` を表示する。
- **CSVエクスポート**: 施設名・メールに加え、**施設ID**列を先頭に追加する（Excel等での照合をしやすくする）。

**対象ファイル（想定）**

- フロント: `frontend/src/views/developer/DeveloperDashboard.vue`（表ヘッダー・td に施設ID列追加）
- バックエンド: `backend/app/api/v1/developer/stats.py` の `export_facilities_csv` で1列目に施設IDを出力

---

### 4.2 システム概要（overview）への追加項目

いずれも既存モデル・既存クエリパターンで取得可能。新テーブル不要。

| 追加項目 | 説明 | 測定方法（案） | 運営での用途 |
|----------|------|----------------|--------------|
| **有料施設数** | プランが Free 以外の施設数 | `Facility.plan_type != 'Free'` の count | 収益基盤の把握、有料転換の推移 |
| **今月の総質問数** | 当月のユーザー発言数 | `Message` の `role='user'` かつ created_at が当月 | 利用トレンド、キャパシティ感 |

- **既存を維持**: 総施設数、アクティブ施設数、総FAQ数、過去7日チャット数、過去7日エスカレーション数、過去24時間エラー数はそのまま。
- **期間の考え方**: 「今月」は暦月（JST）でよい（運営レポートと揃える）。請求期間は施設ごとに異なるため、overview では暦月で統一する。

---

### 4.3 施設一覧への追加列（提案）

| 追加列 | 説明 | 測定方法（案） | 運営での用途 |
|--------|------|----------------|--------------|
| **施設ID** | 上記のとおり | 既存 `id` を表示 | 問い合わせ・ログ・DB対応 |
| **登録日** | 施設の created_at | `Facility.created_at`（日付のみで十分） | 新規獲得推移、在籍期間の目安 |
| **今月の質問数** | 施設別・当月の user メッセージ数 | 施設別・当月で Message を count | 利用度の高い施設の把握、サポート優先度 |
| **エスカレーション数（7日）** | 施設別・過去7日 | 既存の errors_7d と同様、Escalation を施設別・7日で集計 | サポート負荷の偏り、要対応施設の把握 |

- **施設一覧の負荷**: 既存の faq_count / chats_7d / errors_7d と同様、サブクエリまたは集計で取得。既存 `get_facilities_summary` のパターンに揃える（統一・同一化）。
- **フィードバック肯定率**: 施設別に出すとクエリ・表示が重くなりがちなため、**第一次では見送り**とする。必要なら施設管理者ダッシュボードまたは「施設詳細」で確認する形で足りる（シンプル優先）。

---

### 4.4 CSVエクスポートの拡張

- 現在: 施設名・メールアドレスの2列。
- 提案: **施設ID・施設名・メールアドレス**の3列とし、先頭に施設IDを追加。登録日・プラン・今月の質問数等は、必要に応じて後から列追加可能（まずはID追加で統一）。

## 5. 実装の優先度と段階

| 段階 | 内容 | 理由 |
|------|------|------|
| **第1段階** | 施設一覧に施設IDを表示 + CSVに施設ID列を追加 | 要望明示かつ変更範囲が小さい。APIは既に id を返している。 |
| **第2段階** | システム概要に「有料施設数」「今月の総質問数」を追加 | 運営の全体感を把握するため。既存 overview の拡張のみ。 |
| **第3段階** | 施設一覧に「登録日」「今月の質問数」「エスカレーション数（7日）」を追加 | 施設別の利用・負荷を見るため。既存の施設一覧クエリパターンに合わせて集計を追加。 |

## 6. 変更対象ファイル一覧（参照用）

| 種別 | ファイル | 変更内容（案） |
|------|----------|----------------|
| バックエンド | `backend/app/api/v1/developer/stats.py` | overview: 有料施設数・今月総質問数。facilities: 登録日・今月質問数・escalations_7d。export: 施設ID列追加。 |

| フロント | `frontend/src/api/developer.ts` | レスポンス型は types に従うため、特になし（必要なら微修正）。 |

## 7. まとめ

- **PoC KPI**ではなく、**サービス開始後の運営**に役立つ統計に視点を移す。
- **施設ID**は施設一覧・CSVの両方に追加する（必須）。
- **システム概要**には「有料施設数」「今月の総質問数」を追加し、**施設一覧**には「登録日」「今月の質問数」「エスカレーション数（7日）」を追加する案とした。
- 既存のモデル・APIパターンに揃え、新画面や過剰な指標は入れない（大原則：シンプル・統一・具体的）。
- 実装は第1段階（施設ID）→ 第2段階（overview 追加）→ 第3段階（施設一覧の追加列）の順で進めると安全確実である。
