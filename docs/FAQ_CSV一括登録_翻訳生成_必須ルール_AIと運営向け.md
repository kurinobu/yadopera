## 0. まず結論（誤解しやすい点の明文化）

### 0.1 「囲まない」のか？

**いいえ。方針は「囲む」です。**

- CSV で **列ズレを防ぐ最も確実で一般的な方法**は、**区切りと衝突しうるフィールドをダブルクォート `"..."` で囲む**ことです。
- 前回説明で「区切りと衝突しないようにする」と併記したのは、**補助手段**（金額を `1000` と書く、TSV にする等）を列挙したものであり、**「クォートしないでよい」という意味ではありません。**
- **AI が CSV を生成する場合のデフォルト**: **すべてのデータセル（ヘッダー除く）をダブルクォートで囲む（QUOTE_ALL 相当）**ことを推奨する。これにより、英語のカンマ句読点・金額表記・中国語文中の誤った半角カンマ混入にも強くなる。

### 0.2 「¥1,000」のような表記について（運営への影響）

- **半角カンマ `,` を含む文字列**（例: `¥1,000`、`5,000円`）を、**クォートなし**でセルに書くと **列が分裂**し、**中国語が途中で切れ、隣列に次の言語が流れ込む**などの重大不具合になる。
- この表記は **AI が勝手に採用したスタイル**であり、運営が Excel 上で防ぎようがない。**責任は生成側（AI・スクリプト）にある。**
- 以降、本書に従い:
  - **A案（推奨）**: 金額は **`1000円` / `1000日圓` / `¥1000`（カンマなし）** に統一する。  
  - **B案**: `¥1,000` を使う場合は **必ずそのセル全体を `"..."` で囲む**（QUOTE_ALL なら自動で安全）。

## 1. 列ズレの技術的原因（再掲）

- やどぺらの FAQ CSV は **カンマ区切り**として `pandas.read_csv` で読まれる（`csv_parser.py`）。
- **区切り文字は半角カンマ `,`（ASCII U+002C）**。
- **全角読点「，」（U+FF0C）** は区切りにならない。問題になりやすいのは主に:
  - **英語・フランス語などの文の中の半角カンマ**
  - **千の位区切りの `1,000` / `5,000`**
- 中国語「だから列ズレしやすい」の正確な言い方は、**「中国語以外でも半角カンマが入れば同じ」**。**中国語ブロックに千位区切りの半角カンマを書いた場合に顕在化しやすい**、という理解がよい。

## 2. AI・生成スクリプト向け 必須ルール（優先順）

### ルール R1 — クォート方針（最優先）

- **推奨**: CSV 出力時に **`csv.QUOTE_ALL`**（Python）または同等で、**データ行の全セルをダブルクォートで囲む。**
- 手編集で1行だけ書く場合も、**質問・回答など可変長のセルは必ず囲む。**
- セル内に `"` がある場合は **RFC 4180 どおり `""` にエスケープ**する。

### ルール R2 — 金額・数値の半角カンマ

- **原則**: FAQ 本文に **千位区切りの半角カンマを使わない**（`1000円`、`5000円`、`1000 yens` 等）。
- どうしても `1,000` 形式にする場合は **R1（クォート）を必ず満たすこと**。クォート漏れは **重大インシデント**とみなす。

### ルール R3 — 改行

- セル内改行を使う場合は **セル全体をダブルクォートで囲む**（未囲みの改行は行分割になりうる）。

### ルール R4 — 文字コード

- **UTF-8（BOM 付き推奨）** で保存（マニュアル・既存ガイドと整合）。BOM なしでも動くが、Excel 利用者向けには BOM 推奨。

### ルール R5 — 納品前の機械検証（必須）

納品前に **必ず** 次を実行し、**ヘッダーの列数と全データ行の列数が一致**することを確認する。

```bash
python3 -c "
import csv
from pathlib import Path
p = Path('納品するファイル.csv')
with p.open(newline='', encoding='utf-8-sig') as f:
    rows = list(csv.reader(f))
header_n = len(rows[0])
bad = [(i+1, len(r)) for i, r in enumerate(rows[1:]) if len(r) != header_n]
print('header columns:', header_n)
print('OK' if not bad else f'COLUMN MISMATCH rows: {bad[:20]}')
"
```

- `COLUMN MISMATCH` が1行でもあれば **納品禁止**。修正してから再検証。

### ルール R6 — やどぺらのスキーマ準拠

- **必須カラム**: `category`, `language_ja_question`, `language_ja_answer`
- **category** は次のいずれか（小文字）: `basic`, `facilities`, `location`, `trouble`
- **多言語列**（Premium 想定）:  
  `language_en_*`, `language_zh-TW_*`, `language_fr_*`, `language_ko_*`  
  カラム名は `csv_parser.py` の `LANG_PAIRS` と **完全一致**（例: `language_zh-TW_question` のハイフン）。
- **文字数**: 質問 500 文字以内、回答 2000 文字以内（超過はパーサーでエラー）。

詳細は `docs/CSV_bulk_registration_function_implementation_plan/files/FAQ_CSV_format_guide.md` を参照。

### ルール R7 — 翻訳品質（最低限）

- 日本語原文の意味（特に **金額・時間・場所・条件**）を変えない。
- 言語ごとに **誤用しやすい語**を避ける（過去事例: 朝食「1食○円」を韓国語で **「식당」(レストラン)** と訳さない → **「식사」等**）。
- 固有名詞（駅名・店名・Wi‑Fi SSID）は原文・顧客指示を優先。

### ルール R8 — 観光地・固有名詞の表記統一

- 同一行の多言語で **別の観光地を指しているように読める表現**（例: 日本語「江戸城」、フランス語のみ「皇居」寄り）を避け、**顧客確認または統一表記**とする。

### 5.1 Cursor（AI）やこのチャットから、顧客の環境に直接登録できるか？

**原則として「できません」。** 理由は次のとおりです。

- **データベース接続情報（`DATABASE_URL` 等）や Render のシェル権限**がチャット環境に無い。
- **セキュリティ**: 本番・ステージングの接続文字列・パスワードをチャットに貼らない運用とする（漏えいリスク）。
- **同一性**: ステージング用ユーザー（例: `test4@air-edison.com`）は **ステージングDBにのみ存在**し、ローカル Docker の PostgreSQL には無いことが普通である。

**AI（このセッション）でできることの例**

- 顧客向け CSV の **生成・翻訳・形式チェック**（本書 §2・§2 R5）。
- リポジトリ内スクリプト・手順書の **整備**（本節のような記録）。
- **ローカル Docker** で、該当ユーザー・施設が DB に存在するときの **検証用アップロード**（開発者のマシン上）。

**運営（人間）または、DB に到達できるホスト（Render Backend Shell 等）で行うこと**

- 実際の **ステージング／本番への一括登録（INSERT）**。

### 5.2.1 「シェル上のパス」とは何か（方法C・Render Shell を使う前に読む）

手順書に **「シェル上のパスに CSV を置く」「`--csv /path/to/faq.csv`」** とあるとき混乱しやすい。**この節がその用語の正本**である（実装計画書など別文書に分散させない）。

#### （1）シェル（Shell）とは何か

- **シェル**＝コマンドを打ち込む **ターミナル画面**（黒い画面・プロンプトが出る画面）のこと。
- 本書で問題になるのは主に **Render ダッシュボード → 対象の Backend サービス → Shell** で開く画面である。ここは **あなたの Mac や Windows の中ではなく**、**バックエンドが動いている Linux コンテナの中**に接続されている。

#### （2）パスとは何か

- **パス**＝コンピュータ上で **ファイルやフォルダの場所を表す文字列**（いわゆる「住所」）。
- Linux（Render の Backend コンテナもこれ）では、多くの場合 **`/` で始まる絶対パス**で表す。例: `/tmp/faq_upload.csv`、`/app/scripts/foo.py`。

#### （3）「シェル上のパス」とは何か（＝ `--csv` に書いてよい文字列）

- **「シェル上のパス」**＝**いま開いているその Shell が接続先とみなしているマシン（＝コンテナ）の中**で、CSV ファイルが **実際に保存されている場所**を表すパス。
- スクリプトの **`--csv` 引数**に渡すのは **常にこの「コンテナ内の絶対パス」**である。
- **正しさの判定**: 同じ Shell で次を実行し、ファイル名が表示されればそのパスは有効である。

```bash
ls -l /tmp/faq_upload.csv
```

（パスは例。自分が保存した場所に読み替える。）

#### （4）よくある誤解（ここを混同すると一切動かない）

| 誤解 | 事実 |
|------|------|
| 「シェル画面を開いたら、どこかに“パス”と書いてある」 | **表示されない。** パスは **あなたがファイルを保存したときに指定する文字列**（または `curl -o` で指定した保存先）である。 |
| 手元 Mac の `~/Downloads/faq.csv` を `--csv` にそのまま書く | **無効。** それは **Mac 上のパス**であり、Render のコンテナ内にはそのファイルは存在しない。 |
| 「正しいパスは一つに決まっている」 | **決まらない。** コンテナ内の **どこにファイルを置いてもよい**が、**その Shell から `ls` で見える場所**でなければならない。運用では **`/tmp/名前.csv`** を使うと分かりやすい。 |

#### （5）CSV をコンテナ内に置く＝「シェル上のパス」を作ること

Render の Shell に **ファイルアップロード機能がない**場合、コンテナ内にファイルを作る操作が必要になる。いずれも **結果としてコンテナ内の絶対パスにファイルができる**ことが目的である。

- **`curl -fsS -o /tmp/faq_upload.csv "https://..."`** … 取得先 **`/tmp/faq_upload.csv`** がその後の `--csv` に使うパス。
- **`nano /tmp/faq_upload.csv`** で貼り付け保存 … 同上。
- **Docker でローカル検証**する場合: ホストのファイルを **`docker compose cp ./faq.csv backend:/tmp/faq.csv`** のように **コンテナへコピー**し、`--csv /tmp/faq.csv` のように **コンテナ側のパス**を指定する（ホストの `./faq.csv` をそのまま `docker compose exec` 内の `--csv` に書かない。コンテナ内に取り込んだ後のパスを書く）。

#### （6）作業ディレクトリ（`pwd` / `cd /app`）との関係

- **`pwd`** … いま Shell が「ここにいる」と認識している **コンテナ内のフォルダ**（カレントディレクトリ）を表示する。
- **`cd /app`** … 多くのデプロイでアプリのルートが `/app` にあるため、**`python scripts/...` を打つ前に**そこへ移動する、という意味。
- **`--csv` のパス**は **`cd` しなくても絶対パスなら有効**（例: `/tmp/faq_upload.csv`）。相対パス（`./tmp/faq.csv` など）を使う場合だけ、**カレントディレクトリ**に依存するので、`pwd` で確認することが重要。

#### 5.3.0 開発者管理画面からの実行（方法D・主推奨）

1. **環境の確認**  
   ブラウザのアドレスバーで **ステージング**（例: `yadopera-frontend-staging.onrender.com`）か **本番**かを確認する。本番デプロイをしない運用のときはステージングのみで検証する。
2. **開発者ログイン**  
   開発者用パスワードでログイン（`DEVELOPER_PASSWORD`。値は各環境のシークレット参照）。
3. **施設の特定**  
   申し込みメールのログインメール・施設名と、一覧の **施設ID・プラン**を突合。Premium/Standard のみ CSV 一括可（Small 等は拒否される）。
4. **CSV 投入**  
   対象行の「CSV一括登録（FAQ）」→ ファイル選択 → 確認チェック → 実行。成功時は成功件数・処理秒・`uploaded_by`（当該施設ユーザーの ID）が表示される。
5. **（推奨）デプロイ整合**  
   本番に近い確認をする前に、バックエンドで `GET /__debug_env` 等により **デプロイコミット**が想定と一致するか確認する（運用Runbook・実装計画 §4.3 と同趣旨）。

**API（自動化・curl 用の参照）**: `POST /api/v1/developer/facilities/{facility_id}/faqs/bulk-upload`（`multipart/form-data`: `file`, `mode=add`）。フロントの Axios は本処理向けに **60 秒タイムアウト**を設定している（管理画面 CSV と同程度）。

### 5.3.1 代行スクリプトの実行手順（方法C・バックアップ）

**代行申し込みで手に入る情報だけで足りるか**

- **はい（通常）**。スクリプトは **`--user-email` に「やどぺら管理画面にログインするメールアドレス」** を渡すと、DB 上でユーザーを検索し、**そのユーザーに紐づく `facility_id` を自動で決めます**。申し込みフォームに `facility_id` が無くても問題ありません。
- **`--facility-id` は省略可能**です。施設 ID が分かっているときの代替指定（`uploaded_by` はその施設のユーザーの先頭1件）です。

**開発者管理画面の施設一覧との関係**

- 一覧の **施設ID** は、`--facility-id` を使うときの参照用です。
- 一覧の **メールアドレス** は多くの場合 **施設の連絡先メール**（`facilities.email`）であり、**ログインメール（`users.email`）と一致しない**ことがあります。代行履行では **申し込みに書かれたログインメールを `--user-email` に使う**のが正です。
- **プラン（Standard/Premium）・FAQ件数**の確認には施設一覧が便利です。

**前提**

- 対象施設が **Standard または Premium**（それ以外は API／スクリプトとも拒否）。
- 指定したメールのユーザーが DB に存在し、**`facility_id` が紐づいている**こと（未登録・別環境では「ユーザーが見つかりません」等になる）。
- CSV は **UTF-8（BOM 可）**、**§2 R5** で列ズレなし。
- 顧客が既に同趣旨の FAQ を登録していると **重複エラーでトランザクション全体が失敗**しうる。必要なら **FAQ 一覧から削除**するか、CSV から該当行を除く。

#### 5.3.2 具体例: `test4@air-edison.com` で申し込まれた場合（Render Backend Shell・方法C）

以下は **ステージングの Backend サービス**の Shell で実行する想定（本番なら本番の Shell。**サービスを取り違えないこと**）。

1. **環境の確認**  
   Render ダッシュボードで、開いている Shell が **ステージング用 Backend** か、**本番用 Backend** かを確認する。ここを間違えると別 DB に書き込む。

2. **手元で用意した翻訳済み CSV を、§5.2.1 で説明した「コンテナ内の絶対パス」に置く**  
   （＝**その Render Backend の Shell から `ls` すると見える場所**。用語の意味は **§5.2.1 が正本**。）  
   Shell にファイルアップロードが無い場合の例:
   - **一時 URL**: 社内のみアクセス可の場所に CSV を置き、`curl -fsS -o /tmp/faq_upload.csv "https://..."` で取得する。
   - **小さいファイル**: `nano /tmp/faq_upload.csv` や `vi` で貼り付けて保存する。
   - **base64**: 手元で `base64 faq.csv` し、シェルで `base64 -d > /tmp/faq_upload.csv` にパイプする（長いとターミナル制限に注意）。

3. **作業ディレクトリ**  
   コンテナ内でアプリのルートが `/app` のことが多い。`pwd` で確認し、無ければ `cd /app`（リポジトリでは `backend` がデプロイルートに相当する構成）。

4. **ドライラン（必ず先）**

```bash
python scripts/bulk_upload_faq_csv_for_facility.py \
  --user-email "test4@air-edison.com" \
  --csv /tmp/faq_upload.csv \
  --dry-run
```

   出力に **施設名・プラン・CSV行数** が出れば、メールと CSV は受理可能な状態です。

5. **本実行**（問題なければ `--dry-run` を外す）

```bash
python scripts/bulk_upload_faq_csv_for_facility.py \
  --user-email "test4@air-edison.com" \
  --csv /tmp/faq_upload.csv
```

6. **事後確認**（任意）  
   開発者ダッシュボードの施設一覧で **FAQ数** が増えているか、または管理画面にログインして FAQ が見えるかを確認する。

**コマンド引数の整理**

- **`--user-email`（代行の主ルート）**: 申し込みフォームの **ログインメール**と同一にする。`facility_id` は不要。
- **`--facility-id`（補助）**: 施設 ID が分かっているとき。ログインメールが分からない・ユーザー未作成などの例外時に検討。

```bash
# 汎用例: ドライラン
python scripts/bulk_upload_faq_csv_for_facility.py \
  --user-email "顧客のログインメール" \
  --csv /path/to/faq.csv \
  --dry-run

python scripts/bulk_upload_faq_csv_for_facility.py \
  --user-email "顧客のログインメール" \
  --csv /path/to/faq.csv
```

**Docker でローカル検証する場合**（リポジトリルート想定）

- CSV を `backend/tmp/` 等、コンテナ内 `/app/...` から読めるパスに置く。
- `docker compose exec backend python scripts/bulk_upload_faq_csv_for_facility.py ... --csv /app/tmp/faq.csv`

**Render ステージング／本番で実行する場合**

- **対象環境の Backend の Shell** で実行する。`DATABASE_URL` はそのサービスにバインドされている前提のため、**どのサービスの Shell か**が接続先を決める。
- ステージング専用ユーザー（例: `test4@air-edison.com`）は **ステージング DB にのみ存在**しうる。ローカル Docker では同じメールが無いことが多い（§5.1）。

### 5.4 ステージングテストでよく使う例

- 検証用 **`test4@air-edison.com`** は、**`--user-email "test4@air-edison.com"`** だけでよい（`facility_id` を調べる必要はない）。ドキュメント上 **施設例 `facility_id=384`** と書いてあるのは参照用であり、スクリプト実行の必須条件ではない。
- **ローカル Docker の DB には同じユーザーが無い**ため、その検証は **ステージングの Shell** で行う（§5.1 と同じ理由）。

### 5.5 開発者管理画面から CSV を実行する機能（実装済み・2026-03-21）

- **API**: `POST /api/v1/developer/facilities/{facility_id}/faqs/bulk-upload`（`multipart/form-data`、`mode=add`）。コア処理は `FAQService.bulk_create_faqs_from_csv`（施設管理者向け bulk-upload と同一ロジック）。
- **UI**: 開発者ダッシュボード施設一覧の「CSV一括登録（FAQ）」モーダル（`frontend/src/views/developer/DeveloperDashboard.vue`）。
- **監査**: `admin_activity_logs` の `action_type=developer_faq_bulk_upload`（成否・ファイル名・サイズ・SHA256 先頭・開発者 sub 等）。
- **デプロイ**: `develop` → Render ステージング（例: コミット `0cadf4d` で backend staging に反映）。主手順・テスト観点の記録は次を参照。
