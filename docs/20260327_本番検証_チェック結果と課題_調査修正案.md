総括・実装計画で掲げられている意思決定基準に、本書の修正案を揃える。

| 大原則 | 本書での解釈 |

| **根本解決 > 暫定解決** | 表示の誤魔化しではなく、**データ源を一元化**し、上限・言語は **API と UI で同じ規則**にする。 |

## 2. 課題別：事実・原因・大原則に準拠した修正案

### 2.1 施設設定の「対応言語 `en`」と Phase 2 文言

**事実**

- 画面は `facility.languages` を表示し、空なら `en` を表示する。直下の文言は **固定コピー**（「英語のみ」「Phase 2」）。
- バックエンドのプラン定義では Free の対応言語は **`["ja"]`**（`plan_limits.py`）。ゲスト側は別途 `available_languages` を使用。

**問題**

- **管理者向け表示が実態（プラン・DB）とずれる**可能性。利用者が「本当に英語のみか」と誤解する。

**大原則に準拠した修正案（優先順）**

1. **根本（推奨）**: 「対応言語」ブロックを **単一ソースから再構成**する。  
   - 例: 施設設定 API に **正規化済みの `available_languages`（または plan + limits から導出）** を載せ、画面は **それだけ表示**。固定の Phase 2 文は **撤去**し、必要なら **現在のプランと公式ドキュメントへの導線**に置換。
2. **併せて**: `facility.languages` とプランの `languages` の **役割をコードコメントとスキーマで定義**し、将来の二重管理を防ぐ（**統一・同一化**）。
3. **暫定**: Phase 2 文を「管理画面用の古い注釈」として削除または短い事実ベースの文に差し替えるのみ。**単独では不十分**（根本はデータ源の一元化）。

### 2.2 「保存」が反応しないように見える理由

**事実**

- `ゲスト画面にメールを表示する` が ON かつ施設メールがログインメールと同一のとき、**クライアントバリデーションで保存 API を呼ばず return**している。

**問題**

- エラーは **フォーム上部**に出るため、**画面下部の「保存」付近では成功したか失敗したか分かりにくい**。

**大原則に準拠した修正案**

1. **保存クリック時**: バリデーション失敗なら **最初のエラー項目へスクロール**（`scrollIntoView`）、可能なら **フォーカス**。
2. **保存ボタン直近**に、失敗時のみ **要約メッセージ**（例:「上部の入力エラーを修正してください」）を表示。
3. **（より明示的）** ブロック理由が 1 件に絞れるなら **ダイアログ／インラインアラート**で「施設メールとログインメールを分けてください」を再掲。
4. 方針は **モーダル必須ではない**。情報の**具体性**と **操作の最小手数**のバランスで選ぶ（Phase 2 用の巨大モーダルは **シンプル構造**に反しやすい）。

### 2.4 新規 FAQ モーダルの言語が 4 言語固定になる件

**事実**

- `FaqForm.vue` は **en / ja / zh-TW / fr をハードコード**。プラン・`language_limit` と未連動。
- ゲスト用 `SUPPORTED_LANGUAGES` には **zh-CN, es, ko, th** 等もある。**集合が一致しない**。
- バックエンドの `create_faq` は **`language_limit`** を検証するため、**UI では選べそうでも API エラー**になりうる。

**大原則に準拠した修正案**

1. **単一ソース**: 管理画面の選択肢を **`SUPPORTED_LANGUAGES`（または API が返す「当施設で許容する言語コード一覧」）から生成**する。
2. **同一ルール**: ドロップダウンは **`language_limit` と既存 FAQ の言語集合**に基づき、**追加不可の言語は非表示または disabled + 理由ツールチップ**（分かりやすさと統一）。
3. **デフォルト言語**: Free は **初期タブを `ja`** に（英語固定開始をやめる）。**認知負荷とプラン説明の一致**。
4. **サーバ**: 念のため **言語コードのホワイトリスト**を API 側でも共通化（**安全確実**）。

### 2.6 Free プランで「30 件あるのに 31 件目が登録できた」件（新規）

**ユーザー向け文言**: Free は **30 件**などと謳っている一方、**31 件目が手動追加で通る**という報告。

**調査結果（原因）**

1. **`create_faq` に `faq_limit` 超過チェックがない**  
   - `faq_service.py` で `faq_limit` と件数を比較しているのは **CSV 一括登録経路**に限られる。通常の FAQ 作成 API は **DB の上限を見ていない**。
2. **数値定義の二重化・食い違い**  
   - `plan_limits.py` の **`PLAN_FAQ_LIMITS["free"]["max_faqs"]` は 20**。  
   - 一方 **`INITIAL_FAQ_COUNTS["free"]` は 30**（初期自動投入の対象件数）。  
   - 新規施設登録時、`auth_service` は Free の **`faq_limit` を 20** に設定する想定だが、**投入処理は `create_faq` を繰り返すだけ**で上限未検証のため、**実データが 30 件になっても止まらない**動きになりうる。
3. **結果**  
   - **「謳い文句（30）」「初期投入（30 を狙う）」「DB の faq_limit（20）」「API 強制なし」** が同時に存在し、**利用者体験とデータが一致しない**。

**大原則に準拠した修正案**

1. **単一の正**: 「Free の FAQ 上限は ○ 件」を **`plan_limits.py`（または billing の単一モジュール）だけに決める**。マーケ・画面・DB 初期値を **ここから機械的に同期**。
2. **サーバ必須ゲート**: **`create_faq` / `bulk_create` / 更新でインテント数が増える経路すべて**で、`current_count >= faq_limit` なら **409 等で拒否**（**安全確実**）。
3. **既存施設の整合**: 上限導入時に **既に超過している施設**の扱い（締めくくりまで許容／要プランアップグレ／運営調整）を **Runbook に1パターン記載**。
4. **表示**: 管理画面 FAQ に **「現在 ○ / 上限 ○ 件」**を出し、ブロック前に **具体的**にわかるようにする。

## 3. Webhook 入口の稼働確認（再）— 何をどこでやるか

### 3.0 先に事実（あなたの画面とのすり合わせ）

**Stripe は従来の「開発者ダッシュボード」から [Workbench](https://docs.stripe.com/workbench/overview) に移行しており**、Webhook の管理画面も **「イベント送信先（Event destination）」** という体験になっている（公式: [Manage event destinations](https://docs.stripe.com/workbench/webhooks)）。

**2026-03-27 時点のご提供スクリーンショット**では、次のとおりである（見えている範囲の事実のみ記載する）。

- パンくず: **Workbench > YadOPERA**
- 上部タブに **Webhook**、送信先名 **`charming-bliss`（アクティブ）**
- **Endpoint URL** は `https://yadopera-backend-production.onrender.com/api/v1/webhooks/stripe` で、当プロジェクトの想定と一致している
- 右上に **「送信先を編集」** と **三点リーダー（…）** がある
- メイン領域に **概要** と **イベントの配信**（英語 UI なら *Event deliveries* に相当）のサブタブがある
- **その画面のメイン部分には、「テストイベントを送信」「Send test webhook」というラベルのボタンは写っていない**

したがって、**「エンドポイントを開いたらすぐ目に付くボタンでテスト送信」という旧 UI 前提の説明は、現在の Workbench では当てはまらない**。ボタンが見つからないのは **検索ミスというより UI 差分**と解釈するのが妥当である。

**公式ドキュメントが Webhook について書いている操作**は次のとおり（憶測ではなく引用元に基づく）。

- 送信先を選んだうえで **「イベントの配信」（Event deliveries）** で **配信結果・ステータス・再試行**を見る（[同ページ「View event deliveries」節](https://docs.stripe.com/workbench/webhooks)）
- **テスト用にイベントを発生させる**主な手段として、**Stripe CLI** の `stripe trigger …` や **Workbench 下部の Shell**（読み取り専用シェルの有無はアカウント設定による）、または **ダッシュボード上で購読済みのイベント型が実際に発生する操作を行う**ことが [Receive Stripe events… の「Test your handler」/「Triggering test events」](https://docs.stripe.com/webhooks#test-webhook) に書かれている  
  ※ ここには **「概要タブに必ずテスト送信ボタンがある」**という記述はない。

**三点リーダー（…）の中身**はアカウント・権限・Stripe 側の A/B で変わりうるため、**この文書では中身を断定しない**。不明なときは Stripe 画面内ヘルプまたはサポートに「Workbench の送信先でテスト配信を起動する UI はどこか」と問い合わせるのが安全である。

### 3.1 自プロダクト側の確認対象 URL（本番）

本番 Backend のオリジンを `{BASE}` と書く（スクリーンショットでは `https://yadopera-backend-production.onrender.com`）。

- **Webhook エンドポイント**: `{BASE}/api/v1/webhooks/stripe`  
- メソッドは **POST のみ**（実装: `backend/app/api/v1/webhooks/stripe.py`、`/api/v1` + `/webhooks` + `POST /stripe`）。

### 3.2 手順 A：配信一覧で「届いたか」を見る（ダッシュボードのみ・推奨の第一歩）

1. いま開いている **送信先 `charming-bliss`** の画面で、サブタブ **「イベントの配信」** を開く（公式の *Event deliveries* に相当）。
2. **実際に Stripe から POST が飛んだイベント**（例: 本番で発生した決済・サブスク変更など）が一覧に出るか、**Delivered / Failed** と **HTTP ステータス**を確認する。
3. 失敗がある場合、行を開いて **再送（Retry）** が公式に案内されている（[同ページ「Retry sending an event」](https://docs.stripe.com/workbench/webhooks)）。

この方法は **「テスト用の架空ボタン」に依存せず**、**本番またはテストモードで実際に発生したイベントの配送記録**を見る。

### 3.3 手順 B：Stripe が推奨する「テストイベントの起こし方」（Shell / CLI）

公式は **ダッシュボードの単一ボタンより、CLI / Shell での `stripe trigger <イベント型>`** をテストの中心に置いている（[Test your handler](https://docs.stripe.com/webhooks#test-webhook)）。

1. 画面下部の **Stripe Shell** が使える場合、公式例に従い  
   `stripe trigger payment_intent.succeeded`  
   のように、**いまの送信先が購読しているイベント型**に合わせて実行する（購読は右サイドの *Listening to: イベント 5 件* などで確認）。
2. **Shell が読み取り専用で `stripe trigger` が打てない**場合は、PC に [Stripe CLI](https://docs.stripe.com/stripe-cli) を入れ、**同一アカウント・適切なモード（test / live）**で同じコマンドを実行する。

**注意**: `stripe trigger` は **Stripe 側でイベントが生成される**。そのイベントが **登録済み URL（本番 Render）へ配信されるか**は、**その時点のモード（テストキー vs ライブ）と送信先の設定**に依存する。**ライブ Endpoint にテストモードのイベントだけが飛ぶ／飛ばない**のような齟齬はよくあるので、**Event deliveries に行が増えるか**を必ず見る。

### 3.4 手順 C：ブラウザのアドレスバーだけ（補助）

- URL を開くと **GET** になりやすく、当 API は **POST 専用**のため **405** などになりうる。これは **「ルートは存在するが GET は受け付けない」**程度のサインであり、**署名付き POST の成功確認にはならない**。

### 3.5 手順 D（補助）：`curl` で不正 POST

- 署名な省略 POST で **400** などが返ることはありうる（アプリ実装参照）。**必須手順ではない**。

### 3.6 Render ログ

- Render → 本番 **Backend** → **Logs**。Webhook の例外・署名エラー・**503（`STRIPE_WEBHOOK_SECRET` 未設定）** などが出ていないかを、**Event deliveries の時刻**と突き合わせる。

### 3.7 本番 Render の Shell について

- **Render の Shell は YadOPERA の DB / アプリ用**であり、**Stripe から Render の Webhook URL に配信を送る操作は行わない**。Stripe 側（Shell / CLI / 実ビジネスイベント）と **Event deliveries + Render ログ**の組み合わせで確認する。
