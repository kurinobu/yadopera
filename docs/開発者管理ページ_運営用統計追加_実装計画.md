## 1. 目的とご要望の整理

### 1.1 目的

開発者管理ページに、**サービス開始後の運営**に役立つ統計を追加する。  
PoC KPI の計測ではなく、日々の運営・サポート・収益・健全性の把握に使う。

### 1.2 ご要望の3要件と対応する統計

| ご要望 | 対応する統計・指標 | 配置 |
|--------|--------------------|------|
| **料金の妥当性・料金値上げの判断** | 有料施設数、今月の総質問数、施設別「今月の質問数」とプラン上限の関係、ARPU の目安（将来 Stripe 連携で精密化可能） | システム概要 ＋ 施設一覧 |
| **各施設でゲストが活発に利用しているか / やや不活発か** | 施設別：今月の質問数、過去7日のチャット数、過去7日のエスカレーション数。必要なら「活発／やや不活発」の目安ラベル（閾値は固定または後から設定） | 施設一覧 |
| **新規登録・有料プラン移行・ダウングレードの推移が判断できる指標** | 今月の新規登録数、今月の新規登録のうち有料プラン数、解約予定（cancel_at_period_end）施設数。※プラン変更履歴は現行 DB にないため「解約予定数」でダウングレード傾向を補完 | システム概要 ＋ 施設一覧（登録日） |

### 1.3 既存の調査・提案との関係

- `docs/開発者管理ページ_サービス開始後運営用統計_調査と提案.md` の「施設ID」「有料施設数」「今月の総質問数」「登録日」「今月の質問数」「エスカレーション数（7日）」を**すべて実装**する。
- 上記に加え、本実装計画で **「今月の新規登録数」「今月の新規有料数」「解約予定施設数」** をシステム概要に追加する。
- 施設一覧の「エラー（7日）」は現状のまま（errors_7d）。施設別「エスカレーション数（7日）」を**新規列**として追加する（既存の errors_7d は ErrorLog ベースのため別指標）。

## 2. コードベース・環境の現況

### 2.1 開発者管理ページの構成

| 種別 | パス | 役割 |
|------|------|------|
| API | `backend/app/api/v1/developer/stats.py` | GET /developer/stats/overview, GET /developer/stats/facilities, GET /developer/stats/facilities/export |
| スキーマ | `backend/app/schemas/developer.py` | SystemOverviewResponse, FacilitySummaryResponse, FacilityListResponse |
| フロント | `frontend/src/views/developer/DeveloperDashboard.vue` | システム概要カード・施設一覧テーブル・CSV ダウンロード |
| 型 | `frontend/src/types/developer.ts` | SystemOverview, FacilitySummary |
| API クライアント | `frontend/src/api/developer.ts` | getOverview(), getFacilities(), exportFacilitiesCsv() |

### 2.2 利用可能なデータ（DB）

- **Facility**: id, name, email, plan_type, is_active, created_at, cancel_at_period_end, stripe_* 等
- **Message**: role='user' の件数 → 質問数（施設別・期間別集計可能）
- **Conversation**: facility_id → 施設別チャットセッション
- **Escalation**: facility_id, created_at → 施設別・期間別エスカレーション数
- **ErrorLog**: facility_id, created_at → 施設別・期間別エラー数（既存の errors_7d）

※プラン変更履歴テーブルは存在しない。プラン変更は `Facility.plan_type` の更新と Stripe Webhook で反映されるのみ。

### 2.3 Docker 環境・ブランチ戦略

- **大原則**: すべての修正・テストは **Docker 環境**で実行する（拙速 &lt; 安全確実、Docker 環境必須）。
- **ブランチ**: `develop` ＝ ステージング。Render が `develop` を自動デプロイ。
- **検証**: ステージング不整合時はまず `GET /__debug_env` で `RENDER_GIT_COMMIT` を確認する。

## 3. 大原則との対応

| 大原則 | 本実装での適用 |
|--------|----------------|
| 根本解決 &gt; 暫定解決 | 運営に必要な数値を開発者ページで一覧でき、都度 DB やログを掘らないようにする。 |
| シンプル構造 &gt; 複雑構造 | 既存の overview / 施設一覧の拡張に留め、新画面・新概念を増やさない。集計は既存モデル・既存 API パターンで行う。 |
| 統一・同一化 &gt; 特殊独自 | 期間は「7日」「当月（暦月 JST）」など既存（ダッシュボード・請求期間）に合わせる。施設管理者ダッシュボードで使っている指標と名前・意味を揃える。 |
| 具体的 &gt; 一般 | 追加する項目を一覧で明示し、API・フロント・スキーマの変更箇所を特定する。 |
| 拙速 &lt; 安全確実 | 表示系と軽い集計から入れ、重い集計や新テーブルは必要になった段階で検討する。プラン変更履歴は現時点では持たず、「解約予定数」「今月新規登録数」で代替する。 |

## 4. 追加する指標の一覧

### 4.1 システム概要（overview）への追加

| 項目 | 説明 | 測定方法 | 運営での用途 |
|------|------|----------|--------------|
| **有料施設数** | プランが Free 以外の施設数 | `Facility.plan_type != 'Free'` の count | 料金妥当性・収益基盤の把握 |
| **今月の総質問数** | 当月のユーザー発言数 | Message.role='user' かつ created_at が当月（暦月 JST） | 利用トレンド・料金値上げの判断材料 |
| **今月の新規登録数** | 当月に created_at の施設数 | Facility.created_at が当月（暦月 JST） | 新規登録・有料移行・ダウンの推移 |
| **今月の新規有料数** | 当月登録かつ初回から有料の施設数 | Facility.created_at が当月 かつ plan_type != 'Free' | 有料プラン移行の傾向（新規ベース） |
| **解約予定施設数** | 期間末解約予定の施設数 | Facility.cancel_at_period_end == True の count | ダウングレード・解約の傾向 |

既存（維持）: 総施設数、アクティブ施設数、総 FAQ 数、過去7日チャット数、過去7日エスカレーション数、過去24時間エラー数。

### 4.2 施設一覧への追加列

| 列 | 説明 | 測定方法 | 運営での用途 |
|----|------|----------|--------------|
| **施設ID** | 一意識別子 | Facility.id を表示（API は既に返却済み） | 問い合わせ・ログ・DB 対応 |
| **登録日** | 施設の created_at | Facility.created_at（日付のみで十分） | 新規獲得推移・在籍期間の目安 |
| **今月の質問数** | 施設別・当月の user メッセージ数 | 施設別・当月で Message を count | 利用活発度・料金妥当性の判断 |
| **エスカレーション数（7日）** | 施設別・過去7日 | Escalation を施設別・過去7日で集計 | サポート負荷・要対応施設の把握 |

※「活発／やや不活発」ラベルは、第一次では「今月の質問数」「チャット（7日）」の数値を見て**運営者**が判断する形とする。必要なら後から閾値付きラベルを追加可能。  
**運営者**とは、開発者管理ページにログインする人（YadOPERA サービスを運営する開発者・運用担当者）を指す。宿泊施設の管理者（施設管理者）ではない。

### 4.3 CSV エクスポートの拡張

- 現在: 施設名・メールアドレスの 2 列。
- 変更後: **施設ID・施設名・メールアドレス** の 3 列（先頭に施設ID）。登録日・プラン・今月の質問数等は必要に応じて後から列追加可能。

## 7. 技術メモ（実装時）

### 7.1 当月（暦月 JST）の扱い

- サーバーは UTC のため、当月の開始・終了を JST で計算し UTC に変換してクエリする。
- 例: 2026年3月 → JST 2026-03-01 00:00:00 ～ 2026-03-31 23:59:59 を UTC に変換して `Message.created_at` / `Facility.created_at` でフィルタ。

### 7.2 施設一覧のパフォーマンス

- 既存の faq_count / chats_7d / errors_7d と同様、サブクエリまたは JOIN で施設別に「当月質問数」「過去7日エスカレーション数」を取得する。既存の get_facilities_summary のパターンに揃える（統一・同一化）。

### 7.3 解約予定（cancel_at_period_end）

- Facility に cancel_at_period_end カラムが存在する（Phase 4 Stripe 実装で追加済み）。True の施設数を overview で返す。
