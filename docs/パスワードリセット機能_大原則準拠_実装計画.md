# パスワードリセット機能 — 大原則準拠 実装計画

**作成日**: 2026年3月3日  
**前提**: [要約定義書 v0.3](../Summary/yadopera-v03-summary.md)・[アーキテクチャ設計書 10.4 節](../Architecture/やどぺら_v0.3_アーキテクチャ設計書.md) のパスワードリセットフローに準拠。  
**方針**: 大原則に準拠し、**指示があるまで実装は行わない**。本計画は実行順序・変更箇所・検証項目を定義する。

---

## 0. 大原則との対応

| 大原則 | 本実装での対応 |
|--------|----------------|
| **根本解決 > 暫定解決** | 「パスワードを忘れた場合」を問い合わせ対応に頼らず、設計書どおりのリセットフローで根本的に解決する。 |
| **シンプル構造 > 複雑構造** | 既存のメール確認フロー（トークン・有効期限・Brevo）と同じパターンにし、EmailService に 1 メソッド追加・AuthService に 2 メソッド追加に限定する。 |
| **統一・同一化 > 特殊独自** | メール送信は既存 `EmailService`（Brevo・リトライ）を流用。レート制限は `check_resend_rate_limit` と同様の方式（email ベース・60秒）。レスポンス形式は既存 auth スキーマに合わせる。 |
| **具体的 > 一般** | エンドポイント・スキーマ名・ファイルパス・メソッド名・メール件名・URL 形式を本文で具体的に記載する。 |
| **拙速 < 安全確実** | トークンは 1 回限り・有効期限 1 時間。ユーザー不在時も「送信しました」と返し情報漏れを防ぐ。Docker で E2E 確認後にデプロイする。 |
| **Docker環境必須** | 全変更後、Docker でビルド・動作確認を行い、リセット依頼→メール（またはログ確認）→リンク→新パスワード設定→ログインまで検証する。 |

## 1. 仕様サマリー

| 項目 | 内容 |
|------|------|
| **対象** | ログインできない状態で「パスワードを忘れた」ユーザーが、メール経由でパスワードを再設定する。 |
| **DB** | 既存の `users.password_reset_token`・`users.password_reset_expires` を利用（新規マイグレーション不要）。 |
| **依頼** | `POST /api/v1/auth/password-reset`（body: `email`）。該当ユーザーにトークン発行・メール送信。ユーザー不在でも成功レスポンスを返す。 |
| **確定** | `POST /api/v1/auth/password-reset/confirm`（body: `token`, `new_password`, `confirm_password`）。トークン検証・パスワード更新・トークン無効化。 |
| **トークン** | UUID。有効期限 1 時間。使用後は NULL に更新。 |
| **メール** | Brevo 送信。件名「【YadOPERA】パスワードリセットのご案内」。リンク `{FRONTEND_URL}/admin/password-reset/confirm?token={token}`。 |
| **レート制限** | 依頼 API は同一 email で 60 秒に 1 回まで（既存 `check_resend_rate_limit` と同様。別キーでパスワードリセット専用にするか、同一キーで「メール送信系」としてまとめるかは実装時に選択。本計画では「パスワードリセット専用のレート制限」を推奨し、`check_resend_rate_limit` と別変数で管理する）。 |

## 2. 実装ステップ

### Step 1: バックエンド — スキーマ追加

| ファイル | 変更内容 |
|----------|----------|
| `backend/app/schemas/auth.py` | **PasswordResetRequest**: `email: EmailStr` のみ。 |
| 同上 | **PasswordResetConfirmRequest**: `token: str`, `new_password: str`（min_length=8）, `confirm_password: str`（min_length=8）。 |
| 同上 | **PasswordResetResponse**: `message: str`（例: "If an account exists for this email, you will receive a password reset link." または日本語「ご登録のメールアドレスにパスワードリセット用のリンクをお送りしました。」）。依頼・確定の両方で共通のメッセージ用レスポンスとしても可。確定時は "Password has been reset successfully." 等。 |

**統一**: 既存の `ResendVerificationResponse` と同様、`message` を返す形式にする。

---

### Step 2: バックエンド — メール送信

| ファイル | 変更内容 |
|----------|----------|
| `backend/app/services/email_service.py` | **send_password_reset_email(to_email: str, to_name: str, reset_url: str) -> bool** を追加。 |
| 同上 | 件名: `【YadOPERA】パスワードリセットのご案内`。 |
| 同上 | 本文: 既存の `send_verification_email` と同様の HTML/テキスト形式。ボタン・URL・「1時間で無効」の注意。「心当たりがなければ無視してください」の一文。 |
| 同上 | `@retry` は既存と同様（3回・指数バックオフ）を付与。 |
| 同上 | 送信成功時 True、ApiException 等は raise（リトライのため）。 |

**参照**: `send_verification_email` の構造（to, sender, subject, html_content, text_content）をそのまま踏襲する。

---

### Step 3: バックエンド — レート制限（パスワードリセット用）

| ファイル | 変更内容 |
|----------|----------|
| `backend/app/core/rate_limit.py` | **password_reset_attempts: Dict[str, datetime]** を新規追加（`_resend_attempts` と別に管理）。 |
| 同上 | **check_password_reset_rate_limit(email: str, cooldown_seconds: int = 60)** を追加。ロジックは `check_resend_rate_limit` と同一。超過時は 429 と "Please wait {remaining} seconds before requesting another password reset." 等。 |

**統一**: 確認メール再送とパスワードリセット依頼を別キーにすることで、一方の利用が他方の 60 秒制限に影響しないようにする。

---

### Step 4: バックエンド — AuthService

| ファイル | 変更内容 |
|----------|----------|
| `backend/app/services/auth_service.py` | **request_password_reset(db, email: str)** を追加。 |
| 同上 | 処理: (1) `User` を `email` かつ `is_active == True` で検索。(2) いなければログに記録し、成功メッセージを返す（情報漏れ防止）。(3) いれば `uuid.uuid4()` でトークン生成、`password_reset_expires = now + timedelta(hours=1)`、`user.password_reset_token = token`、`user.password_reset_expires = expires`、`db.commit()`。(4) `reset_url = f"{settings.frontend_url}/admin/password-reset/confirm?token={token}"` を組み立て、`EmailService().send_password_reset_email(user.email, user.full_name or user.email, reset_url)` を呼ぶ。Brevo 未設定時はログのみで成功返却（既存の verify と同方針）。(5) 返却は `PasswordResetResponse(message="...")`。 |
| 同上 | **confirm_password_reset(db, token: str, new_password: str, confirm_password: str)** を追加。 |
| 同上 | 処理: (1) `new_password != confirm_password` なら 400。(2) 長さ・強度は既存の change_password と同様（最小8文字）。(3) `User` を `password_reset_token == token` かつ `password_reset_expires > now` で検索。見つからなければ 400「Invalid or expired token」。(4) `user.password_hash = hash_password(new_password)`、`user.password_reset_token = None`、`user.password_reset_expires = None`、`db.commit()`。(5) 返却は `PasswordResetResponse(message="Password has been reset successfully.")` 等。 |

**設定**: `settings.frontend_url` を使用する。未定義の場合は `config` の既存の FRONTEND_URL を確認し、同じキーを使う。

## 4. パスワード見える化の同時実装の調査・検討

### 4.1 調査結果（現況）

| 画面・コンポーネント | パスワード入力 | 見える化 | 備考 |
|----------------------|----------------|----------|------|
| **ログイン** | `LoginForm.vue` で `Input` を `type="password"` で使用 | **なし** | 共通 Input のみ。トグル・アイコンなし。 |
| **新規登録** | `Register.vue` でネイティブ `<input type="password">` を直接使用 | **なし** | Input コンポーネント未使用。 |
| **施設設定（WiFi）** | `FacilitySettings.vue` で `Input` + 外側の「表示/非表示」ボタン | **あり（テキストボタン）** | `showWifiPassword` で type を text/password 切替。アイコンは未使用。 |
| **施設設定（パスワード変更）** | 同上、現在のパスワード・新パスワード・確認の3欄 | **なし** | いずれも `type="password"` のまま。 |
| **パスワードリセット確認** | 計画 Step 8 で新規作成予定の `PasswordResetConfirm.vue` | 未実装 | 新パスワード・確認の2欄を想定。 |

- **共通コンポーネント**: `frontend/src/components/common/Input.vue` は、`type="password"` を受け取るが、見える化用の props・スロット・アイコンは持っていない。エラー時のみ右側にアイコンを表示する構造。
- **既存の見える化パターン**: 施設設定の WiFi パスワードのみ、Input の外側に「表示/非表示」テキストボタンを置き、`:type="showWifiPassword ? 'text' : 'password'"` で切替えている。アイコンは未使用。

### 4.2 同時実装する場合の範囲と方式

**同時実装する場合に揃える対象**

- 新規登録画面：パスワード 1 欄
- ログイン画面：パスワード 1 欄
- パスワードリセット確認画面（新規）：新パスワード・確認の 2 欄
- （任意）施設設定のパスワード変更セクション：現在・新・確認の 3 欄

**実装方式の選択肢**

| 方式 | 内容 | 長所 | 短所 |
|------|------|------|------|
| **A. Input の拡張** | `Input.vue` に `showPasswordToggle?: boolean` を追加。`type="password"` かつ true のとき、右側にトグル用アイコン（目を開く/閉じる）を表示し、クリックで type を text/password 切替。 | 1 コンポーネント修正で、上記 4 画面すべてで同じ UX にできる。大原則の「統一・同一化」に合う。 | Input の責務がやや増える。 |
| **B. 画面ごとにラップ** | 各画面で FacilitySettings の WiFi と同様、Input の外側に「表示/非表示」ボタン（またはアイコンボタン）を配置。 | Input を触らない。 | 同じラップ構造が 4 箇所に重複。アイコンを入れる場合は各所で SVG を書くか共通化が必要。 |
| **C. 専用コンポーネント** | `PasswordInput.vue` を新規作成。内部で Input とトグルを組み合わせ、v-model で値だけ渡す。 | 認証用に特化でき、Input 本体は汎用のまま。 | 新規コンポーネントの追加・保守と、既存 4 箇所の差し替えが必要。 |

**推奨**: **方式 A（Input の拡張）**。認証まわりのパスワード欄を一括で同じ挙動・見た目にでき、「統一・同一化」と「シンプル構造」の両方に合う。既存の施設設定 WiFi は、今後 Input のトグルに統一するか、現状のテキストボタンのままにするかは別判断でよい。

**アイコン**: プロジェクト内に目アイコン用の Heroicons 等は未導入。Input 内にインライン SVG（目を開く / 目を閉じるの 2 種）を 1 セット持たせる実装で足りる。

### 4.3 同時実装するかどうかの結論

| 方針 | 内容 | 推奨度 |
|------|------|--------|
| **推奨: 同時に実装する** | パスワードリセット実装時に、上記 **方式 A** で Input を拡張し、**ログイン・新規登録・パスワードリセット確認**の 3 画面にパスワード見える化（＋アイコン）を入れる。施設設定のパスワード変更セクションは、同じ Input を使っているため、`showPasswordToggle` を付与するだけで同じ UX にできる。 | ◎ |
| 代替: リセットのみ | 新規で作るパスワードリセット確認画面だけ見える化を付ける。ログイン・新規登録は触らない。 | △（認証フロー間で不統一になる） |
| 代替: 別タスク | パスワードリセットは見える化なしで実装し、見える化は後日「認証フォーム UX 改善」で一括対応する。 | △（後で 4 箇所＋共通化を改めて手を入れる必要がある） |

**理由（同時実装を推奨する根拠）**

1. **統一・同一化**: 認証関連のパスワード入力（ログイン・登録・リセット・施設設定のパスワード変更）を同じパターンに揃えられる。
2. **工数**: Input 1 箇所の拡張と、各画面での `showPasswordToggle` 付与（または既存 Input の props 追加）に抑えられる。後から「ログインだけ」「登録だけ」と分けて対応するより、一度で揃えた方が変更箇所が明確でテストもまとめて行える。
3. **パスワードリセット確認画面**: 新規画面なので、最初から見える化付きで設計できる。ここだけ見える化して他はなしにすると、逆に不統一になる。
4. **大原則**: 「シンプル構造」「統一・同一化」に沿う。

**同時実装しない場合の扱い**

- パスワードリセット確認画面は、他と同様 `type="password"` のみで実装する。
- 見える化は別チケット・別計画で、上記 4 箇所＋方式 A を実施する。

### 4.4 同時実装する場合の計画への追記案

本計画に「パスワード見える化」を同時実装する場合、以下を追加する。

- **Step 6 の前または並行**: **Step 5.5 フロント — Input のパスワード見える化**
  - `Input.vue`: props に `showPasswordToggle?: boolean`（default false）を追加。`type === 'password'` かつ `showPasswordToggle === true` のとき、入力欄右側にトグル用ボタン（アイコン: 表示中は「目を閉じる」、非表示中は「目を開く」）を表示。クリックで内部状態 `revealed` を反転し、実際の `type` を `text` / `password` で切替。エラー表示時は、既存どおりエラーアイコンを優先するか、トグルと両立させる（レイアウトを relative で整える）。
  - アイコン: 目を開く / 目を閉じるの SVG を Input.vue 内にインラインで 1 セット定義。
- **LoginForm.vue**: パスワード用 `Input` に `:show-password-toggle="true"` を付与。
- **Register.vue**: パスワード欄を共通 `Input` に差し替え（施設名・メールとスタイルを揃える）、`type="password"` と `:show-password-toggle="true"` を付与。
- **PasswordResetConfirm.vue**（Step 8）: 新パスワード・確認の 2 欄で `Input` に `type="password"` と `:show-password-toggle="true"` を付与。
- **FacilitySettings.vue**: パスワード変更セクションの 3 つの `Input` に `:show-password-toggle="true"` を付与。WiFi パスワードは既存の「表示/非表示」ボタンのままでもよいが、統一するなら同様に Input のトグルに寄せる。
