# CSV 一括登録を「開発者管理画面から実行する」実装計画

**作成日**: 2026年3月21日  
**目的**: 依頼を受けた CSV 一括登録を **開発者管理画面から再現可能に実行できる**よう、API・UI・監査の **実装スコープ**をまとめる。  
**準拠**: 要約定義書 [§0 大原則](Summary/yadopera-v03-summary.md)（とくに **具体的＞一般**、**安全確実＞拙速**）、`開発者によるCSV一括登録代行_手順と方法_20260212.md`（代行＝運営がアップロードまで履行）

**運用・用語（「シェル上のパス」等）の正本**は **本書では扱わない**。`docs/FAQ_CSV一括登録_翻訳生成_必須ルール_AIと運営向け.md` の **§5.2.1** を参照すること。

---

## 1. 背景（なぜこの機能を実装するか）

**2026-03-21 時点で実装・ステージング検証済み。** 代行履行の **第一選択**は **方法D（開発者管理画面から CSV を multipart 投入）** である。従来の **方法C**（`bulk_upload_faq_csv_for_facility.py` を Render Shell 等で実行）は **バックアップ**として残す。Shell・コンテナ・パス・本番／ステージングの取り違えリスクは、方法D により **運営オペから「コンテナ内パス」を不要にできる**（用語整理と方法C の手順は `FAQ_CSV一括登録_翻訳生成_必須ルール_AIと運営向け.md` の §5.2.1・§5.3 に集約済み）。

- 代行契約時に **顧客へ CSV だけ渡して自分でアップロードしてもらう**のは履行にならない、というビジネス要件は **変わらない**。
- 本機能はあくまで **履行のための実行入口をブラウザに寄せる**ための開発タスクである。

## 2. 現状の実装調査（コードベース）

### 2.1 施設管理者向け（既存・本番利用中）

- **API**: `POST /api/v1/admin/faqs/bulk-upload`（`backend/app/api/v1/admin/faqs.py`）
- **認証**: `get_current_user` → **`current_user.facility_id` の施設にのみ**書き込む。
- **フロント**: FAQ 管理のモーダルから `faqApi.bulkUploadCsv`（`frontend/src/api/faq.ts`）が同一エンドポイントを呼ぶ。

### 2.2 開発者向け（2026-03-21 時点）

- **あるもの**: `/api/v1/developer/*` でログイン（`type=developer` の JWT）、施設一覧・統計・エラーログ等に加え、**`POST /api/v1/developer/facilities/{facility_id}/faqs/bulk-upload`**（multipart）と **開発者ダッシュボードの「CSV一括登録（FAQ）」モーダル**（`DeveloperDashboard.vue`、`developer.ts` の 60 秒タイムアウト）。
- **監査**: `admin_activity_logs` の `action_type=developer_faq_bulk_upload`。
- **方法C**: 障害時・緊急時・オフライン検証用の **バックアップ**として `bulk_upload_faq_csv_for_facility.py` を維持。

### 2.3 代行申し込みフォーム

- `POST /api/v1/admin/csv-bulk-request` は **メール転送**まで。登録実行は行わない（`csv_bulk_request.py`）。

### 2.4 コア処理の共通化状況

- 実際の登録ロジックは **`FAQService.bulk_create_faqs_from_csv`**（`faq_service.py`）に集約されている。  
  開発者向け API は **このサービスを呼ぶ薄い層**を追加すればよく、ビジネスルール（プラン・件数上限・言語・CSV パース）は流用できる。

## 3. 設計方針（実装レベル）

1. **実行の入口をブラウザに統一する**  
   開発者が **ステージング用／本番用のフロント URL** にログインし、**施設を特定したうえで CSV を選択 →（ドライランがあれば）本実行**できること。

2. **環境の取り違えを UI で減らす**  
   Shell のサービス選択と同型のミスを減らす（URL・デプロイ運用とセットで文書化）。

3. **multipart で受信**  
   サーバ側で一時ファイルまたはメモリ処理。運営は手元ファイルを選ぶだけにする。

4. **監査と安全**  
   `uploaded_by` の扱い、操作ログ、施設名・ID・プランの再表示・確認チェックで **誤施設登録**を抑える。

5. **方法C の位置づけ**  
   障害時・緊急時・未リリース時のバックアップ。手順・用語は **FAQ 必須ルール文書 §5.2.1・§5.3** を正とする（本書では繰り返さない）。

## 4. 実装計画（フェーズ分割）

### 4.1 推奨実施順（結論）

| 順序 | フェーズ | 説明 |
|------|----------|------|
| **1** | **Phase A** | バックエンド API を先に実装する。**理由**: UI から叩く先が無いと結合テスト・ステージング確認ができない。 |
| **2** | **Phase B** | 開発者ダッシュボードのモーダル・API 呼び出し。**理由**: A の契約（パス・multipart・レスポンス）が固まってから。 |
| **3** | **Phase C** | 監査ログ・権限方針の明文化。**C のうちマニュアル更新（方法D追記）は** 機能リリース **直前〜直後**でよい。 |
| **並行可** | **Phase D** | 必須ルール文書 §5.2・§5.3 の「主手順」を開発者画面第一に差し替える作業。**✅ 完了（2026-03-21）** — `FAQ_CSV一括登録_翻訳生成_必須ルール_AIと運営向け.md` v1.5、本書、代行文書 `開発者によるCSV一括登録代行_手順と方法_20260212.md`、総括を更新。 |

**まとめ**: **Phase A → B → C → D** が完了。**Phase D** はコード変更のブロッカーではなく、**リリース後の運用文言の確定**として実施した。

### 4.2 後続・任意のステップ（本書スコープ外だが計画として記録）

本機能のコア（A〜C）とは別に、コードベース調査で挙がった **任意・後続** は次のとおり（優先度はプロダクト判断）。

| 項目 | 内容 |
|------|------|
| **非同期ジョブ化** | 超大容量 CSV でタイムアウトする場合のみ検討（§6 のリスクと同趣旨）。 |
| **テスト拡充** | Docker 上での統合テスト・権限テストを、金銭・テナンシー境界で厚くする（大原則）。 |
| **開発者ダッシュボードの N+1** | 施設一覧まわりの `prefetch` 等（本機能の UI 追加時に合わせて検討）。 |
| **認可の DRY** | レポート系の `login_required + _can_view_reports` 重複をデコレータ等に寄せる（**別リファクタ**）。 |
| **ドキュメント整合** | コードレビューで、**レポート閲覧権限**の説明（例: docstring）と `utils.can_view_reports` 実装の間に **「staff」表記の齟齬**がないか確認し、必要なら **別タスクで修正**（本機能のブロッカーではない）。 |
| **デザイナー専用 vs `piece_*` ビュー** | ミドルウェアが `/home/piece/` を許可していても、現状 `_can_view_sanad` で 403 になりうる。**製品仕様の整理は本機能スコープ外**だが、将来「デザイナー向けに印刷のみ許可」するなら別計画。 |

### 4.3 テスト・デプロイの推奨順（実務）

**誤解しやすい点**: Phase A〜D を **すべて完了してから** 初めてローカルテスト、という順序 **ではない**。実装フェーズの途中で **都度** ローカル（Docker）検証し、問題なければデプロイしてステージングで最終確認する。

| タイミング | 実施内容 |
|------------|----------|
| **Phase A 実装直後** | **ローカル（Docker）** で API を検証。pytest に加え、**実CSV** を multipart で送る **手動テスト（curl / HTTP クライアント）** があると、列ズレ・プラン拒否・成功件数を早期に確認できる（**任意だが推奨**）。 |
| **Phase B 実装直後** | **ローカル（Docker）** で開発者画面から **実CSV** を選んで実行する **手動テストを必須**とする（UI 経路の本番相当確認）。 |
| **Phase C 実装直後** | 監査ログ・成否が **ローカル**で期待どおりか確認。 |
| **Phase D** | 運用手順・必須ルール文書の更新。**✅ 完了（2026-03-21）**。ステージングで `develop` デプロイ（例: `0cadf4d`）後に FAQ CSV 投入・ログ確認を行い、文言を確定。 |
| **デプロイ** | ローカルで十分に確認後、`develop` を **ステージングへデプロイ**（既存の Render 運用に従う）。 |
| **ステージングテスト** | デプロイ後、**ステージング URL** で **実CSV**（またはステージング用データ・匿名化済み）を使った **手動確認**。総括の「CSV一括登録の手動テスト（開発者代行）」の受け皿。**`GET /__debug_env` でコミット一致を確認してから**実施するとよい。 |

**実CSVを主に使うフェーズ（目安）**

- **API だけ先に試す**: Phase A 後のローカル（任意・推奨）。
- **画面からの本番相当確認**: Phase B 後のローカル（**必須**）。
- **本番に近い環境での最終確認**: **デプロイ後のステージング**（**必須**）。

### Phase A — バックエンド API（最優先）

**エンドポイント案**（命名は実装時に既存ルータ規約に合わせる）:

- `POST /api/v1/developer/facilities/{facility_id}/faqs/bulk-upload`  
  - **認証**: `get_current_developer` 必須。  
  - **Body**: `multipart/form-data`（`file`, `mode=add`）。既存管理画面 API と同形でよい。  
  - **処理**:  
    1. `facility_id` の施設が存在するか。  
    2. `plan_type` が Standard / Premium か（既存と同じ拒否条件）。  
    3. ファイルサイズ上限（既存 10MB と揃える）。  
    4. **`uploaded_by`**: 施設に紐づく代表ユーザー 1 件を採用（現行スクリプトと同様）または **専用システムユーザー**を DB に用意して採用するか、**運用方針を決定**してから実装する。  
  - **レスポンス**: 既存 `BulkUploadResult` スキーマを流用。

**テスト**:

- Docker 上で開発者 JWT を取得し、ステージング相当 DB に対して **ドライラン相当**（まずは本番ロジックと同じトランザクションで、テストではロールバック or テスト施設のみ）を検討。  
- 少なくとも **403（開発者以外）**、**404（施設なし）**、**400（CSV 不正・プラン不適合）** の単体／統合テストを追加。

### Phase B — フロント（開発者ダッシュボード）

- **施設一覧テーブル**に行アクション「CSV 一括登録（FAQ）」を追加（または詳細ドロワーから）。  
- **モーダル内容**:  
  - 施設 ID・施設名・プランの表示（誤操作防止）。  
  - ファイル選択。  
  - **「ドライラン」ボタン**（API が dry-run に対応する場合）または **「内容確認」チェックリスト**（先に `FAQ_CSV一括登録_翻訳生成_必須ルール` の R5 相当を運営が済ませたことを宣言、など）。  
  - 実行結果（成功件数・処理秒・エラーメッセージ）の表示。  
- **本番／ステージング**: 同一ビルドで URL が変わるだけにし、**Shell のサービス選択と同型のミス**を減らす。

※ dry-run を API に持たせる場合は、`FAQService` に読み取りのみの経路を追加するか、既存スクリプトの `--dry-run` と同様の検証をエンドポイントで共有する **小さな内部関数**に切り出す。

### Phase C — 運用・ガバナンス

- **監査ログ**（推奨）: 開発者 ID（sub）、施設 ID、ファイル名（ハッシュのみでも可）、成否、タイムスタンプを DB または構造化ログに残す。  
- **権限**: 現状どおり **開発者パスワードを知る者のみ**。必要なら IP 制限・2FA はインフラ層で検討。  
- **マニュアル更新**: `開発者によるCSV一括登録代行_手順と方法_20260212.md` に **「方法D: 開発者管理画面（推奨）」**を追記し、方法C を「バックアップ」に降格。

### Phase D — ドキュメントの具体化（並行可能）

- `FAQ_CSV一括登録_翻訳生成_必須ルール_AIと運営向け.md` の §5.2・§5.3 の **主手順**を、**本機能リリース後は「開発者管理画面を第一」**に差し替える（方法C はバックアップとして残す）。  
- 用語 **「シェル上のパス」** の正本は **§5.2.1** のまま維持する（方法C を読む人向け）。

## 5. 受け入れ基準（依頼を受けられるライン）

運営担当者が **初日から**次を満たせること:

1. 申し込みメールに記載の **ログイン用メール**と **施設名**から、開発者画面の一覧で対象施設を特定できる。  
2. **ステージング URL** で試し打ちし、問題なければ **本番 URL** で同じ操作を行える（Shell を開かない）。  
3. 失敗時、**API が返す `detail`** と開発者画面の表示から、顧客・社内への説明に使える理由が分かる（プラン、件数上限、重複 intent、列ズレ等）。

## 6. リスクと回避

| リスク | 緩和策 |
|--------|--------|
| 誤施設への投入 | UI で施設名＋ID を明示、確認チェック、直近の監査ログ |
| 本番／ステージング取り違え | フロントの環境ラベル表示（既存パターンがあれば流用）、ブックマーク運用の文書化 |
| 大容量 CSV のタイムアウト | 管理画面と同様 60 秒タイムアウト等を開発者 API でも揃え、必要なら非同期化は後続フェーズ |
| `uploaded_by` の解釈 | スクリプトと同一ポリシーにするか、監査用の固定ユーザーを DB に作るかを **リリース前に文書化** |
