# CSV一括登録 代行申し込み — Formspree 選定事故の原因と再発防止策

**発生日**: 2026年3月13日  
**対象**: CSV一括登録 有料オプション「フロート申し込み機能」（Step 3）  
**結果**: 申し込みフォームで添付ファイルを送信すると Formspree が「File Uploads Not Permitted」で 400 を返し、要件（FAQ内容の添付受信）を満たせず、ロールバックおよび計画の作り直しに至った。

---

## 1. 事故の原因

### 1.1 直接原因

- **申し込みの送信先に Formspree（既存お問い合わせフォームと同じエンドポイント）を採用したが、その Formspree プランではファイルアップロードが許可されていなかった。**
- 計画・実装の**前**に「Formspree の利用プランでファイルアップロードが可能か」を**確認していなかった**。

### 1.2 根本原因

| 要因 | 内容 |
|------|------|
| **要件と手段の検証不足** | 計画書では「FAQ内容のファイル添付」を申し込みの必須要件として記載していたにもかかわらず、選定した送信手段（Formspree）がその要件を満たすかどうかを**仕様・プラン・制限**の観点で検証しなかった。 |
| **「既存に合わせる」の優先** | 「統一・同一化」の原則から「既存のお問い合わせ（Support.vue + Formspree）に合わせる」とし、**新機能の固有要件（添付受信）** と既存手段の**制限**を照合するステップを省略した。 |
| **第三者サービス選定時のチェックリスト不在** | フォーム送信・ファイル受信に外部サービスを採用する際に、「プラン・仕様で当該機能が使えるか」「制限（サイズ・形式・有料化）は何か」を**必ず確認する**という手順が明文化されておらず、確認が抜けた。 |

### 1.3 なぜ気づくのが遅れたか

- 実装は「既存パターンに合わせて Formspree に送信」で進め、**添付付き送信の E2E テスト**を実装完了時点で行っていなかった。
- ユーザーがローカルでブラウザテストして初めて「File Uploads Not Permitted」が判明した。

---

## 2. 再発防止策

### 2.1 第三者サービス・外部APIを選定するときの必須手順

**「送信先」「受信手段」「ファイル/データの扱い」に外部サービス（Formspree、Brevo、Stripe 等）を使う場合、以下を**実装着手前**に実施する。**

| 番号 | 実施項目 | 内容 |
|------|----------|------|
| R1 | **要件の明確化** | 当該機能で「必須で満たす要件」を一覧にする（例: 申し込み内容の受信、**ファイル添付の受信**、件数制限、等）。 |
| R2 | **候補手段の機能・制限の確認** | 候補とするサービスについて、**公式ドキュメントまたはプラン説明**で以下を確認する: (a) 当該機能（例: ファイルアップロード）が**利用可能か**、(b) **無料/有料の別**、(c) **制限値**（ファイルサイズ、件数、形式）。確認結果を**短文で記録**する（例: 「Formspree 無料プラン: ファイルアップロード不可。有料プランで対応。」）。 |
| R3 | **要件と手段の突き合わせ** | 要件一覧と R2 の確認結果を照合し、「選定した手段で要件を満たせるか」を**明示的に判定**する。満たせない場合は、手段の変更（別サービス・自前API）または要件の見直しを行う。 |
| R4 | **記録の残し方** | 上記 R1〜R3 の結論（採用手段と、その手段で要件を満たす根拠）を**実装計画書または設計メモ**に数行でよいので記載する。後から「なぜこの手段にしたか」「添付は可能か」を追えるようにする。 |

### 2.2 本プロジェクトで「フォーム送信・ファイル受信」を扱う場合のルール

| ルール | 内容 |
|--------|------|
| **添付が必要な受信は自前API + 既存メール基盤を優先** | ユーザーがファイルを添付して送る「申し込み」「問い合わせ」は、**添付対応の有無が不明・または制限が厳しい外部フォームサービスに依存しない**。可能な範囲で **バックエンド API（multipart 受信）+ 既存の Brevo 等で運営あてに転送**する設計を優先して検討する。 |
| **「既存に合わせる」は要件を満たす範囲で** | 「統一・同一化」で既存の送信先（例: Formspree）に合わせる場合でも、**当該機能の必須要件（例: ファイル受信）を満たせるか**を先に確認する。満たせない場合は、当該機能だけ別手段（自前API + Brevo）にし、他（例: 通常お問い合わせ）は従来どおりでよい。 |
| **実装完了時に「要件を満たす動作」を確認** | フォーム送信・ファイル送信を実装したら、**実機またはローカルで「送信〜受信側で内容・添付が届く」まで**を一度は確認する。特に「添付付き送信」が要件の場合は、**添付あり**のパターンを必ずテストする。 |

### 2.3 実装計画・設計レビュー時のチェック項目（テンプレート）

新機能で「フォーム送信」「ファイルアップロード」「外部サービスで受信」のいずれかを含む場合、計画・設計段階で以下を確認する。

- [ ] **要件**: 当該機能で「必須」の受信内容（テキストのみ / ファイル添付の要否 / 形式・サイズ）を明文化したか。
- [ ] **手段**: 送信先・受信手段（外部サービス or 自前API）を何にしたか記載したか。
- [ ] **検証**: その手段で、上記要件（特にファイル添付）を**満たせるか**を公式仕様・プランで確認し、結論を 1〜2 行で記録したか。
- [ ] **満たせない場合**: 要件を満たせない手段を選んでいないか。満たせない場合は手段変更または要件見直しをしたか。

## 3. 今回の事故に照らした振り返り

| 再発防止策 | 今回やっていればどうなったか |
|------------|------------------------------|
| R1 要件の明確化 | 「FAQ内容のファイル添付を受信する」が必須であることを一覧にしていれば、次の R2 で「添付」を必ず確認対象にできた。 |
| R2 候補の機能・制限確認 | Formspree のプラン説明で「無料プランではファイルアップロード不可」を確認していれば、当初から Formspree を申し込みの送信先から外せた。 |
| R3 要件と手段の突き合わせ | 「Formspree で添付受信ができるか」を Yes/No で判定するステップがあれば、No の時点で自前API + Brevo を採用できた。 |
| 添付が必要な受信は自前API優先 | 最初から「申し込みは API + Brevo」としていれば、Formspree に依存した実装にはならなかった。 |
| 実装完了時に添付ありでテスト | 実装直後に「ファイルを付けて送信」を一度テストしていれば、本番前に問題に気づけた。 |

## 4. 今後の実装者・AI への指示

- **フォーム送信・ファイル受信を伴う機能**を設計・実装するときは、本ドキュメント **§2.1 必須手順（R1〜R4）** と **§2.2 ルール** に従うこと。
- 実装計画書に「送信先」「受信手段」を書くときは、**「当該手段でファイル添付（またはその他必須要件）が可能か」の確認結果を 1 行でもよいので併記**すること。
