## 1. 「フロート」とは何か

**フロート（float）** とは、本計画・総括で使っている用語で、**画面上で常時または特定画面で表示される「浮いている」UI要素**を指します。

- **具体的な例（本プロジェクト内）**  
  - **ヘルプボタン**: 管理画面の右下に固定表示されている丸いボタン（`HelpButton.vue`）。クリックするとヘルプモーダルが開く。スクロールしても画面上の同じ位置（例: 右下）に留まる。  
  - このように、**ページのレイアウトの流れに埋め込まれるのではなく、画面上の決まった位置に重ねて表示されるUI**が「フロート」です。

- **本計画で言う「フロート申し込み機能」の意味**  
  - 施設管理者が **「CSV一括登録の有料代行を申し込む」** ための**申し込み用のUI**を、画面上に用意する機能全体を指します。  
  - 「フロート」という語は、**申し込み導線を「右下の固定ボタン」のような形で出す案**も含めて議論するために使っています。実装方針としては、**まずはFAQ管理画面内の「代行を申し込む」リンク（CTA）＋専用の申し込みページ（またはモーダル）** で申し込みを受け付け、必要に応じて後から「ヘルプボタンの隣に申し込み用フロートボタンを追加する」形も選べます。  
  - まとめると: **「フロート申し込み機能」＝施設管理者が画面上から有料代行を申し込めるようにする機能**であり、UIの形態は「固定ボタン（フロート）」に限定せず、**申し込みフォームへ誘導する導線＋申し込みフォームそのもの**までを含みます。

### 申し込み時に求めるもの：FAQ内容の添付と言語選択

- **添付ファイル形式**  
  **Excel（.xlsx）**、**CSV（.csv）**、**テキスト（.txt）** を主な受付形式とする。加えて、**Markdown（.md）** を推奨形式の一つとする（Q&Aを見出しやリストで書けば構造が分かりやすく、ChatGPT 等での翻訳にも貼り付けやすい）。  
  - **Excel / CSV**: 表形式のため、そのまままたは簡単な加工で本システムのCSV形式に揃えやすい。  
  - **txt**: 汎用性が高く、区切りを「Q:」「A:」や空行で付けるなどルールを決めておけば変換しやすい。  
  - **Markdown**: 見出し・リストでQ&Aを書いてもらうと、パースしやすく翻訳ツールにも渡しやすい。  
- **事前説明・記載例**  
  施設管理者には、申し込みフォームの前またはフォーム内で、**QとAの記載例**と**注意事項**を必ず読んでもらう。  
  - 「1行または1ブロック＝1組のQ&A」「質問と回答を明確に分ける」「カンマや改行を含む場合は引用符で囲む」など、**受領後にCSVへ変換しやすく、かつ ChatGPT 等で翻訳しやすい書き方**を、短文の例付きで説明する。  
  - テンプレート（Excel/CSV/txt のサンプル）をダウンロードできるようにしておく。  
- **言語の選択**  
  申し込み時点で**希望する言語**（日本語＋英語、日本語＋英語＋繁体中国語 など、プランに応じた組み合わせ）を選択してもらう。これにより受付後に料金・翻訳範囲が確定し、Stripe の決済内容とも揃えやすい。

### 受付後のStripe決済について

申し込み時に **FAQ内容の添付** と **言語選択** を受け取るようにすると、受付・確認の時点で「プラン・件数・言語」が揃い、**料金表（§4.1）に当てはめて金額を一意に決められます**。そのため、**支払い方法を料金表に当てはめて Stripe で決済する**ことが可能になります。

- 流れの例: **2. 受付・確認** で内容・言語を確認し金額を確定 → **3. 見積・決済案内** で Stripe の決済リンク（Payment Link や Checkout セッション）を送付 → 施設管理者が **4. 契約・決済** で Stripe 上で支払い → 入金確認は Stripe の Webhook 等で自動化できる。  
- 現状は Stripe 未連携でも、同じフローで「見積→手動振込案内→入金確認」で運用可能。Stripe 実装後は「見積→Stripe決済リンク→決済完了」に切り替え可能。

### 受注した代行の履行方法について

**受注した代行の履行は、開発者（あなた）がスクリプトを使ってCSVを対象施設に一括登録する方法（方法C）で間違いありません。**  
施設にCSVだけ渡して施設側でアップロードしてもらうのは代行契約の履行ではなく、運営側で **`bulk_upload_faq_csv_for_facility.py` 等のスクリプトを実行し、対象施設の `facility_id` に紐づけて一括登録する**形が正しい履行です（開発者代行手順・plan2 のとおり）。

### 4.4 フロート申し込みを「運用可能」にするための必須設定（メールが届くこと）

**CSV代行申し込みフォームは、運営（あなた）に申し込み内容が届いて初めて意味がある。** 客が申し込んでも運営が何も知らない状態では、実装した意味がない。

- **実装完了の定義**: フロート申し込み機能の「実装完了」＝**コードの実装** ＋ **申し込みメールが運営に届く設定**。どちらかが欠けていれば未完了とする。
- **必須環境変数**（ステージング・本番で申し込みを受け付ける場合）:
  - **`ADMIN_NOTIFICATION_EMAIL`** … 申し込みメールの**送信先**（運営が受信するメールアドレス）。未設定だと API が 503 を返し、メールは送られない。
  - **`BREVO_API_KEY`** … Brevo でメールを送るために必須。未設定だと送信処理でエラーになる。
- **設定しないとどうなるか**: 客がフォームから送信すると「申し込みを受け付けました」と表示されるが、**運営には1通も届かない**。依頼内容を把握できず、仕事を進められない。
- **誰が設定するか**: **運営（あなた）が手動で設定する。** APIキー・受信先メールアドレスは第三者やAIが設定できない。
- **どこに・どう設定するか**: 手順は **`docs/CSV_bulk_registration_function_implementation_plan/CSV一括登録_有料オプション_運用開始チェックリスト.md`** の「環境・設定」内 **「設定方法（運営が手動で行う）」** に記載。ローカルなら `backend/.env` に追記してバックエンド再起動。ステージング・本番なら Render.com のバックエンドサービスの **Environment** で Key/Value を追加。変数名の一覧は `backend/.env.example` を参照。

## 9. CSV代行申し込み 送信先の修正案（Formspree → バックエンドAPI + Brevo）

### 9.1 反省：添付可否の事前調査の欠如

- **経緯**: Step 3 で申し込み送信先を「既存お問い合わせ（Support.vue + Formspree）のパターンに合わせる」として Formspree（`mvzzapae`）を採用した。その際、**Formspree の利用プランでファイルアップロードが許可されているか**を事前に調査していなかった。
- **結果**: ローカルテストで「File Uploads Not Permitted」が発生し、計画書で必須としている「FAQ内容の添付」が届かない事態となった。**添付可否の確認をせずに送信先を決めたことは計画・実装側のミスである。**

### 9.2 調査結果の要約

| 項目 | 内容 |
|------|------|
| **Formspree** | 無料プランではファイルアップロードは不可。有料プラン（Business 等）で初めて添付対応。同一エンドポイント `mvzzapae` でファイル付き送信を許可するにはプラン変更が必要。[^formspree] |
| **Brevo** | 既にアプリで採用済み（認証メール・パスワードリセット・管理者通知等）。トランザクションメール API で **添付ファイル（base64）** をサポート。`SendSmtpEmail` の `attachment` に `[{ "name": "filename", "content": "<base64>" }]` を指定即可。[^brevo] |
| **既存実装** | バックエンドは `admin/faqs.py` で `UploadFile` + `Form()` による multipart 受信、ファイルサイズ 10MB 制限、認証 `get_current_user` を既に使用。`EmailService` は Brevo のみ利用。`admin_notification_email` で運営あてメール送信の宛先を一元管理。 |

[^formspree]: Formspree の無料プランではファイルアップロード非対応。有料で File Upload 機能を利用可能（公式プラン・機能説明による）。  
[^brevo]: Brevo API: Send a transactional email — `attachment` に name と content（base64）を指定。

### 9.3 大原則に準拠した修正方針

| 大原則 | 本修正での適用 |
|--------|----------------|
| **根本解決 > 暫定解決** | 申し込みを「Formspree の制限に合わせて添付を諦める」のではなく、**自前APIで multipart を受信し、Brevo で運営あてに内容＋添付を送る**形にし、要件どおり添付を届ける。 |
| **シンプル構造 > 複雑構造** | 既存の **認証（get_current_user）・multipart（faqs bulk-upload）・EmailService（Brevo）・admin_notification_email** のパターンを流用し、新規ルーター＋1 エンドポイント＋EmailService 1 メソッド追加に限定する。 |
| **統一・同一化 > 特殊独自** | **メール送信は Brevo に統一**。申し込み受信も「バックエンドが Brevo で運営へ転送」する形にし、問い合わせ（Support.vue）は従来どおり Formspree のまま、申し込みのみバックエンド経由とする。 |
| **拙速 < 安全確実** | ファイルサイズ上限（例: 10MB）・拡張子制限（.xlsx / .csv / .txt / .md）をバックエンドで検証する。宛先は既存の `admin_notification_email` とし、未設定時は 503 等で明示し、運用設定漏れを防ぐ。 |

### 9.4 修正内容（実施する変更の一覧）

#### 9.4.1 バックエンド

| 番号 | 内容 | 詳細 |
|------|------|------|
| B1 | **新規ルーター** | `backend/app/api/v1/admin/csv_bulk_request.py` を新規作成。`prefix="/admin/csv-bulk-request"`、タグ `admin`。 |
| B2 | **POST エンドポイント** | `POST /admin/csv-bulk-request`。`multipart/form-data` で受信。項目: `csv_facility_name`, `csv_plan`, `csv_desired_count`, `csv_languages`, `csv_email`, `csv_contact_name`, `csv_notes`（いずれも `Form(...)`）、`csv_faq_file`（`UploadFile = File(None)` で任意）。認証: `Depends(get_current_user)`。 |
| B3 | **プラン制限** | `current_user.facility_id` から施設を取得し、`plan_type in ("Standard", "Premium")` でなければ 403 を返す。 |
| B4 | **ファイル検証** | 添付がある場合: サイズ 10MB 以内、拡張子 `.xlsx`, `.csv`, `.txt`, `.md` のいずれかに限定。違反時は 400。 |
| B5 | **EmailService の拡張** | `EmailService` に `send_csv_bulk_request_email(self, form_data: dict, file_bytes: Optional[bytes] = None, filename: Optional[str] = None) -> bool` を追加。本文は form_data を表形式などで整形した HTML。添付がある場合は Brevo の `attachment` に base64 で付与。宛先は `settings.admin_notification_email`。未設定時は呼び出し元で 503 を返す想定。 |
