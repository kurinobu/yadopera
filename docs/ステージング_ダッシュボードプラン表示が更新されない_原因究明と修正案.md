# ステージング：ダッシュボードのプラン表示が更新されない — 原因究明と修正案

**日付**: 2026-03-11  
**現象**: Premium → Standard → Premium と変更し、プランページでは Premium に更新されているが、ダッシュボードは強制リロード・ページキャッシュ消去・再ログイン後も Standard のまま。  
**指示**: 原因の調査分析と大原則に準拠した修正案の提示。修正は行わない。

---

## 1. 現状の仕様整理

### 1.1 データの流れ

| 画面 | API | データ取得元 | キャッシュ |
|------|-----|--------------|------------|
| プラン・請求 | `GET /admin/plans` | DB の `facility.plan_type` をその場で取得 | なし |
| ダッシュボード | `GET /admin/dashboard` | `DashboardService.get_dashboard_data(facility_id)` | Redis（TTL 300 秒）。キャッシュヒット時は DB を読まず `monthly_usage.plan_type` を含むレスポンスを返す |

### 1.2 既存の無効化実装（前回対応）

- **場所**: `backend/app/api/v1/admin/billing.py`
- **処理**: プラン変更・即時解約の `db.commit()` の直後に `_invalidate_dashboard_cache(facility.id)` を実行。
- **内容**: `cache_key("dashboard:data", facility_id=facility_id)` でキーを組み立て、`delete_cache(key)` で 1 キー削除。

キー形式は `dashboard_service.get_dashboard_data()` の保存時と同一（`cache_key("dashboard:data", facility_id=facility_id)` → 例: `dashboard:data:facility_id=123`）。理論上、無効化で削除しているキーと保存時のキーは一致する。

---

## 2. 無効化後も表示が変わらない原因の候補

### 2.1 想定しうる原因

| 候補 | 内容 | 確認方法 |
|------|------|----------|
| **A. 無効化コードがステージングで動いていない** | ステージングにデプロイされているバックエンドが、キャッシュ無効化を入れる前のビルドのまま。プラン変更時に `delete_cache` が一度も呼ばれていない。 | ステージングのログでプラン変更リクエスト時に無効化処理やログが出ているか確認。またはデプロイ履歴・ブランチの確認。 |
| **B. delete_cache が失敗しているが握りつぶしている** | Redis 接続エラー・タイムアウト・キー不在などで `delete_cache` が False や例外になっていても、呼び出し側で結果を見ておらず、ユーザーには「プラン変更成功」と見える。 | 無効化の前後でログを追加し、削除結果（成否・削除キー）を記録。ステージングでプラン変更後にログを確認。 |
| **C. キー形式のずれ** | 何らかの理由で「保存時のキー」と「削除時のキー」が一致していない（環境別プレフィックス、型の違い、別実装の cache_key など）。 | ステージングの Redis で `SCAN 0 MATCH *dashboard*` 相当を実行し、実際のキー一覧を取得。`cache_key("dashboard:data", facility_id=<当該施設ID>)` の結果と比較。 |
| **D. 複数インスタンス・Redis レプリカ** | レプリカ遅延などで、削除が別ノードにまだ反映されておらず、たまたま古いキャッシュを読んでいる。 | 通常 Render の Redis は単一インスタンスで、同一アプリ内では同じ redis_client を参照するため可能性は低いが、環境次第では要確認。 |

### 2.2 コード上わかっていること

- `dashboard_service` と `billing` は同じ `cache_key("dashboard:data", facility_id=facility_id)` を使用しており、キー文字列は一致する設計。
- 他モジュールでは `delete_cache_pattern` でダッシュボードを消している（例: `overnight_queue_service` は `dashboard:data:*facility_id={id}*`）。billing は `delete_cache( exact_key )` のみで、パターン削除はしていない。
- 無効化の成否をログに出していないため、ステージングで「削除が本当に実行されているか」「Redis で削除されているか」をコードだけでは判断できない。

## 3. ステージングでのコンソールテスト（推奨）

以下をステージング環境で実行し、原因を切り分けることを推奨する。

### 3.1 Redis のキー確認

ステージングの Redis に接続できる場合（Render の Redis インスタンスの接続情報を使用）:

```bash
# Redis CLI でダッシュボード関連キーを一覧
SCAN 0 MATCH *dashboard* COUNT 100
```

- プラン変更**前**にダッシュボードを 1 回開き、上記で `dashboard:data:facility_id=<施設ID>` に相当するキーが存在するか確認。
- プラン変更**直後**に同じコマンドを再度実行し、そのキーが消えているか確認。
  - 消えていれば: 無効化は動いているが、別原因（例: 別キーで保存されている、別キャッシュ層）を疑う。
  - 残っていれば: 無効化が動いていないか、削除に失敗している（A または B）。

### 3.2 バックエンドログの確認

- プラン変更 API（`POST /admin/plans/change`）の処理中に、次のようなログを一時的に追加する:
  - 無効化を呼ぶ直前に「ダッシュボードキャッシュ無効化を実行 facility_id=...」
  - `delete_cache` の戻り値を受け、「削除結果: True/False」をログ出力。
- ステージングでプラン変更を 1 回実行し、上記ログが出ているか・削除結果が True かどうかを確認。
  - ログが出ない → 無効化コードがステージングに載っていない（A）。
  - ログは出るが False → Redis 削除失敗（B または C）。

### 3.3 無効化をパターン削除に変えた場合の挙動

- 一時的に billing の無効化を `delete_cache(key)` の代わりに  
  `delete_cache_pattern(f"dashboard:data:*facility_id={facility_id}*")` に変更し、ステージングで再デプロイ。
- 同じ手順（プラン変更 → ダッシュボード強制リロード）で表示が変わるか確認。
  - 変わる → キー形式のずれ（C）の可能性が高い。
  - 変わらない → 無効化コードが動いていない（A）か、Redis 自体が効いていない可能性。

## 4. 大原則に準拠した修正案

**大原則**: 根本解決 > 暫定対応、統一・同一化 > 特殊な独自実装。

### 案 A（推奨）: プラン表示をキャッシュ対象から外す（根本解決）

**方針**: ダッシュボードの「プラン表示」に必要な `monthly_usage`（`plan_type` 含む）を**キャッシュに含めず、毎リクエストで DB から取得する**。

- **利点**  
  - プラン変更後にキャッシュを消し忘れても、ダッシュボードのプラン表示は常に最新になる。  
  - 無効化タイミングやキー形式に依存しない。  
  - 「プラン・請求とダッシュボードで別ソース（DB とキャッシュ）を見ている」という不整合を設計で解消できる。
- **実装イメージ**  
  - `DashboardService.get_dashboard_data()` を次のように変更する。  
    - キャッシュに保存する payload から `monthly_usage` を**除外**する（またはキャッシュには summary / conversations / overnight_queue / feedback_stats / ai_automation / escalations / coupon_lead_count のみを入れ、`monthly_usage` は含めない）。  
    - レスポンスを返す直前に、**毎回** `get_monthly_usage(facility_id)` を呼び、その結果をマージして返す。  
  - これにより `monthly_usage.plan_type` は常に DB 由来となり、プラン変更が即反映される。  
  - 他項目（週次サマリ、チャット履歴など）は従来どおりキャッシュし、負荷はほぼ維持できる。

### 案 B: 無効化の確実化と観測可能性の向上（暫定強化）

**方針**: 現行の「プラン変更時にキャッシュ 1 キー削除」は維持するが、**無効化を確実にし、原因切り分けしやすくする**。

- **やること**  
  1. **無効化ロジックの集約**  
     - ダッシュボードキャッシュのキー生成・削除は `DashboardService`（またはキャッシュ用共通モジュール）にまとめ、`invalidate_dashboard_cache(facility_id)` のような関数を 1 つにする。billing と overnight_queue の両方からその関数を呼ぶ。  
     - キー形式のずれを防ぎ、統一・同一化に寄せる。  
  2. **削除方法の二重化**  
     - まず `delete_cache(exact_key)` を実行。  
     - 続けて、既存の overnight_queue_service と同様に `delete_cache_pattern(f"dashboard:data:*facility_id={facility_id}*")` をフォールバックで実行し、パターンに一致するキーがあれば削除。  
  3. **ログの追加**  
     - 無効化実行時に対象 `facility_id`、使用したキー（またはパターン）、`delete_cache` の戻り値、`delete_cache_pattern` の削除件数を info または debug で記録。  
     - ステージングで「プラン変更後にログで削除が成功しているか」を確認できるようにする。

### 案 C: パターン削除への統一（キーずれへの耐性）

**方針**: 既存の `overnight_queue_service` と同じく、ダッシュボード無効化は **exact key ではなく pattern で削除**に統一する。

- **やること**  
  - billing の `_invalidate_dashboard_cache` で、`delete_cache(key)` の代わりに  
    `delete_cache_pattern(f"dashboard:data:*facility_id={facility_id}*")` を呼ぶ。  
  - キーにプレフィックスや微妙な違いがあっても、当該施設のダッシュボードキャッシュだけを削除できる。  
  - 必要に応じて、無効化の集約（案 B の 1）とログ（案 B の 3）もあわせて行う。

## 5. まとめ

| 項目 | 内容 |
|------|------|
| **原因の可能性** | (A) ステージングに無効化コードが載っていない、(B) delete_cache が失敗しているが未検知、(C) キー形式のずれ。いずれも現状のコードだけでは断定できないため、ステージングでのコンソールテスト（Redis キー確認・ログ確認・パターン削除の試行）で切り分けることを推奨。 |
| **推奨修正** | **案 A**（monthly_usage をキャッシュ対象から外し、毎回 DB から取得してマージ）を根本解決として採用し、プラン表示の不整合を設計で解消する。 |
| **併用・代替** | 案 A の実装コストや影響を避けたい場合は、まず **案 B + 案 C**（無効化の集約・パターン削除・ログ追加）で無効化を確実にし、ステージングで再現テストする。そのうえで、まだ不整合が続く場合は案 A の導入を検討する。 |

以上。指示があるまで実装は行わない。
