### 0.5 大原則（準拠必須）

1. **根本解決 > 暫定解決** … Stripe のイベントと DB を正しく同期する設計にする。
2. **シンプル構造 > 複雑構造** … サブスクリプション・従量・解約・領収書を Stripe 標準のモデルで扱う。
3. **統一・同一化 > 特殊独自** … 既存の `plan_type` / `plan_limits` / `get_plan_defaults` / 請求期間ロジックを拡張し、Stripe と一貫させる。
4. **具体的 > 一般** … 本計画内でエンドポイント・Webhook・環境変数・テスト手順を具体化する。
5. **拙速 < 安全確実** … ステージングで入念にテストし、本番は別フェーズで切り出す。
6. **Docker環境必須** … すべての修正・テストは Docker（docker-compose）で実行する。

## 2. Stripe 設定（アカウント取得済みを前提）

### 2.1 Stripe ダッシュボードで行う設定（テストモード）

1. **API キー**
   - **テストモード**で「開発者」→「API キー」から取得:
     - 公開可能キー: `pk_test_...`
     - シークレットキー: `sk_test_...`
   - 本番用は本計画では使用しない。

2. **商品（Products）と価格（Prices）**
   - 次の**サブスクリプション用**商品を作成（テストモード）:
     - **Mini**: ¥1,980/月（従量は別メーター）
     - **Small**: ¥3,980/月
     - **Standard**: ¥5,980/月
     - **Premium**: ¥7,980/月
   - **Free**: 無料のため、Stripe 上は「無料の Price」を1つ作るか、または「未契約＝Free」とし、サブスク未作成で扱う（設計で決定）。
   - **従量課金**: 「¥30/質問」用のメーターまたは Price を1つ作成（Stripe Billing の Usage-based または Invoice の Line Item で対応する方式を採用）。

3. **Webhook**
   - ステージング用エンドポイント: `https://<ステージングバックエンド>/api/v1/webhooks/stripe`
   - リッスンするイベント（目安）:
     - `customer.subscription.created`
     - `customer.subscription.updated`
     - `customer.subscription.deleted`
     - `invoice.paid`
     - `invoice.payment_failed`
     - （従量課金を使う場合）`invoice.finalized` やメーター関連イベント
   - 署名シークレット: `whsec_...` を環境変数に格納。

4. **顧客（Customer）**
   - 施設（またはユーザー）1件につき Stripe Customer を1つ作成する方針。施設 ID またはユーザー ID を `metadata` に保持。

### 2.2 環境変数（バックエンド）

| 変数名 | 説明 | 例（テスト） |
|--------|------|---------------|
| `STRIPE_SECRET_KEY` | シークレットキー | `sk_test_...` |
| `STRIPE_WEBHOOK_SECRET` | Webhook 署名検証用 | `whsec_...` |
| `STRIPE_PUBLISHABLE_KEY` | フロント用（Checkout 等で使う場合） | `pk_test_...` |
| `STRIPE_PRICE_ID_MINI` | Mini プラン月額 Price ID | `price_...` |
| `STRIPE_PRICE_ID_SMALL` | Small プラン月額 Price ID | `price_...` |
| `STRIPE_PRICE_ID_STANDARD` | Standard プラン月額 Price ID | `price_...` |
| `STRIPE_PRICE_ID_PREMIUM` | Premium プラン月額 Price ID | `price_...` |
| `STRIPE_PRICE_ID_USAGE_PER_QUESTION` | 従量 ¥30/質問 用 Price ID（メーターに紐づく従量用 Price を1つ作成し、その ID） | `price_...` 等 |

## 3. データベース・バックエンドの拡張

### 3.1 施設テーブル拡張（Alembic マイグレーション）

- `facilities` に以下を追加（案）:
  - `stripe_customer_id` (VARCHAR, nullable, unique): Stripe Customer ID
  - `stripe_subscription_id` (VARCHAR, nullable): 現在のサブスクリプション ID
  - `subscription_status` (VARCHAR, nullable): `active`, `canceled`, `past_due`, `unpaid` 等（Stripe と同期）
  - `cancel_at_period_end` (BOOLEAN, default False): 期間末解約フラグ

既存の `plan_type` / `plan_started_at` / `plan_updated_at` は維持し、Stripe のサブスク変更時にこれらを更新する。

### 3.2 バックエンド構成

- **新規**
  - `backend/app/services/stripe_service.py`: Stripe API 呼び出し（Customer 作成、Subscription 作成/更新/解約、Invoice/Receipt 取得）、エラーハンドリング。
  - `backend/app/api/v1/webhooks/stripe.py`: Webhook 受信、署名検証、イベント種別ごとの処理（subscription created/updated/deleted、invoice paid/failed 等）、DB 更新。
  - `backend/app/api/v1/admin/billing.py`（または `subscription.py`）: 管理画面用 API（プラン一覧、変更、解約、領収書一覧/URL）。
- **既存拡張**
  - `auth_service`: 新規登録時に Stripe Customer 作成＋無料または有料サブスク開始（設計に応じて）。
  - `facility_service` または新規 `billing_service`: プラン変更時に `get_plan_defaults` と Stripe サブスク更新を一括実行、`plan_updated_at` 更新。
  - `dashboard_service`: 既存の請求期間ロジックはそのまま利用し、従量課金の「今月の質問数」は Stripe メーターまたは自前集計のいずれかで整合させる。

### 3.3 API 設計（案）

| メソッド | パス | 説明 |
|----------|------|------|
| GET | `/api/v1/admin/plans` | 変更可能なプラン一覧（現在プラン・他プラン・価格・制限値）。 |
| POST | `/api/v1/admin/plans/change` | プラン変更（アップグレード/ダウングレード）。Body: `target_plan_type`。Stripe 更新 + DB の `plan_type`, `plan_updated_at`, 制限値更新。 |
| POST | `/api/v1/admin/subscription/cancel` | 解約（即時または期間末）。Stripe で `cancel_at_period_end` または即時解約 + DB の `subscription_status` 等更新。 |
| GET | `/api/v1/admin/invoices` | 請求履歴一覧（Stripe Invoice 一覧を返すか、DB キャッシュ）。 |
| GET | `/api/v1/admin/invoices/{invoice_id}/receipt` | 領収書 URL または PDF 取得用 URL（Stripe の Receipt/Invoice PDF へのリンク）。 |

※ Free プランは「サブスクなし」として扱う場合、解約時は Stripe サブスク削除＋`plan_type=Free` にする等、設計で統一する。

## 4. 解約ボタン実装

### 4.1 仕様

- 管理画面（例: 設定または「プラン・請求」専用ページ）に「解約する」ボタンを配置。
- クリック時: 確認モーダル（「期間末で解約」「即時解約」の選択があれば表示）→ 実行後、Stripe でサブスク解約（または `cancel_at_period_end`）し、Webhook で `subscription.deleted` / `updated` を受けて DB を更新。
- 解約後: `plan_type` を Free にするか、`subscription_status=canceled` のみにして「利用不可」とするかは仕様で決定。既存の QR・FAQ は調査報告どおり継続利用可能とする。

### 4.2 フロント

- 解約ボタンと確認ダイアログを追加。API は `POST /api/v1/admin/subscription/cancel` を呼ぶ。
- 成功時は「プラン・請求」またはダッシュボードにリダイレクトし、トーストで完了表示。

## 5. 領収書発行機能

### 5.1 仕様

- Stripe の Invoice は自動で領収書（Receipt）が紐づく。管理画面から「請求履歴・領収書」一覧を表示し、各請求について「領収書を表示」リンクで Stripe の Hosted Invoice Page または Receipt URL を開く。
- 必要に応じて「PDF をダウンロード」は Stripe の Invoice PDF リンクをそのまま利用する。

### 5.2 実装

- `GET /api/v1/admin/invoices`: Stripe の `Invoice.list(customer=stripe_customer_id)` を返す（または DB にキャッシュ）。
- `GET /api/v1/admin/invoices/{id}/receipt`: 該当 Invoice の `hosted_invoice_url` または `invoice_pdf` を返す。フロントは別タブで開くか、PDF ダウンロードリンクとして表示。

## 6. 料金プランのアップグレード・ダウングレード

### 6.1 仕様

- 管理画面に「プラン変更」画面または設定内セクションを設け、現在プランと他プラン（価格・FAQ数・月間質問数・言語数）を表示。
- 「このプランに変更」で `POST /api/v1/admin/plans/change` を呼ぶ。バックエンドで:
  1. Stripe のサブスクを変更（新しい Price にスイッチ、proration は Stripe 任せ）。
  2. `get_plan_defaults(target_plan_type)` で `monthly_question_limit` / `faq_limit` / `language_limit` を取得し、施設の `plan_type`, `subscription_plan`, 各制限値, `plan_updated_at` を更新。
- 既存 QR・FAQ は変更しない（料金プラン変更とQR・FAQ継続利用の調査報告どおり）。

### 6.2 ダウングレード時の注意

- 既存 FAQ 件数が新しい `faq_limit` を超える場合でも、既存 FAQ は削除せず「新規追加のみ新上限で制限」とする（現行仕様を維持）。
- Stripe 側は「次回請求日から新プラン」とするか、即時変更とするかはビジネス判断で決定し、ドキュメントに明記する。
