# 2026-03-24 ステージングDB全消失インシデント 原因調査報告

## 1. 事象概要

- 事象: ステージング開発者ページの施設一覧が 0 件表示になった。
- 影響: PoC 実行中のステージングデータ（users/facilities/faqs/messages 等）が消失。
- 検知: 2026-03-24、Owner からの報告で発覚。

---

## 2. 事実確認（検証結果）

- Railway 接続先 `postgresql://...@tramway.proxy.rlwy.net:50673/railway` で件数確認:
  - `users=0`, `facilities=0`, `faqs=0`, `conversations=0`, `messages=0`（事故発覚時）。
- その後、同接続先へ `pg_restore` を実施し復旧:
  - `users=27`, `facilities=28`, `faqs=629`, `conversations=1154`, `messages=2490`。
- ステージング API 実測:
  - `GET /api/v1/health` -> `200`
  - `POST /api/v1/developer/auth/login` -> `200`
  - `GET /api/v1/developer/stats/facilities` -> `200` / `facilities_count=28`

---

## 3. 根本原因（RCA）

### 3.1 直接原因

- `TRUNCATE ... RESTART IDENTITY CASCADE` を、`tramway.proxy.rlwy.net:50673/railway` 接続先に実行した。
- 当該接続先は実運用上、ステージング（PoC）で利用していた DB だったため、ステージングデータが全削除された。

### 3.2 なぜ起きたか（寄与要因）

1. **環境名と接続先の同一視**
   - 「本番データ削除」の意図で実行したが、実際には Railway の同一 DB URL を本番/ステージングで運用していた。
2. **破壊系SQL前の接続先検証不足**
   - `TRUNCATE` 実行前に「この DB は本当に本番専用か」を機械的にブロックする手順がなかった。
3. **Railway Hobby のバックアップ非対応**
   - プラットフォーム側復元ポイントに依存できず、手元 `pg_dump` が唯一の復旧手段だった。

## 5. 再発防止策（必須）

### 5.1 破壊系SQLの実行ゲート（新ルール）

`TRUNCATE/DELETE/DROP` 実行前に以下を必須化する。

1. 接続先指紋確認（実行ログ保存）
   - `select current_database(), inet_server_addr(), inet_server_port();`
2. 主要件数の事前採取
   - `users/facilities/faqs/messages` の `count(*)`
3. Owner による 2段階承認
   - 「接続先ホスト」「対象テーブル集合」をそれぞれ明示承認
4. 直前バックアップ取得
   - `pg_dump -Fc` を同日取得し、サイズ確認まで必須

### 5.2 環境分離の運用ルール

- 「本番」「ステージング」の論理区分だけでなく、**DB インスタンス単位で分離**する。
- 共有 DB を使う場合は、破壊系SQLを禁止し、論理削除または限定削除のみ許可する。
