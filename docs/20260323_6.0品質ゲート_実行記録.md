## いつ・どこでコマンドを実行するか

**はい。手元のターミナルで実行して問題ありません。**

前提は次のとおりです。

1. **リポジトリのルート**（`docker-compose.yml` があるディレクトリ）に `cd` する。  
2. **Docker Desktop（または同等）が起動**しており、`docker compose` で Backend / Frontend / Postgres 等が立ち上がっていること（例: `docker compose up -d`）。  
3. コマンドは **ホストのシェル**から `docker compose exec ...` でコンテナ内を叩く形が本記録の前提（CI と揃えやすい）。

CI や別マシンでも同じ環境変数と DB 前提が揃えば、**同じコマンドで再現**できます。

## 1. バックエンド pytest（CI 同等・SQLite / メモリ DB）

**条件**: `USE_POSTGRES_TEST=false`・`USE_OPENAI_MOCK=true`（GitHub Actions `staging-deploy.yml` の `test` ジョブと整合）

```bash
cd /path/to/yadopera

docker compose exec -e USE_POSTGRES_TEST=false -e USE_OPENAI_MOCK=true backend pytest tests/ -q
```

## 2. バックエンド pytest（PostgreSQL + pgvector・Docker 内 Postgres 向け）

Compose サービス名が `postgres` のとき、**テスト専用 DB** を一度だけ用意する（既にあればスキップ可）。

```bash
cd /path/to/yadopera

docker compose exec postgres psql -U yadopera_user -d postgres -tc \
  "SELECT 1 FROM pg_database WHERE datname = 'yadopera_test'" | grep -q 1 \
  || docker compose exec postgres psql -U yadopera_user -d postgres \
       -c "CREATE DATABASE yadopera_test;"
```

**注意**: ホストから `localhost:5433` を指定するデフォルト URL では、**コンテナ内の backend** からは Postgres に届かない。Compose 上では **`postgres` ホスト名**と **5432** を明示する。

```bash
docker compose exec -e USE_POSTGRES_TEST=true -e USE_OPENAI_MOCK=true \
  -e TEST_DATABASE_URL=postgresql+asyncpg://yadopera_user:yadopera_password@postgres:5432/yadopera_test \
  backend pytest tests/ -q
```

**本記録時点の結果**: **159 passed**, **3 skipped**, **51 warnings**（約 32.5s。再現コマンドは §2 参照）

**補足（conftest）**: テスト終了時に `drop_all` するため、ステージングの既存 DB を指す `TEST_DATABASE_URL` では**データが消える**。ステージングを汚さずに調査だけしたい場合は `PRESERVE_POSTGRES_TEST_DB=true` を付与（実装計画・`backend/tests/conftest.py` の意図に従うこと）。
