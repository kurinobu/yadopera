## 1. 目的・スコープ・経緯

### 1.1 目的

- **5a**: 簡体中国語（zh-CN）・スペイン語（es）を実装に追加し、ゲストが選択・チャットで利用できるようにする。
- **5b**: ゲスト画面の言語選択からドイツ語（de）・ベトナム語（vi）を削除する。

### 1.2 スコープ

| 項目 | 内容 |
|------|------|
| 追加言語 | 簡体中国語（zh-CN）、スペイン語（es） |
| 削除（ゲスト画面のみ） | ドイツ語（de）、ベトナム語（vi）の選択肢を表示しない |
| 既存維持 | 日本語（ja）、英語（en）、繁体中国語（zh-TW）、フランス語（fr）、韓国語（ko）は変更なし |
| プランと言語 | Free=1（ja）、Mini=2（ja,en）、Small=3（ja,en,zh-TW）、Standard=4（ja,en,zh-TW,fr）、Premium=対応言語すべて（zh-CN・es を追加し、de・vi は含めない） |

### 1.3 経緯・現況

- 2026-02-13 に繁体中国語・フランス語・韓国語の3言語追加をデプロイ済み。FAQ プリセット 30 件×5 言語、プラン別言語フィルタ、ゲスト言語選択・チャット対応が動作している。
- **2026-03-11**: 計画「5」を実装・develop マージ・ステージングデプロイ完了。ステージングで RENDER_GIT_COMMIT=7a828ca を確認し、Premium 施設の `GET /api/v1/facility/{slug}` が 7 言語（ja, en, zh-TW, zh-CN, fr, ko, es）を返すこと、ゲスト画面 `/f/{slug}` で 7 枚の言語カードが表示され de・vi が表示されないことをブラウザ検証済み。総括 doc の「5. 多言語」は **✅ 完了** に更新済み。
- 料金プラン表（総括 §4）では、Premium は「無制限」言語のため、対応言語リストの更新のみでよい。Small/Standard の言語構成（日・英・中繁・仏）は変更しない。

### 3.1.1 ステージング実行状況の観測（根本原因の証拠・経過）

- **調査時点（修正前）**: ステージングの `GET /__debug_env` で `RENDER_GIT_COMMIT=ff3fd411...` を確認。同コミットのバックエンドは Premium の `available_languages` を 5 言語で返しており、公開 API 実測でも 5 言語だった。
- **結論（当時）**: 根本解決は **多言語_5 を `develop` にマージしステージングへ再デプロイ**である（フロントのPremiumフォールバックは不採用）。
- **実施後（2026-03-11）**: 上記の通り実装・マージ・再デプロイを実施。ステージングで `RENDER_GIT_COMMIT=7a828ca...` を確認し、Premium の `available_languages` が 7 言語で返ること、ゲスト画面で 7 枚の言語カード表示をブラウザで確認済み。

### 3.2 既存の「Premium 全言語」リスト

- `facility_service.py`（約153行）: `available_languages = ["ja", "en", "zh-TW", "fr", "ko"]`
- `chat_service.py`（約85行）: 同上

**変更方針**: 上記を **["ja", "en", "zh-TW", "zh-CN", "fr", "ko", "es"]** に更新する（zh-CN・es を追加、de・vi はもともと含まれておらず、今後も含めない）。ゲスト画面はバックエンドが返す `available_languages` のみ表示するため、フロントの `SUPPORTED_LANGUAGES` から de・vi を外し zh-CN を追加すれば、選択肢として de・vi は出なくなる。

### 3.3 FAQ プリセット構造

- `backend/app/data/faq_presets.py`: 30 件の FAQ。各件に `translations` 配列で ja, en, zh-TW, fr, ko の5言語。
- `filter_faq_presets_by_plan`（plan_limits.py）でプラン別に言語フィルタ済み。Premium は `allowed_languages is None` で全翻訳を投入。
- **追加作業**: 30 件すべてに `language: "zh-CN"` と `language: "es"` の翻訳を1件ずつ追加（30×2＝60 翻訳）。既存の ja/en/zh-TW/fr/ko と同一 intent で内容を用意する。

### 3.4 Docker・ブランチ戦略

- **Docker**: `docker-compose.yml` で postgres / redis / backend / frontend を起動。修正・テストはすべて Docker 内で実行する（大原則）。
- **ステージング**: Render.com Blueprint（`render.yaml`）。バックエンドは `runtime: docker`、フロントは `runtime: static`。本タスクでは render.yaml の変更は不要。
- **ブランチ**: `main`（本番）、`develop`（ステージング）、`feature/*`（開発）。本タスクは `develop` から `feature/multilingual-5-zhcn-es-remove-de-vi` を作成し、完了後に `develop` へマージする。

## 4. 実装ステップ（実施順）

### Step 0: 準備

- `develop` を最新に pull。
- ブランチ作成: `feature/multilingual-5-zhcn-es-remove-de-vi`。
- Docker 起動: `docker-compose up -d`。バックエンド・フロント・DB・Redis が正常であることを確認。

### Step 1: バックエンド — 対応言語リストの更新（5b の一部・5a の前提）

| ファイル | 変更内容 |
|----------|----------|
| `backend/app/services/facility_service.py` | Premium 時の `available_languages` を `["ja", "en", "zh-TW", "zh-CN", "fr", "ko", "es"]` に変更。 |
| `backend/app/services/chat_service.py` | 同上。 |

- de・vi はもともとリストに含まれていないため、追加しない限りゲストには出ない。フロントで de・vi を外すことで、万が一過去データで de/vi が渡っていても選択肢には表示されない。

### Step 2: バックエンド — FAQ プリセットに zh-CN と es を追加（5a）

| ファイル | 変更内容 |
|----------|----------|
| `backend/app/data/faq_presets.py` | 各 FAQ の `translations` に、既存5言語の後に `zh-CN` と `es` のオブジェクトを追加。全30件×2言語＝60翻訳。 |

- 翻訳方針: 既存多言語実装計画書と同様。簡体中国語は簡体字で統一。スペイン語は中南米・スペインで自然な表現（必要に応じて vosotros は避ける等、既存 fr/ko と同レベルの方針でよい）。
- 文字数目安: 質問 150 文字以内、回答 500 文字以内（日本語基準に合わせる）。

### Step 3: フロントエンド — ゲスト用言語定数の更新（5a・5b）

| ファイル | 変更内容 |
|----------|----------|
| `frontend/src/utils/constants.ts` | `SUPPORTED_LANGUAGES` から `de` と `vi` を削除し、`zh-CN`（表示名: 簡体中文、フラグ: 🇨🇳 等）を追加。`es` は既にあるためそのまま。並び順は ja, en, zh-TW, zh-CN, fr, es, ko を推奨（th は現状ゲストで使わないが残してもよい。ドキュメント上は「ゲストで使う言語」からは de・vi を削除）。 |

- これにより、バックエンドが返す `available_languages` に含まれる言語のうち、SUPPORTED_LANGUAGES に存在するものだけがゲスト画面に表示される。de・vi は定数から外すため表示されない。

### Step 4: フロントエンド — クーポンコピー対応言語の記載更新

| ファイル | 変更内容 |
|----------|----------|
| `frontend/src/utils/couponCopy.ts` | ファイル先頭のコメント「対応: ja, en, zh-TW, fr, es, de, ko, th, vi」を、実際の対応言語リスト（de・vi 削除、zh-CN 追加）に合わせて更新。実装で zh-CN 用のコピーを追加するかは別判断（未対応の言語は en フォールバックのため、zh-CN を追加しなくても動作上は問題ないが、コメントと実装の一致を推奨）。 |

- 既存の `getCouponCopy(lang)` に zh-CN が渡った場合の分岐がなければ、en にフォールバックする実装になっているか確認し、必要なら zh-CN を追加する。

### Step 5: 既存テストの更新

| 対象 | 変更内容 |
|------|----------|
| `backend/tests/test_facility_available_languages.py` | Premium の期待値を `["ja", "en", "zh-TW", "zh-CN", "fr", "ko", "es"]` に変更。 |
| `backend/test_staging_facility_available_languages_with_users.py` 等 | Premium の期待言語リストを同様に更新。 |

- 他に `available_languages` や言語リストをハードコードしているテストがあれば同様に更新する。

## 6. リスクと注意

- **翻訳品質**: zh-CN・es の 30 件×2 は、既存の zh-TW/fr/ko と同水準（宿泊業界用語・敬語）で揃える。機械翻訳利用時は簡体字・スペイン語表記の統一を行う。
- **RAG・プロンプト**: 既存の「request.language で回答言語を強制」する仕様はそのまま。zh-CN・es が request.language で渡れば、既存と同様にその言語の FAQ が使われる。
- **フォールバックメッセージ**: OpenAI API 障害時などのフォールバック文言に zh-CN・es を追加するかは、既存の en/ja 等の扱いに合わせて判断する（必要なら別 PR でも可）。
