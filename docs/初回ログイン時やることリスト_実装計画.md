# 初回ログイン時やることリスト 実装計画

**作成日**: 2026年3月12日  
**対象課題**: Phase 3 残存課題「初回ログイン時やることリスト」（中・未着手）  
**基準文書**: 要約定義書 v0.3、アーキテクチャ設計書、現況総括（20260307）、大原則

## 1. 目的・全体像・大原則の把握（要約）

### 1.1 目的・経緯

- **やどぺら**: 小規模宿泊施設向け外国人ゲスト対応自動化SaaS。QRコードでAIが24時間多言語応答。外国人対応70%自動化を目標。
- **現フェーズ**: Phase 3（PoC実施）実行中。Phase 4 Stripe は完了。
- **本機能の位置づけ**: 新規登録した施設管理者が「初回ログイン時」に迷わず初期設定（FAQ登録・QR案内・張り出し）に進めるよう、やることリストをモーダルで表示し、ご利用マニュアル「初期設定」へ誘導する。

### 1.2 大原則（遵守事項）

| 優先順位 | 原則 | 本実装での適用 |
|----------|------|----------------|
| 1 | **根本解決 > 暫定解決** | 初回判定は施設単位でDBに保持し、表示済みフラグで制御する |
| 2 | **シンプル構造 > 複雑構造** | 既存の Modal / 認証API パターンを流用し、新規コンポーネントは最小限にする |
| 3 | **統一・同一化 > 特殊独自** | 既存の `Modal.vue`・`/api/v1/auth/me`・管理画面ルーティング規約に合わせる |
| 4 | **具体的 > 一般** | やることリストの文言はマニュアル「1.4 初期設定」と一致させる |
| 5 | **拙速 < 安全確実** | すべて Docker 環境で動作確認してからステージングデプロイする |
| 6 | **Docker環境必須** | 修正・テストは `docker-compose` で実行する |

## 2. コードベース・ブランチ・Docker 現況

### 2.1 ブランチ戦略

- **main**: 本番用（LP は GitHub Pages でデプロイ）。
- **develop**: ステージング用。Render.com のバックエンド・フロントは `develop` を参照。
- **feature/\***: 開発用。本機能は `feature/first-login-onboarding-modal` 等で実装し、develop にマージする想定。

### 2.2 Docker 環境

- **docker-compose.yml**: `postgres`（pgvector/pg15）、`redis`、`backend`（FastAPI）、`frontend`（Vite）の4サービス。
- 起動: `docker-compose up -d`。Backend: http://localhost:8000、Frontend: http://localhost:5173。
- すべての修正・テストはこの環境で実施する（大原則）。

### 2.3 認証・ログイン周りの現状

- **ログイン後遷移**: `useAuth.login()` → `authStore.login()` → `router.push('/admin/dashboard')`。常にダッシュボードへ。
- **ユーザー情報**: `LoginResponse` / `GET /api/v1/auth/me` で `UserResponse`（id, email, full_name, role, facility_id, is_active, email_verified）を返す。**初回ログイン／オンボーディング表示済みのフラグは現状なし。**
- **施設モデル**: `backend/app/models/facility.py` の `Facility` に `created_at` はあるが、**初回ログイン用の「やることリスト表示済み」を示すカラムはない。**
- **ユーザーモデル**: `User` に `last_login_at` はある（ログイン時に更新）。`first_login_at` や施設側の「初回表示済み」は未実装。

### 2.4 管理画面のモーダル・マニュアル

- **共通モーダル**: `frontend/src/components/common/Modal.vue`。`v-model`・`title`・`size`・`closable`・`beforeClose` 対応。既存で `PlanBilling.vue`・`FaqManagement.vue` 等が利用。
- **マニュアル**: `frontend/src/views/admin/Manual.vue`。パス `/admin/manual`、ルート名 `AdminManual`。セクションは `id` でアンカー指定可能（例: `intro-initial-setup`（1.4 初期設定）、`practice-checklist`（12.1 初期設定チェックリスト））。

### 2.5 結論（現況）

- 初回ログインの判定と「やることリスト表示済み」の永続化を行うには、**施設単位のフラグ（または日時）をDBに追加**し、**ログイン応答または /me で「表示要否」を返す**形が自然。
- モーダルは **ダッシュボード表示直後に一度だけ表示**し、閉じたら「表示済み」をAPIで記録する実装が、既存パターンと整合する。

## 4. 実装計画（大原則準拠）

### 4.1 要件の整理

| 項目 | 内容 |
|------|------|
| 表示タイミング | 新規登録後の**初回ログイン時**（施設単位で「まだ表示していない」ときのみ） |
| 表示場所 | 管理画面ダッシュボード（`/admin/dashboard`）を開いた直後 |
| UI | モーダルで「やることリスト」を箇条書き表示。**「FAQ登録」と「QRコード発行」は最重要**として、重要マーク・太字・色のいずれかで他より明確に表現する。各項目のタイトル下に目的・理由を記載（4.4 参照）。 |
| 誘導 | ご利用マニュアル「初期設定」（1.4）へのリンクを掲載（FAQ登録・QR案内・張り出しを促す） |
| 二度表示しない | モーダルを閉じた（または「マニュアルを見る」で遷移した）時点で「表示済み」を記録し、以降は表示しない |

### 4.2 バックエンド

1. **DB拡張（Alembic）**
   - `facilities` テーブルに `onboarding_modal_shown_at`（`TIMESTAMP WITH TIME ZONE`, nullable）を追加。
   - 意味: `NULL` ＝ まだ初回やることリストモーダルを表示していない。日時が入っていれば表示済み。

2. **API**
   - **GET /api/v1/auth/me の拡張**: レスポンスに `show_onboarding_modal: bool` を追加。  
     - 条件: 現在ユーザーの `facility` の `onboarding_modal_shown_at` が `NULL` なら `true`、それ以外は `false`。
   - **POST /api/v1/auth/onboarding-seen**（新規）: 認証必須。現在ユーザーの施設について `onboarding_modal_shown_at = now()` に更新。レスポンスは `{ "ok": true }` 等の簡易形で可。

3. **ログイン応答**
   - `LoginResponse` に `show_onboarding_modal: bool` を含める（`/me` と同様の算出ロジック）。フロントはログイン直後もダッシュボード表示時に「表示要否」を判定できるようにする。

4. **スキーマ**
   - `UserResponse` に `show_onboarding_modal: bool` を追加（/me 用）。
   - `LoginResponse` の `user` は `UserResponse` のため、上記で両方に載る。または `LoginResponse` のトップレベルに `show_onboarding_modal` を追加してもよい（設計は「統一」の観点で `UserResponse` に含めるのがおすすめ）。

### 4.3 フロントエンド

1. **型・ストア**
   - `UserResponse` 相当の型に `show_onboarding_modal?: boolean` を追加。
   - `authStore` の `user` にこの値が入るようにする（`initAuth` / ログイン時のレスポンスで反映）。

2. **ダッシュボードでのモーダル表示**
   - `Dashboard.vue` で、`authStore.user?.show_onboarding_modal === true` のときのみ「初回ログイン時やることリスト」モーダルを表示する。
   - 表示内容（案）:
     - タイトル: 「はじめにやること」
     - 本文: マニュアル 1.4 に沿った箇条書き。**「FAQ登録」と「QRコード発行」は最重要として他より明確に強調する**（★重要マーク・太字・色のいずれかまたは組み合わせ。詳細は 4.4・4.4.1 参照）。
     - フッター: 「マニュアルで見る」（`/admin/manual#intro-initial-setup` へ遷移し、モーダルを閉じたうえで「表示済み」を記録）、「閉じる」（モーダルを閉じるだけでも「表示済み」を記録）。

3. **「表示済み」の記録**
   - モーダルを閉じる際（「閉じる」クリックまたは「マニュアルで見る」で遷移する際）に `POST /api/v1/auth/onboarding-seen` を呼ぶ。
   - 成功したら `authStore` の `user.show_onboarding_modal` を `false` に更新（または /me を再取得して上書き）し、次回以降モーダルを出さない。

4. **コンポーネント**
   - 新規: `OnboardingTodoModal.vue`（または既存 `Modal.vue` をラップした `FirstLoginTodoModal.vue`）を用意し、`Dashboard.vue` から利用。既存の `Modal.vue` の `title`・スロット・`beforeClose` をそのまま利用する。

5. **マニュアル誘導**
   - リンク: `router.push({ name: 'AdminManual', hash: '#intro-initial-setup' })` で 1.4 にスクロール。必要なら 12.1 へのリンクもモーダル内にテキストで記載。

### 4.4 やることリストの文言（モーダル用）

マニュアル 1.4 に合わせた案。**「FAQ登録」と「QRコード発行」は最重要**として、重要マーク（★）・太字・色変更などのいずれかまたは組み合わせで他項目より明確に目立たせる。

1. 施設設定の確認（施設名・連絡先・WiFi・チェックイン/アウト・館内ルールなど）
2. **★ FAQの登録・編集**（最重要）  
   → タイトル下に以下を追加（冗長にしない範囲で1〜2行）:  
   「登録数が多く、内容が具体的なほどAIの回答精度が上がり、自動応答率が高まります。初期デフォルトを確認し、施設に合わせて追加・編集してください。」
3. **★ QRコードの発行と設置**（最重要）  
   → タイトル下に以下を追加:  
   「ゲストが質問にアクセスする入口です。掲示・貼り付けに加え、チェックイン時にQRコードを読み込んでもらうと利用率が上がります。入口・客室など見やすい場所へ設置し、読み取りテストをしてください。」
4. 動作確認（ゲスト画面・AI回答・エスカレーション）
5. 初期設定チェックリストの確認（マニュアル第12章）

末尾に「詳しくはご利用マニュアルの**初期設定**をご覧ください。」とし、リンクを「マニュアルで見る」とする。

#### 4.4.1 最重要項目の視覚的表現（実装時の仕様）

- **対象**: 「FAQの登録・編集」と「QRコードの発行と設置」の2項目。
- **表現方法**（いずれかまたは組み合わせで、他項目より重要であることが一目で分かるようにする）:
  - **重要マーク**: 項目の先頭に ★ や「最重要」ラベルを付与。
  - **太字**: 項目タイトルを太字（`font-weight: bold` または `font-semibold`）で表示。
  - **色**: テキスト色や背景色を変更（例: アクセント色・警告色に近い色で「重要」を表現。ダークモード時も視認性を確保）。
- **目的**: 初回利用者が「まずFAQとQRに取り組む」ことを促し、効果的な初期設定に導く。

### 4.5 実装ステップ（順序）

| Step | 内容 | 備考 |
|------|------|------|
| 1 | Alembic で `facilities.onboarding_modal_shown_at` を追加 | 新規マイグレーション |
| 2 | `UserResponse` に `show_onboarding_modal` 追加。`/me` とログイン処理で Facility を参照し設定 | 既存ロジックに組み込み |
| 3 | `POST /api/v1/auth/onboarding-seen` を新設し、自施設の `onboarding_modal_shown_at` を更新 | 認証必須 |
| 4 | フロント: 型・authStore で `show_onboarding_modal` を扱う | |
| 5 | `OnboardingTodoModal.vue`（または同等）を作成し、Dashboard で条件表示・閉じたときの API 呼び出し | 既存 Modal 利用 |
| 6 | Docker で全フロー確認（ログイン → ダッシュボード → モーダル表示 → 閉じる/マニュアルへ → 再表示されない） | 必須 |
| 7 | ステージング（develop）へマージ・デプロイ後、ブラウザで再確認 | 任意で 2.1 文面追記 |

### 4.6 注意事項

- **新規登録直後の「メール確認待ち」**: 初回ログインは「メール確認完了後、初めてログインしたとき」を指す。メール確認前の画面ではモーダルは出さない（ダッシュボードに来るのは認証済みユーザーのみのため、現状の前提で問題なし）。
- **既存施設**: 既に運用中の施設は `onboarding_modal_shown_at = NULL` のままなので、本番反映後「初回」としてモーダルが1回出る。運用上許容するか、一度限りのマイグレーションで「既存施設は `created_at` が過去なら `onboarding_modal_shown_at = created_at` で埋める」などの対応は要件次第で検討可。
