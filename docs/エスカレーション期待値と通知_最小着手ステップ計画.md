# エスカレーション期待値と通知 — 最小着手ステップ計画

**作成日**: 2026年3月25日  
**最終更新**: 2026年3月25日（v1.6: **A-4 詳細は別紙** `エスカレーション_A-4_スタッフ通知メール_実装計画.md` へ）  
**目的**: 「次に何をするか」を **ステップで固定**し、**再訪着手のみ**か **手動返信の実装も含むか** を一目で判別できるようにする。さらに **ゲストが見る受付番号・スタッフが画面上で見る番号・通知（メール等）に載る番号が同一であること**を、この計画の **Go 条件**に組み込む。  
**関連**: `docs/手動返信機能_実装計画.md`（手動返信の Phase 分け・過去 revert 記録）／**A-4 の実装手順・現状ギャップの詳細**: `docs/エスカレーション_A-4_スタッフ通知メール_実装計画.md`

### 受付番号の一致（本計画の必須原則）

ゲストに「受付番号 **123** です」と見せたあと、電話やチャットで「受付 **123**」と伝えられた場合、**スタッフが管理画面上で同じ数値をラベル付きで突き合わせできる**ことは、運用上の最低ラインである。

| 観点 | 定義（開発者向け・内部名） | 画面・通知での見せ方 |
|------|---------------------------|----------------------|
| **正とする ID** | DB の `escalations.id`。API では `escalation_id` 等として返す。**他の ID（例: `session_id`、会話 UUID）と混同しない** | ゲスト・スタッフ向けとも **「受付番号」** とラベルし、**この数値だけ**を指す |
| **一致の意味** | あるエスカレーションについて、次の **すべて**が **同一の数値**であること | ゲスト完了表示 ＝ **未解決一覧と会話詳細の両方**に出す受付番号 ＝ **A-4 実装後**はメール本文（および可能なら件名）の受付番号 |
| **不一致が出たら** | No-Go。どれか一方だけ別の列を表示している・別テーブルの連番を出している等は **仕様バグ**として直す | 「番号が合わない」と現場が混乱するため、**見た目の便宜で別番号を出さない** |

※ 管理画面の文言は宿泊客向けではないため、「受付番号」「エスカレ」など **運用で通じる表現**でよい。ゲスト画面では §「ゲスト画面の文言」に従う。

### ゲスト画面の文言（必ず守ること）

ゲストは旅行中で疲れていることが多く、**分かりにくい言葉ほど不満につながります。** 次の語は **ゲストに見せる画面・ダイアログ・トーストでは使わない**（説明やマニュアルでも、ゲスト向けセクションでは使わない）。

- **使わない例**: SLA、モーダル、エスカレーション、API、ポーリング、PR、revert、セッション（代わりに「このチャット」など）、DOM、503 などの開発用語。
- **使う例**: スタッフ、お問い合わせ、受付番号、お返事、しばらくお待ちください、夜は返事が遅れることがあります、電話が必要なときは、送信する、戻る、閉じる、更新する。

※ 本書のうち **ファイル名・API 名・実装手順**は開発者向けです。**ゲストにそのまま表示しないでください。**

**文案の決め方**: ゲストに見せる文は、施設オーナー（または運用担当）と **合意済み文案** をメモ・表・issue などで決め、実装に載せる。

## 1. トラック A — 最小着手（固定順・A-1 → A-4）

以下は **実装順序の契約** とする。飛ばす場合は理由を本書または PR に残す。

| 順 | ステップ | 主な成果物 | 手動返信を触る？ | 完了の見え方（例） |
|----|----------|------------|------------------|---------------------|
| **A-1** | コピー改修 | ボタン文言・**送信前**の確認・**送信後**のメッセージ（ゲスト） | **いいえ** | **合意済み文案**どおり・主要言語で自然 |
| **A-2** | 対応目安表示 | **お返事までの目安**（日中／夜など）を平易な文で | **いいえ** | 決めた条件で文が切り替わる（実装方針は PR に明記） |
| **A-3** | 受付番号表示（**ゲストとスタッフで同一数値**） | サーバーが返す **`escalations.id`（API: `escalation_id`）** を、**ゲスト画面（実装済み）** に加え、管理画面の **未解決一覧・会話詳細の両方**に **「受付番号」** として表示（**どちらか一方だけでは未完了**） | **いいえ** | ゲスト完了UI ＝ 未解決一覧の該当行 ＝ **会話詳細の当該会話** で **同一数値** が揃っている |
| **A-4** | スタッフ通知の実送強化 | メール等 **既存通知チャネル** の確実化・失敗時ログ。**本文（および可能なら件名）に A-3 と同一の受付番号**を載せる | **いいえ** | スタッフ側で受信確認できる／失敗が追える／**メールの受付番号がゲスト表示と一致** |

**順序に関する注意**

- **A-3 と A-4**: メール（A-4）に載せる受付番号は **A-3 で定義した `escalations.id` と同じ値だけ**を使う（別フィールド・別採番を増やさない）。論理的には **先にゲスト＋管理画面で番号を揃えたうえで、同じ値をメールに埋め込む**と運用しやすい。実装は **A-3 → A-4** の順を推奨。
- **A-2**: 施設ごとの「不在時間」に連動させる場合、**公開 API またはチャット取得時のレスポンス拡張**が必要になることがある。仕様決定後に **A-2 を A-1 の前後どちらで入れるか**再検討してよい（**順序固定は「推奨」**で、理由付きで入れ替え可）。

**大原則（開発）**: 検証は **Docker 環境**で回す。ブランチは **`feature/*` → `develop`**。詳細は `README.md`・`docs/サービス開始までの手順_Runbook.md` を参照。

※ 下表の **ゲスト向け**の列は、宿泊客がそのまま読む文案のイメージです（§「ゲスト画面の文言」を守る）。

| 項目 | 期待する動作（開発） | ゲスト向けの表示イメージ（案） | スタッフ（管理画面） |
|------|----------------------|--------------------------------|----------------------|
| **正の値** | 表示する数値は **常に `escalations.id`**（API レスポンスの `escalation_id` と同一）。**クライアント側で別採番・ハッシュ表示をしない** | （同上） | **未解決一覧の各行** と **会話詳細の会話情報エリア**の **両方**に **受付番号: 123**（**ゲストに見せた 123 と同一**） |
| 番号（ゲスト） | 送信成功直後にモーダルで上記 ID を表示 | **本リポで実装済み**（`Chat.vue`＋`escalationGuestCopy.ts`）。「受付番号 **123**」相当 | — |
| 番号（スタッフ） | **一覧と詳細の両方**に同じラベル・同じ数値 | — | **未解決一覧**: ゲストが伝えた番号で行を特定できる。**会話詳細**: 電話・対応メモの最中も **同一番号がヘッダ付近で常に見える**（一覧に戻らないと番号が分からない状態にしない） |

| **完了条件** | **Go**: 1 件エスカレで **(1) ゲスト成功モーダルの番号 (2) 未解決一覧の該当行 (3) その会話の会話詳細画面 (4) API の `escalation_id`** の **4 点が同一整数**。**一覧のみ・詳細のみ**は **どちらも A-3 未完了** | — | — |

**実装の前提（現状コード）**: 会話詳細 `ConversationDetail.vue` は `chatApi.getHistory` のみ参照しており、**現行の `ChatHistoryResponse` には `escalation_id` が無い**。よって詳細で受付番号を出すには、例: **履歴 API に当該セッションの未解決エスカレの `id` を載せる**、**未解決一覧から `escalation_id` をクエリで渡す** など、**いずれかで `escalations.id` を画面が取得できる**ようにする（**別採番・別フィールドの表示は禁止**）。

## 2.2 トラック C — 連絡可能状態の可視化（B と分離して段階導入）

手動返信（B）を運用で迷わず使うために、**「誰に返せる状態か」**を受付番号単位で明示する。  
ここでの状態は「別番号」ではなく、**既存の `escalations.id`（受付番号）に付く運用状態**として扱う。

| 順 | ステップ | 内容 | 手動返信を触る？ |
|----|----------|------|------------------|
| **C-1** | 状態定義の固定 | `連絡先あり` / `連絡先なし` の2状態を固定。受付番号・会話・通知で同じ意味で使う。 | **いいえ** |
| **C-2** | 管理画面表示 | 未解決一覧と会話詳細に連絡可能状態を表示（同じラベル・同じ判定ロジック）。 | **いいえ** |
| **C-3** | 同意ベース収集（必要時） | ゲストが任意同意した場合のみ、最小項目の連絡先を受付番号に紐づける。 | **いいえ** |

### 2.3 B + C の実装順（今回着手の固定）

過去の失敗（UI白画面・検証分離不足）を踏まえ、**責務境界を先に固定してから実装**する。

| 順 | 実装単位 | 目的 | 完了条件（Go） |
|----|----------|------|----------------|
| 1 | **境界固定（B-0/C-0）** | 「返信投稿（B）」と「連絡可能状態表示（C）」を API・UI で分離。 | PR 上で境界図と責務が明記され、同一PR抱き合わせを避ける |
| 2 | **B-1** | 管理画面から返信投稿を成立。 | `POST /admin/conversations/{session_id}/reply` が 201、`messages.role=staff` 保存 |
| 3 | **C-1/C-2** | 一覧と詳細で連絡可能状態を同じ基準で表示。 | 同一受付番号で一覧と詳細の状態が一致 |

### 2.4 B + C の干渉防止ルール

- **別PR原則**: B と C は同時に進めても、マージ単位は分ける（revert容易性を維持）。
- **単一正本**: 受付番号は常に `escalations.id`。C の状態表示でも別採番を作らない。
- **No-Go 条件**: B-2 のゲスト表示で白画面・初期化失敗が再発した場合は即停止し、B-1/C を巻き込まない。

### 2.6 C-3（同意ベース連絡先収集）の有効化手順（安全運用）

本機能は **既定で無効**。有効化は以下の順で行う。

1. バックエンド環境変数で `ENABLE_CONTACT_CAPTURE=true` を設定する。  
2. フロントエンド環境変数で `VITE_ENABLE_CONTACT_CAPTURE=true` を設定する。  
3. ステージングで以下を確認する。  
   - ゲスト画面: エスカレーション完了モーダルに同意フォームが表示される。  
   - ゲスト送信: `POST /api/v1/chat/contact-consent` が 200 を返す。  
   - 管理画面: 未解決一覧/会話詳細の連絡可能状態が `連絡先あり` へ変わる。  
4. 本番反映後、監視期間中にエラー率・白画面有無・問い合わせ件数を観察する。  

### 2.7 C-3 ロールバック手順（即時復旧）

異常時は、以下を上から順に実施する。

1. フロントの `VITE_ENABLE_CONTACT_CAPTURE` を `false` に戻す（UI入力停止）。  
2. バックエンドの `ENABLE_CONTACT_CAPTURE` を `false` に戻す（API停止）。  
3. 管理画面で連絡可能状態が `連絡先なし` へ戻ることを確認する。  
4. B-1（返信API/管理画面返信）は継続利用し、C-3 のみ切り離して原因調査する。  

### 2.8 C-3 Go/No-Go チェックリスト（本番有効化前）

- [ ] フラグ OFF で既存導線（エスカレーション・返信）が従来通り動く  
- [ ] フラグ ON で同意フォームが表示され、同意なしでは送信できない  
- [ ] 同意あり送信後、会話詳細と未解決一覧の状態表示が一致する  
- [ ] B-1 返信API（201）と C-3 同意API（200）が同時利用で干渉しない  
- [ ] 白画面・初期化失敗・重大エラーが再発しない  
- [ ] ロールバック（両フラグOFF）で即時に旧運用へ戻せる  

運用時の実施手順は `docs/operations/202604/20260423_C3_連絡先同意機能_ステージング検証手順.md` を参照する。

## 3. Go / No-Go（意思決定用）

- **「再訪着手だけ」Go**: トラック A の該当ステップだけマージし、**トラック B のコミットを含まない**。
- **「手動返信も」Go**: トラック B の **B-1 から**別ブランチで着手し、`手動返信機能_実装計画.md` の Phase 別判定に従う。
- **No-Go**: トラック A と B を同一 PR で抱え合わせ、どちらが壊したか判別できない状態。
- **受付番号の突合（トラック A）**:
  - **A-3 Go**: **ゲスト成功モーダル ＝ 未解決一覧の該当行 ＝ 会話詳細（同一セッション） ＝ API `escalation_id`（DB `escalations.id`）** の **4 箇所すべてが同一整数**。**会話詳細に出していない**状態は **A-3 未完了**。
  - **A-4 Go**: 上記に加え、**メール本文（および件名に載せる場合は件名）の受付番号も同一**。
