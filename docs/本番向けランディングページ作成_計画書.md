## 1. 目的・期待する効果

### 1.1 目的

- **SNS・広告などから誘引した訪問者**を、申し込み（Freeプラン開始）に導く**本番用ランディングページ**を用意する。
- **即契約（Freeプラン開始）を最終CTA**とする設計とし、現行の「準備中・1通だけご連絡」のティザーLPから、**本番ローンチ向けの訴求**に切り替える。

### 1.2 期待する効果

| 効果 | 説明 |
|------|------|
| **リード獲得** | ヒーロー・課題・解決策・料金・導入の声で理解と信頼を醸成し、Freeプラン申し込みへ誘導する。 |
| **認知の統一** | 「説明を止めない」「やらないことを決めているAI」など、LP_v1.0のメッセージでサービス像を統一する。 |
| **離脱抑制** | 補助金・数値効果は別ページに分離し、LP本体はシンプルに保ち、主導線を邪魔しない。 |
| **運用開始可能** | 本番環境構築・Stripe本番導入と並行または先行して、ドメイン（yadopera.com）で本番LPを公開できる。 |

## 2. 「ホームページ（最小版）」との構造上の関係

現況総括 §7.1 では、順序2が**本番向けランディングページ**、順序4が**ホームページ（最小版）**となっている。両者の関係は以下のとおり。

| 項目 | 本番向けLP（順序2） | ホームページ最小版（順序4） |
|------|---------------------|-----------------------------|
| **役割** | **申し込みに特化した1本の導線**。yadopera.com のトップ＝LP。 | **補助情報のまとまり**。LPの主導線を邪魔しない「調べたい人向け」の別ページ群。 |
| **内容** | ヒーロー、課題、解決策、料金、導入の声、CTA（Freeプランで今すぐ〜）。 | トップ（思想＋ミッション）、運営情報／セキュリティ方針／導入事例アーカイブ／**補助金・制度解説**／メディア・お知らせ。 |
| **LP_v1.0 での対応** | **Ⅰ. 本番LP 本文**（1ページ）。 | **Ⅱ. 別ページ（LP外）メニュー構成案**（補助金・効果・事例・Free利用条件・法人向け等）。 |
| **実施順・依存** | **先に実施**（順序2）。LPのヘッダー／フッターに「補助金・制度対応について」等の**リンク先**を設けるが、リンク先ページの実体は後続でよい。 | **後続**（順序4）。LPからリンクされる「補助金・制度対応について」「運営情報」等の**実ページ**を作成するタスク。 |

**結論**:  
- **LP** ＝ ドメインの入口であり「売る・申し込ませる」ための単一ページ。  
- **ホームページ最小版** ＝ LP からリンクされる**補助ページ群**（運営情報・補助金・導入事例等）を整備するタスク。  
- 構造上は **LP が親（入口）、ホームページ最小版が子（補助情報）**。LP 作成時にヘッダー／フッターに「補助金・制度対応について」等のリンクを張り、リンク先の実体は順序4で作成する形でよい。

## 3. 現況の把握

### 3.1 コードベース

| 項目 | 現況 |
|------|------|
| **LPの所在** | リポジトリ直下 `landing/index.html`（静的HTML・単一ファイル） |
| **デプロイ** | **GitHub Pages**（`main` ブランチから自動デプロイ）。公開URL: `https://yadopera.com/` |
| **技術** | Tailwind CSS CDN、Noto Sans JP、Formspree（ティザー登録用）、Google Analytics（G-BE9HZ0XGH4） |
| **現行の訴求** | 「準備中｜2026年春 提供開始予定」「サービス開始時に1通だけご連絡」＋ Formspree フォーム |
| **Vueアプリ** | `frontend/` は SPA。ルート `/` は `PWABoot.vue`（認証済み→ダッシュボード、未認証かつ last_facility_url なし→404）。**新規登録**は `/admin/register`（`frontend/src/views/admin/Register.vue`） |

### 3.2 デプロイ・環境

| 環境 | フロント | LP（landing/） |
|------|----------|----------------|
| **本番（想定）** | 未構築（Phase 4）。想定: `app.yadopera.com`（Vue SPA）。§3.5 参照。 | **GitHub Pages** → `yadopera.com`（LP） |
| **ステージング** | Render Static Site（`frontend` を Vite ビルド）→ `yadopera-frontend-staging.onrender.com` | **未デプロイ**（staging では LP はビルド対象外） |

- **render.yaml**: バックエンドは `runtime: docker`、フロントは `rootDir: frontend` の静的サイト。`landing/` は含まれない。
- 本番LPは引き続き **GitHub Pages（main）** で配信する前提で問題なし。本番アプリのURLが決まり次第、LP の CTA リンク先（申し込みページ）をそのURLに合わせる。

### 3.3 ブランチ戦略（要約定義書・アーキテクチャより）

| ブランチ | 用途 |
|----------|------|
| **main** | 本番。GitHub Pages は main からデプロイ → yadopera.com の LP となる。 |
| **develop** | ステージング（Render Blueprint）。フロント・バックは develop からデプロイ。 |
| **feature/*** | 開発用。 |

- LP 変更は **feature → develop で検証後、main にマージ** すると、GitHub Pages で本番LPが更新される。
- ステージングでは LP 単体のプレビューは、**ローカルで `landing/index.html` を開く**か、必要なら GitHub Pages のプレビュー（例: 別ブランチの Pages）で確認する運用で足りる。

### 3.4 大原則との整合（プロジェクト現況総括 §0.3）

| 原則 | 本LP作成での反映 |
|------|------------------|
| 根本解決 > 暫定解決 | ティザー用フォームを「申し込み」に差し替えるのではなく、CTA を「Freeプランで今すぐ始める」に統一し、申し込み先をアプリの新規登録ページに一本化する。 |
| シンプル構造 > 複雑構造 | 静的 HTML のまま維持。Vue に組み込まず、`landing/index.html` 単一ファイルで完結させる。 |
| 統一・同一化 > 特殊独自 | 既存の Tailwind CDN・Noto Sans JP・CSP・gtag の構成を踏襲し、LP_v1.0 の文言・構成に合わせて差し替える。 |
| 拙速 < 安全確実 | 変更後はローカル表示確認に加え、必要に応じて Docker 周辺（バックエンド等）に影響がないことを確認。LP 単体は HTML のため Docker 必須の対象外だが、CTA 先の `/admin/register` はステージングで動作確認する。 |
| Docker環境必須 | LP 自体は静的のため Docker 不要。ただし CTA リンク先（新規登録フロー）の確認は、ステージング（Render デプロイ＝実質 Docker 基盤）で行う。 |

### 3.5 本番での LP とアプリの URL（競合回避）

**結論（推奨URL設計）**

| 役割 | 本番URL | 配信元 |
|------|---------|--------|
| **LP（ランディングページ）** | **https://yadopera.com/** | GitHub Pages（`main` → ルート） |
| **Vueアプリ（ゲスト＋管理）** | **https://app.yadopera.com/** | Render Static Site（または他。Phase 4 で構築） |
| **バックエンドAPI** | **https://api.yadopera.com/** | Render Web Service（Phase 4） |

- **競合を避ける理由**: 現在、`yadopera.com` のルート（`/`）は **GitHub Pages で LP** が配信されている。アーキテクチャ設計書には「本番 Static Site (Vue.js) の URL: https://yadopera.com」とあるが、LP をルートに据えるなら、**Vueアプリは別オリジンに出す必要がある**。同一ドメインの同一パス（`/`）に LP と Vue の両方を載せることはできない。
- **コードベースの根拠**:
  - **バックエンド**（`backend/app/core/config.py`）: `frontend_url` のデフォルトは `https://yadopera.com`。この値が **QRコードのゲストURL**（`qr_code_service.py`: `{frontend_url}/f/{facility_slug}?location=...`）と **認証メール内リンク**（`auth_service.py`: `{frontend_url}/admin/verify-email`, `{frontend_url}/admin/password-reset/confirm`）のベースになる。
  - つまり **FRONTEND_URL は「Vueアプリのオリジン」** である必要がある。LP 用の `yadopera.com` と Vue 用のオリジンが分かれる場合、**FRONTEND_URL は Vue 側のオリジンに統一**する（例: `https://app.yadopera.com`）。
  - CORS は `CORS_ORIGINS` で指定。LP は静的で API を呼ばないため、本番では **Vue のオリジン**（`https://app.yadopera.com`）を許可すれば足りる。LP からフォーム送信等で API を叩く将来対応をする場合は、`https://yadopera.com` も追加する。
- **本番でやること（Phase 4 時）**:
  1. **yadopera.com** → GitHub Pages のまま（LP＋将来的にホームページ最小版の静的ページもここか別サブドメインで配信可）。
  2. **app.yadopera.com** → Vue のビルド成果物（`frontend/dist`）を配信。ゲストは `https://app.yadopera.com/f/{facility_slug}`、管理・新規登録は `https://app.yadopera.com/admin/*`。
  3. バックエンドの **FRONTEND_URL** = `https://app.yadopera.com`。**CORS_ORIGINS** = `https://app.yadopera.com`（必要なら `https://yadopera.com` を追加）。
  4. LP の CTA（「Freeプランで、今すぐ〜」）のリンク先 = **https://app.yadopera.com/admin/register**。
- **アーキテクチャ設計書との違い**: 同書では `ADMIN_URL=https://admin.yadopera.com` とある。管理専用サブドメインにする場合は、Vue を「ゲスト用」と「管理用」で二つに分けて配信するか、同一 Vue を `admin.yadopera.com` にデプロイしつつゲストを `app.yadopera.com` にデプロイする構成も可能だが、**QR・メールリンクはすべて FRONTEND_URL ベース**のため、ゲストと管理が同一 Vue であれば **FRONTEND_URL を 1 つ**（例: `https://app.yadopera.com`）にしておき、管理も `https://app.yadopera.com/admin/*` で提供するのがシンプルで競合しない。

## 4. 作成方針（LP_v1.0 準拠）

### 4.1 LP 本文の構成（全面アップデート）

| セクション | 内容概要 |

| **CTA（初出）** | 「▶ Freeプランで、今すぐ使ってみる」→ 申し込みページへのリンク |
| **H2** | 宿泊施設は、夜だけが大変なわけじゃない（課題列挙） |
| **H2** | YadOPERAがやることは、ひとつだけ（説明を止めない／一次対応／判断は人に戻す） |
| **H2** | AIに任せすぎない設計だから、安心して使えます（医療・安全は即スタッフ、夜間は翌朝対応） |

- **LP本体では「補助金」「数値効果」を語らない**（LP_v1.0 備考）。補助金・効果・事例は別ページとする。

### 4.2 申し込み導線（CTA）

- **文言**: 「Freeプランで、今すぐ使ってみる」「Freeプランで、今すぐ始められます」等、LP_v1.0 に合わせる。
- **リンク先**（§3.5 本番URL設計に合わせる）:
  - **本番**: `https://app.yadopera.com/admin/register`（Vueアプリを配信するサブドメイン。Phase 4 で確定）。
  - **開発・検証時**: ステージング `https://yadopera-frontend-staging.onrender.com/admin/register` を一時指定可能。
- 現行の Formspree「開始時に連絡を受け取る」フォームは、**本番LPでは削除**し、上記 CTA ボタンに統一する（ティザー用フォームは別ページや別URLで残すかは任意）。

### 4.3 ヘッダー・フッター・別ページ（LP_v1.0 Ⅱ）

- **ヘッダー**: LP 内アンカー（料金プラン等）＋「補助金・制度対応について」へのリンク（別ページ。運用実績後に「導入事例・導入店一覧」等も追加可）。
- **フッター**: 運営、プライバシーポリシー、ご利用規約、特商法表記、FAQ（リンク先は別ページまたはアプリ内を想定。本番ドメイン設計に合わせて設定）。
- **別ページ**（LP外）は、本タスクでは「構成案」として文書化のみ行い、実装は別タスク（ホームページ最小版等）とし、**本番ローンチ時点では「補助金・制度対応について」をヘッダーからリンク**する想定（LP_v1.0 本番ローンチ時点）。

### 4.4 SEO・メタ・SNS

- **title / description**: LP_v1.0 の H1 とサービス説明に合わせて更新。「準備中」は外す。
- **og:title / og:description / twitter:card**: 同様に本番向けに更新。
- **robots**: 本番公開時は `noindex` を外し、必要に応じて `index, follow` に変更。
- SNS シェア用のボタンは、Phase 3 計画書の「SNSシェアボタン」に従い、必要であれば設置する。

### 4.5 料金プラン表示

- 現行の要約定義書・現況総括の料金表（Free／Mini／Small／Standard／Premium）に合わせる。
- LP 上では「Freeプラン：日本語対応」「有料プラン：多言語対応・拡張機能」「まずはFreeで十分」程度の簡潔な表現とする（LP_v1.0 の「料金プラン」セクション）。

## 5. 技術・ファイル方針

### 5.1 変更対象

- **主**: `landing/index.html` の全面書き換え（構成・文言を LP_v1.0 に合わせ、CTA を Free 申し込みリンクに統一）。
- **副**: 必要に応じて `landing/thanks.html` 等の有無を確認し、CTA 先が「申し込み完了」ではなく「新規登録ページ」になるため、thanks は不要とするか、別用途に回す。

### 5.2 維持するもの

- Tailwind CSS CDN、Noto Sans JP、既存のカラー・クラス（.text-primary, .bg-primary 等）は再利用可能な範囲で維持。
- CSP、gtag（G-BE9HZ0XGH4）はそのまま利用。
- デプロイ方式は **GitHub Pages（main）** のまま。

### 5.3 リンク先の環境別切り替え（任意）

- CTA の URL を本番／ステージングで切り替える必要がある場合、GitHub Pages は静的のため **ビルド時や JS で分岐しない**。本番用に `https://app.yadopera.com/admin/register`（§3.5 推奨）を直書きし、ステージングで LP を確認するときは、一時的に staging URL に差し替えたブランチで確認する運用で可。

### 6.1 ローカル環境でのブラウザテスト

**可能です。** 以下のいずれかで実施できます。

| 確認内容 | やり方 |
|----------|--------|
| **LP 単体の表示・レイアウト・アンカー** | `landing/index.html` をブラウザで開く。ファイル直接（`file:///...`）でも可。Tailwind CDN やフォントはオンライン参照のためネット接続が必要。 |
| **LP を HTTP で表示（file 制約を避けたい場合）** | リポジトリルートで `npx serve landing -p 3000` 等を実行し、`http://localhost:3000/` で表示。 |
| **LP → 申し込みページの一連の流れ（ローカルで完結）** | (1) 上記で LP を表示（例: `http://localhost:3000/`）、(2) フロントは `cd frontend && npm run dev` で起動（Vite は既定で `http://localhost:5173`）、(3) LP の CTA の `href` を**一時的に** `http://localhost:5173/admin/register` に書き換えてクリックし、新規登録画面まで疎通確認。 |

- **注意**: LP は静的 HTML のため、CTA を本番／ステージング／ローカルで切り替える場合は **href の書き換え**で対応。ローカル確認後、コミット時には本番またはステージング URL に戻す。
- バックエンド（API）まで含めてローカルで試す場合は、Docker で backend を起動し、フロントの `.env` で `VITE_API_BASE_URL=http://localhost:8000` 等を指定したうえで、LP の CTA のみ上記のとおり `http://localhost:5173/admin/register` にすればよい。
