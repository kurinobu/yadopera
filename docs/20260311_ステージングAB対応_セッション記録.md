## 1. 対象とした現象（再掲）

| 現象 | 内容 |
|------|------|
| **A. ダッシュボードのプラン表示不整合** | プラン・請求ページでは変更後のプラン（例: Premium）が表示されるが、ダッシュボードの「利用状況」では古いプラン（例: Free）のまま。強制リロード・キャッシュ消去・再ログイン後も継続。 |
| **B. ゲスト画面で 7 言語が表示されない** | Premium 施設のゲスト画面で 5 言語カードのみ表示（簡体中国語・スペイン語が表示されない）。原因はステージングのバックエンドが多言語_5 対応前のコードのため。 |

## 2. 現象 A についての本セッションでの調査・作業・結果

### 2.1 実施した修正（経緯）

| 順 | 修正内容 | 実施箇所 | 結果 |
|----|----------|----------|------|
| 1 | プラン変更・解約時にダッシュボードキャッシュを無効化（Fix A） | `billing.py` の `change_plan` / `cancel_subscription` で `delete_cache(dashboard:data, facility_id)` を呼ぶ | デプロイ後も不整合継続 |
| 2 | monthly_usage をキャッシュに含めず常に DB から取得（案A） | `dashboard_service.py` の `get_dashboard_data`: キャッシュヒット時も `get_monthly_usage` で上書き、キャッシュ保存時は `monthly_usage=None` | デプロイ後も不整合継続 |
| 3 | エンドポイントで monthly_usage を必ず再取得して上書き（案1） | `dashboard.py` の `get_dashboard` で `dashboard_data.monthly_usage = await dashboard_service.get_monthly_usage(facility_id)` を実行 | デプロイ後も不整合継続 |

### 2.2 結論（現象 A）

- **根本解決は未達**。キャッシュ無効化・案A・案1 のいずれを実施しても、ステージングでは問題が解消しなかった。
- **「全て DB 都度取得（キャッシュ廃止）」** により確実に最新を返すことは可能だが、**遅く重くなりユーザー体験が悪化する** ため、現時点では実施しない判断。
- **残存課題** として、`docs/20260307_プロジェクト現況と今後の計画_総括.md` の §3.1 に記載済み。今後の計画で要対応。

**追記（2026-03-17）**: 修正計画に沿い Phase 1 を実施したのち、利用者がステージングでプラン変更→表示一致を確認。**現象 A は解消**。総括 §3.1 を「✅ 完了（2026-03-17）」に更新済み。参照: `docs/プランページとダッシュボード_プラン表示不整合_修正計画.md` §7・§8。

## 3. 現象 B についての本セッションでの調査・作業・結果

### 3.1 実施した作業

- **バックアップ作成**: `backups/20260311_guest_languages_premium_fallback/` に以下を保存。  
  `facility_service.py`, `chat_service.py`, `LanguageSelect.vue`, `constants.ts`
- **暫定案の実装と取りやめ**:  
  `LanguageSelect.vue` に「Premium かつ API が 5 言語しか返さない場合は、フロントで 7 言語リストで上書きして表示する」フォールバックを**一時的に実装したが、大原則に反するため取りやめ**。現在は **API の `available_languages` を唯一の正として表示する** 実装に戻している。

### 3.1.1 追加調査（後続セッションで確定した証拠）

後続セッションで、現象 B の原因について「未更新と判断」ではなく、外部観測とコミット照合により **未更新を断定**できた。

- **ステージング実行コミット**: `GET /__debug_env` の `RENDER_GIT_COMMIT=ff3fd411...` を確認。
- **公開 API の実測（Premium施設）**: Premium 施設 slug を `/api/v1/admin/facility/settings`（ログイン後）で取得し、`GET /api/v1/facility/{slug}` の `facility.available_languages` が `["ja","en","zh-TW","fr","ko"]`（5言語）で返ることを確認。
- **コミット照合**: `ff3fd41...` の `facility_service.py` / `chat_service.py` が Premium を 5 言語で返す実装であることを確認（＝上記実測と一致）。

結論として、ステージングで Premium が 7 言語にならない根本原因は「フロント実装」ではなく「ステージングが多言語_5 を含まないコミットを実行していること」である。

### 3.2 大原則に基づく判断（現象 B）

- **根本解決 > 暫定解決**: 原因は「ステージングにデプロイされているバックエンドが多言語_5 対応前のコードのまま」であること。**根本解決** は、**ステージングへ正しいバックエンド（`facility_service.py` で Premium の `available_languages` を 7 言語で返すコードが含まれるもの）をデプロイする** こと。
- **統一・同一化 > 特殊独自**: フロントで「Premium なら 7 言語」を定数として持つと、**言語リストがバックエンドとフロントで二重定義** になり、将来の変更で不整合が生じる。正は **バックエンド 1 か所** とし、フロントは API の `available_languages` をそのまま表示するのみとする。
- したがって **フロントでのフォールバックは採用しない**。B の解消は **ステージングへの正しいバックエンドのデプロイ** で行う。

## 6. 今後の実施者への注意（再発防止）

1. **現象 A（プラン表示不整合）**: 残存課題のまま。暫定で「フロントだけ書き換える」「キャッシュだけ消す」など、根本原因を置き換える実装は行わない。今後の計画で、デプロイ状況・プロキシ・DB レプリカ等の切り分けと、大原則に沿った解決策を検討すること。
2. **現象 B（ゲスト 7 言語表示）**: **フロントに「Premium なら 7 言語で上書き」するフォールバックを追加しないこと**。正はバックエンドの `GET /api/v1/facility/{slug}` の `available_languages`。解消は **多言語_5 対応済みバックエンドのステージングへのデプロイ** で行う。
3. **大原則**: 根本解決 > 暫定解決、統一・同一化 > 特殊独自。暫定案を実装する前に、上記に反していないか必ず確認すること。
