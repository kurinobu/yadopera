# ダッシュボード・プラン表示不整合 — 完全調査と大原則に準拠した修正案（第3回）

**日付**: 2026-03-11  
**現象**: プランページとダッシュボードで表示されるプランが異なる。強制リロード・キャッシュ消去・再ログイン後も継続。案A（monthly_usage をキャッシュ対象外）をデプロイ済みだが未解消。  
**指示**: 原因の完全な調査分析と、大原則に準拠した修正案の提示。実装は指示があるまで行わない。

---

## 1. データフロー全体の追跡

### 1.1 プランページのプラン表示

| 段階 | 場所 | 内容 |
|------|------|------|
| 1 | フロント | `GET /admin/plans` を呼ぶ（billingApi.getPlans） |
| 2 | バックエンド | `billing.get_plans` → `current_user.facility_id` で Facility を **その場で** select、`facility.plan_type` を返す。キャッシュなし。 |
| 3 | フロント | レスポンスの `current_plan_type` を表示。「(現在)」はこの値。 |

→ **常に DB の最新が返る**。

### 1.2 ダッシュボードのプラン表示

| 段階 | 場所 | 内容 |
|------|------|------|
| 1 | フロント | `GET /admin/dashboard` を呼ぶ（dashboardApi.getDashboard、`_t: Date.now()`・`Cache-Control: no-cache`） |
| 2 | バックエンド | `dashboard.get_dashboard` → `current_user.facility_id` で `DashboardService.get_dashboard_data(facility_id)` を呼ぶ。 |
| 3 | サービス | **キャッシュヒット時（案A）**: `cached_data` を取得 → `get_monthly_usage(facility_id)` を呼び、`return DashboardResponse(**{**cached_data, "monthly_usage": monthly_usage_fresh})`。**キャッシュミス時**: 並列取得のうち `get_monthly_usage(facility_id)` の結果を `monthly_usage` に使い、キャッシュには `monthly_usage=None` で保存。 |
| 4 | サービス | `get_monthly_usage`: `select(Facility).where(Facility.id == facility_id)` で施設を取得し、`plan_type = facility.plan_type or 'Free'` で月次利用状況を組み立てて返す。 |
| 5 | エンドポイント | `return dashboard_data`。FastAPI が `response_model=DashboardResponse` でシリアライズ。 |
| 6 | フロント | レスポンスの `monthly_usage.plan_type` を `MonthlyUsageCard` に渡して表示。 |

→ **案A どおりであれば、キャッシュヒット・ミスどちらでも `monthly_usage` はそのリクエスト内で `get_monthly_usage()` の結果（DB 参照）になっているはず**。

## 2. 不整合が続く原因候補の整理

### 2.1 デプロイ・実行コード

| 候補 | 説明 | 確認方法 |
|------|------|----------|
| **A. 案A がステージングに載っていない** | ビルドキャッシュ・別ブランチ・デプロイ失敗などで、monthly_usage をキャッシュ外して「常に再取得」する変更が入っていない。キャッシュヒット時に古い `cached_data["monthly_usage"]` がそのまま返っている。 | ステージングの該当サービスで `get_dashboard_data` のキャッシュヒット分岐に「monthly_usage を再取得して上書き」するコードがあるか確認。または一時的にログを仕込み、キャッシュヒット時に `get_monthly_usage` が呼ばれているか確認。 |

### 2.2 キャッシュ・レスポンス内容

| 候補 | 説明 | 確認方法 |
|------|------|----------|
| **B. 古いキャッシュの構造** | 案A デプロイ前に保存されたキャッシュが残っており、`cached_data` に `monthly_usage` が dict で入っている。マージの順序や Pydantic の解釈で、意図した `monthly_usage_fresh` が効いていない可能性（コード上は上書きしているが、本番で何か違う条件がある）。 | キャッシュヒット時にログで `monthly_usage_fresh.plan_type` を出力し、実際にレスポンスに含まれる `plan_type` と一致するか確認。 |
| **C. response_model による上書き** | FastAPI の `response_model=DashboardResponse` が、返却オブジェクトを再検証・再構築する過程で `monthly_usage` を別ソース（例: キャッシュ由来の dict）で上書きしている。 | FastAPI のドキュメント・ソースで、返り値が Pydantic モデルインスタンスの場合の挙動を確認。通常はそのままシリアライズされるが、サブモデルの扱いを要確認。 |

### 2.3 DB・セッション

| 候補 | 説明 | 確認方法 |
|------|------|----------|
| **D. 同一リクエスト内での Facility の二重読み** | `get_current_user` では Facility は読んでおらず、`get_monthly_usage` 内の `select(Facility)` がそのリクエストで唯一の Facility 読み。ただし、別経路で同じ Facility がセッションに載り、アイデンティティマップで古いインスタンスが返っている可能性は理論上ある。 | ダッシュボード取得時に `db.expire_all()` や Facility の `refresh` を一時的に入れ、強制的に DB 再読する実験。 |
| **E. レプリカ遅延** | 書き込みはプライマリ、読み取りはレプリカに振っている場合、プラン変更直後のダッシュボード読みでレプリカがまだ追いついておらず古い plan_type が見える。 | インフラで DB の役割（単一/プライマリ・レプリカ）を確認。Render 等のマネージド DB の仕様確認。 |

### 2.4 フロント・経路

| 候補 | 説明 | 確認方法 |
|------|------|----------|
| **F. リクエストがダッシュボード API に届いていない** | 別 URL・別サービス・CDN 等で古いレスポンスが返っている。 | ブラウザのネットワークタブで `GET /admin/dashboard`（または実際のパス）が発行されているか、レスポンス JSON の `monthly_usage.plan_type` を確認。 |
| **G. フロントの表示ソースの取り違え** | ダッシュボード画面で、別 API（例: プラン一覧）の結果やローカル状態を表示している。 | コード上、利用状況カードの `plan_type` が `dashboardData.monthly_usage.plan_type` のみに依存しているか確認（済: 依存している）。 |

## 3. 推奨: 原因切り分けのための確認手順

1. **ブラウザの開発者ツール**  
   - ダッシュボード表示中に `GET /admin/dashboard` のレスポンスを開く。  
   - `monthly_usage.plan_type` の値を確認。  
   - ここがプランページと一致していればフロント側、一致していなければバックエンド（API またはその前段）の問題。

2. **バックエンドのログ（案A が動いているか）**  
   - キャッシュヒット時に `get_monthly_usage` を呼ぶ分岐で、  
     `logger.info("dashboard cache hit, monthly_usage_fresh plan_type=%s", getattr(monthly_usage_fresh, "plan_type", None))` を追加。  
   - ステージングでダッシュボードを開き、キャッシュヒットが発生する 2 回目以降のリクエストで上記ログと実際のレスポンスの `plan_type` を照合。

3. **キャッシュのリセット**  
   - 案A デプロイ後、既存のダッシュボード用 Redis キーを削除（例: `dashboard:data:*` を FLUSH または該当キーのみ削除）してから再度テスト。古いキャッシュが残っていると、案A 以前の形で保存されたデータが返る可能性がある。

## 4. 大原則に準拠した修正案（確実に効く案）

**大原則**: 根本解決 > 暫定対応、統一・同一化 > 特殊な独自実装。

### 案1（推奨）: エンドポイントで monthly_usage を必ず上書きする

**方針**: サービス層の実装（キャッシュの有無・案A の有無）に依存せず、**API エンドポイントの時点で「返却直前のレスポンスから monthly_usage だけ必ず DB 由来で上書きする」**。

- **実装**:  
  - `backend/app/api/v1/admin/dashboard.py` の `get_dashboard` 内で、  
    `dashboard_data = await dashboard_service.get_dashboard_data(facility_id)` の直後に、  
    `dashboard_data.monthly_usage = await dashboard_service.get_monthly_usage(facility_id)` を実行する（上書き代入）。  
  - そのうえで `return dashboard_data` する。
- **効果**:  
  - キャッシュヒット・ミス、案A の有無、Pydantic の挙動、デプロイのずれなどに左右されず、**レスポンスに含まれる monthly_usage は常にそのリクエストで取得した DB の最新**になる。
- **デメリット**:  
  - キャッシュヒット時も `get_monthly_usage` が 1 回余分に呼ばれる（案A と同様。DB 1 クエリ＋軽い計算のみで負荷は限定的）。

### 案2: ダッシュボードのキャッシュを廃止する

**方針**: ダッシュボードレスポンス全体のキャッシュをやめ、毎リクエストで DB から組み立てる。

- **実装**:  
  - `DashboardService.get_dashboard_data` 内で、`get_cache` / `set_cache` を使わず、常に「キャッシュミス時」と同様の並列取得＋`DashboardResponse` 組み立てのみ行う。
- **効果**:  
  - プランに限らず、ダッシュボードのあらゆる項目が常に最新。不整合の原因がキャッシュにない場合も解消する。
- **デメリット**:  
  - 毎回 8 系統の集計などが走るため、レスポンス時間と DB 負荷が増える。ステージング・本番の負荷に応じて可否を判断する。

### 案3: get_monthly_usage の Facility を明示的に再読する

**方針**: 案A は維持しつつ、`get_monthly_usage` 内で Facility を読む前にセッションのキャッシュを無効化し、**必ず DB から再読**させる。

- **実装**:  
  - `get_monthly_usage` の先頭で、`await self.db.execute(select(Facility).where(Facility.id == facility_id))` の前に、  
    `self.db.expire_all()` を呼ぶ（または該当 Facility だけ `refresh`）。  
  - その後、これまでどおり `select(Facility)...` で取得し `plan_type` を使用。
- **効果**:  
  - 同一リクエスト内で Facility が別経路でセッションに載っていても、古い plan_type が使われる可能性を減らせる。
- **デメリット**:  
  - セッションの expire は他のオブジェクトにも影響する。レプリカ遅延には効かない。

## 5. まとめと実施順序の提案

| 項目 | 内容 |
|------|------|
| **原因の切り分け** | 上記 2 の候補 A〜G を、3 の手順（レスポンス JSON 確認・ログ・キャッシュ削除）で順に確認する。 |
| **確実に効く修正** | **案1（エンドポイントで monthly_usage を必ず上書き）** を実施する。サービス層のキャッシュや案A の有無に依存せず、プラン表示だけは常に最新にできる。 |
| **案1 の位置づけ** | サービス層の「monthly_usage をキャッシュに含めない」という設計（案A）に加えて、**API 層で二重に保証する**形であり、根本解決（常に最新を返す）と運用の確実性の両方を満たす。 |
| **案2・案3** | 負荷やインフラの制約に応じて、案1 では解消しない場合（例: レプリカ遅延）や、さらに確実にしたい場合に検討する。 |

**実施順序の提案**  
1. まず **案1 を実装**し、デプロイ後にブラウザでレスポンスの `monthly_usage.plan_type` とプランページの表示が一致するか確認する。  
2. それでも不整合が続く場合は、3 の確認手順で「API レスポンスは既に正しいか」「キャッシュヒット時に get_monthly_usage が呼ばれているか」を確認し、必要に応じて案2 または案3 を検討する。

以上。指示があるまで実装は行わない。

## 6. 修正計画への誘導（2026-03-15 追記）

本課題の**実施順序・原因切り分け・案3/案2 の位置づけ**は、大原則に準拠した **修正計画書** にまとめた。今後の実装は以下に従って進める。

- **修正計画**: `docs/プランページとダッシュボード_プラン表示不整合_修正計画.md`
  - Phase 1: 原因切り分け（ステージング実行コミット確認・API レスポンスの `monthly_usage.plan_type` 確認・必要ならキャッシュ削除・ログ追加）
  - Phase 2: `get_monthly_usage` 内で Facility の明示的再読（案3: `expire_all()` または `refresh`）
  - Phase 3: 解消しない場合のみ案2（キャッシュ廃止）・レプリカ遅延の調査

### 解消（2026-03-17 追記）

Phase 1 実施後、利用者がステージングでプラン変更を行い、プランページとダッシュボードのプラン表示が一致することをスクショ・ネットワークタブ・サーバーログで確認。**本課題は解消**。案A・案1・Fix A が有効に機能している。詳細は修正計画 §7（実施結果）・§8（結果の説明と評価）を参照。
