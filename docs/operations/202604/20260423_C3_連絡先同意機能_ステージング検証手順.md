# C-3 連絡先同意機能 ステージング検証手順（B+C）

## 0. 目的

- C-3（同意ベース連絡先収集）を、既存導線を壊さず段階導入できることを確認する。
- 受付番号運用（B-1）と連絡可能状態表示（C-1/C-2）が干渉しないことを確認する。

## 1. 事前条件

- 対象ブランチに B-1/C-1/C-2/C-3 実装が反映済みであること。
- ステージング DB にテスト用施設と管理者ユーザーが存在すること。
- API と Frontend のデプロイ先が同一環境（staging）であること。

## 2. 変更対象フラグ

- Backend: `ENABLE_CONTACT_CAPTURE`
- Frontend: `VITE_ENABLE_CONTACT_CAPTURE`

## 3. ケースA（既定OFF）検証

### 3.1 設定

- Backend: `ENABLE_CONTACT_CAPTURE=false`
- Frontend: `VITE_ENABLE_CONTACT_CAPTURE=false`

### 3.2 手順

1. ゲスト画面で通常の質問送信を行う。
2. 「スタッフに連絡」実行後、受付完了モーダルを確認する。
3. 管理画面未解決一覧から対象受付番号の行を開く。
4. 会話詳細で手動返信を投稿する。

### 3.3 期待結果

- ゲスト画面に連絡先同意フォームが表示されない。
- `POST /api/v1/chat/contact-consent` は `404`（無効）である。
- 管理画面の連絡可能状態は `連絡先なし` のまま。
- 手動返信（B-1）は正常（`201`）で投稿できる。

## 4. ケースB（段階ON）検証

### 4.1 設定

- Backend: `ENABLE_CONTACT_CAPTURE=true`
- Frontend: `VITE_ENABLE_CONTACT_CAPTURE=true`

### 4.2 手順

1. ゲスト画面で「スタッフに連絡」を実行し、受付完了モーダルを開く。
2. メール・同意チェックを入力して連絡先送信を実行する。
3. 管理画面未解決一覧を再読み込みする。
4. 対象受付番号の会話詳細を開く。
5. 管理画面から手動返信を投稿する。

### 4.3 期待結果

- 連絡先同意フォームが表示される。
- 同意なしでは送信できない。
- `POST /api/v1/chat/contact-consent` が `200` を返す。
- 未解決一覧と会話詳細の連絡可能状態がともに `連絡先あり` になる。
- B-1 返信投稿は継続して成功する（C-3 と干渉しない）。

## 5. No-Go 判定

以下のいずれかが発生した場合は No-Go。

- ゲスト画面で白画面・初期化失敗が発生する。
- 一覧と詳細で連絡可能状態が不一致になる。
- B-1 手動返信 API が失敗する（`201` 以外）。
- 同意送信で 5xx が継続発生する。

## 6. 即時ロールバック

1. Frontend の `VITE_ENABLE_CONTACT_CAPTURE=false` に戻す（UI停止）。
2. Backend の `ENABLE_CONTACT_CAPTURE=false` に戻す（API停止）。
3. ゲスト画面で同意フォーム非表示を確認する。
4. 管理画面で B-1 返信導線が継続利用可能であることを確認する。

## 8. APIスモークテスト（任意・推奨）

事前に以下を設定:

- `BASE_URL`（例: `https://staging-api.example.com`）
- `FACILITY_ID`
- `SESSION_ID`

### 8.1 C-3 OFF時の確認（404）

```bash
curl -i -X POST "$BASE_URL/api/v1/chat/contact-consent" \
  -H "Content-Type: application/json" \
  -d "{
    \"facility_id\": $FACILITY_ID,
    \"session_id\": \"$SESSION_ID\",
    \"email\": \"guest@example.com\",
    \"guest_name\": \"Guest\",
    \"consent\": true
  }"
```

期待: `404` / `Contact capture is not enabled`

### 8.2 C-3 ON時の確認（200）

```bash
curl -i -X POST "$BASE_URL/api/v1/chat/contact-consent" \
  -H "Content-Type: application/json" \
  -d "{
    \"facility_id\": $FACILITY_ID,
    \"session_id\": \"$SESSION_ID\",
    \"email\": \"guest@example.com\",
    \"guest_name\": \"Guest\",
    \"consent\": true
  }"
```

期待: `200` / `contactability_status: contactable`

補助テンプレート: `docs/operations/202604/20260423_C3_ステージング検証コマンドテンプレート.sh`

## 9. 失敗時の一次切り分け

1. **フラグ差分確認**: Backend と Frontend のフラグ値が同一意図になっているかを確認。  
2. **セッション確認**: `session_id` が実在会話に紐づくかを確認。  
3. **API単体確認**: UIではなく `curl` で `/chat/contact-consent` の応答を確認。  
4. **表示整合確認**: 管理画面の未解決一覧と会話詳細が同じ状態か確認。  
5. **No-Go判定**: 影響がある場合は §6 のロールバックを先に実施。

## 9.1 実行に必要な値

- `BASE_URL`: staging API のベースURL
- `FACILITY_ID`: 検証対象施設ID
- `SESSION_ID`: 検証対象セッションID
- `ADMIN_BEARER_TOKEN`: 管理者JWT（B-1確認用）

## 10. マニュアル反映確認（main反映前の必須）

- `frontend/src/views/admin/Manual.vue` の更新が `/admin/manual` に反映されていることを確認する。
- 反映手順は `docs/マニュアル更新・改訂_必読.md` および `docs/マニュアルManual編集後の反映手順.md` に従う。
- 実施結果は `docs/operations/202604/20260423_C3_連絡先同意機能_ステージング実施記録.md` の「7. マニュアル更新反映確認」に記録する。

## 11. main 反映手順（確定順）

1. `develop` 上で本RunbookのケースA/Bとマニュアル反映確認を完了し、実施記録を更新する。  
2. 差分確認（コード + docs）を行い、C-3関連以外を混入させない。  
3. `develop` から PR を作成し、レビューで以下を確認する。  
   - B-1（手動返信）が継続成功  
   - C-3（フラグON/OFF）が期待どおり  
   - マニュアル更新が `/admin/manual` に反映  
4. PR 承認後、`main` にマージする。  
5. 本番で最小確認を実施する。  
   - フラグOFF時: 同意フォーム非表示・既存導線維持  
   - フラグON時（段階有効化する場合）: 同意送信と状態表示が一致  
6. 問題があれば C-3 フラグをOFFへ戻し、B-1運用のみ継続する。  

main反映前の最終判定は `docs/operations/202604/20260423_B1_C3_main反映前チェックシート.md` を使用する。
