# 2026-03-24 本番認証メール確認障害 — 調査記録と次セッション引き継ぎ

## 1. 目的

- 本番で発生した「確認メール再送/メール確認」障害の経緯を、成功・失敗を含めて時系列で残す。
- 次セッションで迷わず Stripe 本番（live）投入フェーズへ進めるため、現状と未完了を明確化する。

---

## 2. 症状と事実

- 症状A: `/admin/verify-email-pending` の「確認メールを再送信」で UI に「予期しないエラー」または「ネットワークエラー」。
- 症状B: メールは届くが、メール内リンクの `/admin/verify-email?token=...` で「確認に失敗」。
- ブラウザ Console で `net::ERR_NAME_NOT_RESOLVED` を確認。
- エラー対象ホストは `api.yadopera.com`（DNS 未解決状態）。
- 同時に、Backend への直接疎通（`yadopera-backend-production.onrender.com`）では再送 API は 200 応答を確認済み。

---

## 3. 根本原因（確定）

- 本番 Frontend の `VITE_API_BASE_URL` が `https://api.yadopera.com` だった。
- その時点では `api.yadopera.com` の DNS が未解決で、ブラウザが API に到達できなかった。
- これは CORS より前段（名前解決）で失敗するため、UI では汎用のネットワークエラーとして表示された。

---

## 4. 失敗事例（再発防止用）

1. 再送 API が成功するケースと失敗するケースが混在し、当初は API 実装不具合に見えた。  
   - 実際は「どのバンドル（古い/新しい）を見ているか」「どの API ホストに向いているか」の差だった。
2. メール確認リンクを開くページだけ失敗する現象が出た。  
   - 原因は確認ページ側の API 呼び出しも同じ `VITE_API_BASE_URL` に依存しているため。
3. PWA（Service Worker）キャッシュにより、設定変更後もしばらく古い挙動が残ることがあった。  
   - 対処として強制リロード・サイトデータクリアが有効だった。

---

## 5. 実施した修正・対処

### 5.1 運用設定（Render）

- Frontend（Static）の `VITE_API_BASE_URL` を、DNS 解決可能な本番 Backend オリジンへ変更。  
  `https://yadopera-backend-production.onrender.com`（末尾 `/api/v1` なし）
- Save and rebuild 実施後、再送と確認リンクの両方で成功を確認。

### 5.2 フロントエンドコード

- 429 エラーコードの表記ゆれを吸収（`RATE_LIMIT` / `RATE_LIMIT_EXCEEDED`）。
- 429 で backend の detail メッセージを優先表示するよう調整。

### 5.3 Runbook 反映

- `docs/サービス開始までの手順_Runbook.md` §3.0 に、`VITE_API_BASE_URL` と `ERR_NAME_NOT_RESOLVED` の注意を追記済み。
- 同 Runbook §3.2 の管理者導線 4 項目を完了（[x]）へ更新済み。
