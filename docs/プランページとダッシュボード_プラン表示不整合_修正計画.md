# プランページとダッシュボードのプラン表示不整合 — 修正計画

**作成日**: 2026年3月15日  
**最終更新日**: 2026年3月17日  
**目的**: 残存課題「プランページとダッシュボードのプラン表示不整合」を、大原則に準拠した手順で原因を切り分け・修正する。  
**基準文書**: 要約定義書、アーキテクチャ設計書、総括、`docs/ステージング_ダッシュボードプラン表示不整合_完全調査と修正案.md`、`docs/20260311_ステージングAB対応_セッション記録.md`

---

## 1. 目的・全体像・大原則の把握

### 1.1 大原則（総括 §0.1）

| 優先 | 原則 |
|------|------|
| 1 | **根本解決 > 暫定解決** |
| 2 | **シンプル構造 > 複雑構造** |
| 3 | **統一・同一化 > 特殊独自** |
| 4 | **具体的 > 一般** |
| 5 | **拙速 < 安全確実** |
| 6 | **Docker環境必須** |

重要メモ（総括 §4）: **APIを正（Single Source of Truth）にする**。ステージング不整合時はまず `GET /__debug_env` で `RENDER_GIT_COMMIT` を確認する。

### 1.2 現象の整理

- **プランページ**: `GET /admin/plans` → `_get_facility_for_user()` でその都度 `select(Facility)`。キャッシュなし。→ **常に DB 最新**。
- **ダッシュボード**: `GET /admin/dashboard` → `DashboardService.get_dashboard_data()` が Redis 5分キャッシュを使用。利用状況カードの `plan_type` は `monthly_usage.plan_type` に依存。
- **不整合**: プラン変更後、プランページでは変更後プラン、ダッシュボード「利用状況」では古いプランが表示され続ける（強制リロード・キャッシュ消去・再ログイン後も継続）。

## 2. コードベース・環境の現況

### 2.1 既に実装済みの対策

| 対策 | 実装箇所 | 内容 |
|------|----------|------|
| Fix A | `billing.py` | プラン変更・解約時に `_invalidate_dashboard_cache(facility_id)` で `dashboard:data:{facility_id}` を削除。 |
| 案A | `dashboard_service.py` | キャッシュヒット時も `get_monthly_usage(facility_id)` で再取得し `monthly_usage` を上書き。キャッシュ保存時は `monthly_usage=None`。 |
| 案1 | `dashboard.py` | `get_dashboard` 内で `dashboard_data.monthly_usage = await dashboard_service.get_monthly_usage(facility_id)` で上書き。 |

2026-03-11 セッションでは上記をデプロイ済みだが、ステージングで不整合が解消しなかったと記録されている。

### 2.2 データフロー（現行）

1. **ダッシュボードAPI**  
   `get_dashboard` → `get_dashboard_data(facility_id)` → （キャッシュヒット時）`cached_data` + `get_monthly_usage(facility_id)` でマージ、または（ミス時）並列取得のうち `get_monthly_usage` を利用。  
   続けて **エンドポイントで** `dashboard_data.monthly_usage = await get_monthly_usage(facility_id)` で再上書き → `return dashboard_data`。

2. **get_monthly_usage**  
   `select(Facility).where(Facility.id == facility_id)` で施設取得 → `plan_type = facility.plan_type or 'Free'` で `MonthlyUsageResponse` を組み立て。

3. **キャッシュ**  
   `get_cache` / `set_cache` は JSON 入出力。保存時 `cache_payload["monthly_usage"] = None`。

### 2.3 ブランチ・デプロイ

- **ステージング**: `develop` を Render が自動デプロイ。実行コミットは `GET /__debug_env` の `RENDER_GIT_COMMIT` で確認。
- **開発**: `feature/*` で作業し、検証後に `develop` にマージ。修正・テストは Docker 上で実行する。

## 3. 不整合が続く原因候補（完全調査と修正案 §2 の整理）

| 候補 | 内容 | 本計画での扱い |
|------|------|----------------|
| A. 案A/案1 がステージングに載っていない | デプロイ失敗・別ブランチ・ビルドキャッシュ | Phase 1 で実行コミット・コード一致を確認 |
| B. 古いキャッシュ構造 | 案A 以前のキャッシュが残りマージで上書きが効いていない | Phase 1 で Redis キー削除または TTL 待ちで再検証 |
| C. response_model による上書き | FastAPI の `response_model=DashboardResponse` が再構築で古い値を復元 | Phase 1 でレスポンス実体を確認。必要なら案2 でキャッシュ廃止 |
| D. Facility の同一セッション内二重読み | アイデンティティマップで古い Facility が返っている | Phase 2 で `get_monthly_usage` 内の Facility を明示再読（案3） |
| E. レプリカ遅延 | 読み取りがレプリカで古い plan_type | インフラ確認。Phase 3 で必要に応じて調査 |
| F. リクエストが届いていない / G. フロントの表示ソース | 別 URL・CDN・フロントの別ソース参照 | Phase 1 でブラウザのネットワークタブ・レスポンス JSON を確認 |

## 4. 修正計画（実施順序）

### Phase 1: 原因切り分け（実施必須）

**目的**: 「API レスポンスが既に正しいか」「どの層で古い値が出ているか」を確定する。

| 順 | 作業 | 内容 |
|----|------|------|
| 1.1 | ステージング実行コミットの確認 | `GET https://yadopera-backend-staging.onrender.com/__debug_env` で `RENDER_GIT_COMMIT` を取得。`develop` の最新コミットと一致するか確認。一致しない場合はプッシュ・デプロイ完了を待つ。 |
| 1.2 | プラン変更後の API レスポンス確認 | プラン変更後、ブラウザ開発者ツールで `GET /admin/dashboard`（またはフロントが実際に呼んでいる URL）のレスポンス本文を開く。`monthly_usage.plan_type` の値を確認。プランページの「現在プラン」と一致するか記録。 |
| 1.3 | 解釈 | ・**API の `monthly_usage.plan_type` が正（プランページと一致）** → 原因はフロント表示またはキャッシュ。フロントの利用状況カードのデータソース（`dashboardData.monthly_usage.plan_type` のみか）を再確認。  
　　・**API の `monthly_usage.plan_type` が誤（古いプラン）** → 原因はバックエンド。1.4 へ。 |
| 1.4 | キャッシュのリセット（バックエンドが誤の場合） | ステージングの Redis で該当キー `dashboard:data:facility_id={id}` を削除するか、TTL（5分）経過後に再度 1.2 を実施。それでも誤なら Phase 2 へ。 |
| 1.5 | オプション: ログ追加 | キャッシュヒット時に `get_monthly_usage` が呼ばれているか、および `monthly_usage_fresh.plan_type` を一時ログ出力。デプロイ後、2 回目以降のダッシュボード表示でログとレスポンスを照合。 |

### Phase 2: Facility の明示的再読（案3）

**目的**: SQLAlchemy の同一セッション内で古い Facility が使われている可能性を排除する（根本解決の一環）。

| 順 | 作業 | 内容 |
|----|------|------|
| 2.1 | get_monthly_usage 内で Facility を再読 | `dashboard_service.py` の `get_monthly_usage` で、`select(Facility)...` の**前**に `await self.db.expire_all()` を実行する。または該当 Facility のみ `await self.db.refresh(facility)` する場合は、先に select で取得した facility に対して `refresh`。推奨は **expire_all()** で、当該リクエスト内のセッションキャッシュを無効化し、続く select で DB から再読させる。 |
| 2.2 | テスト | Docker 上で `tests/test_dashboard_service_billing_period.py` の `get_monthly_usage` 関連テストを実行。既存テストが通ることを確認。 |
| 2.3 | デプロイ・検証 | develop にマージしステージングへデプロイ。Phase 1 と同様にプラン変更 → ダッシュボード API の `monthly_usage.plan_type` とプランページの一致を確認。 |

### Phase 3: それでも解消しない場合

| 対応 | 内容 |
|------|------|
| 案2（キャッシュ廃止）の検討 | ダッシュボードレスポンス全体の Redis キャッシュを廃止し、毎リクエストで DB から組み立てる。負荷・レスポンス時間を計測し、UX 許容範囲であれば採用。大原則「根本解決」に沿うが、負荷増は許容できる範囲で実施。 |
| DB レプリカ遅延の確認 | Render / Railway 等のマネージド DB で読み取りレプリカを使っている場合、プラン変更直後の読み取りがレプリカに振られ古い値が見えている可能性を調査。 |
| 関係文書の追記 | 完全調査と修正案ドキュメント §2 の候補 C, E の確認結果を記録。 |

## 5. 実装・検証のルール

- **Docker 必須**: 修正・単体テストは `docker compose` 上の backend で実行する。
- **ブランチ**: 作業は `feature/dashboard-plan-display-fix` 等の feature ブランチで行い、検証後に `develop` にマージ。
- **確認**: ステージングでは「プラン変更 → ダッシュボード表示 → 利用状況のプラン表示がプランページと一致すること」をブラウザで確認する。
- **暫定禁止**: フロントだけで「プランページのプランをダッシュボードに上書き表示する」などの正を置き換える実装は行わない（総括 §4・20260311 セッション記録 §6）。

## 8. 結果の説明と評価（2026-03-17 ステージング検証）

### 8.1 検証内容

利用者がステージング（`yadopera-frontend-staging.onrender.com`）で以下を実施した。

1. **プラン・請求ページ**（`/admin/billing`）でプラン変更（例: Mini → Small）を実行。
2. 「プランを変更しました。」モーダル表示後、プラン一覧に「Small (現在)」等が表示されることを確認。
3. **ダッシュボード**（`/admin/dashboard`）に遷移し、「利用状況」の質問数セクションに「Smallプラン」等が表示されることを確認。
4. 開発者ツールの Network タブで、`POST /api/v1/admin/plans/change`・`GET /api/v1/admin/plans`・`GET /api/v1/admin/dashboard` がすべて 200 OK であることを確認。

### 8.2 サーバーログから分かること

- `POST /api/v1/admin/plans/change` → 200 OK（複数回）。Stripe API `subscriptions` への post/get が 200。
- Webhook: `subscription.created`（facility_id=384）、`subscription.updated`（facility_id=384, status=incomplete）が記録。
- プラン変更直後に `GET /api/v1/admin/plans` および `GET /api/v1/admin/dashboard?_t=...` が 200 OK で実行され、レスポンスが返っている。

→ プラン変更が DB および Stripe に反映され、その直後にプラン取得・ダッシュボード取得が成功しており、**API が最新のプラン情報を返している**ことが裏付けられる。

### 8.3 評価

| 項目 | 結果 |
|------|------|
| プランページの表示 | 変更後のプラン（例: Mini, Small）が「(現在)」として正しく表示。 |
| ダッシュボードの表示 | 「利用状況」に同一プラン名（例: Miniプラン, Smallプラン）が表示。 |
| API の応答 | plans/change 200、続く plans・dashboard の GET が 200。 |
| 結論 | **プランページとダッシュボードのプラン表示不整合は、本検証時点で解消している**と評価する。既存の案A・案1・Fix A（プラン変更時のキャッシュ無効化）が有効に機能している。Phase 2・Phase 3 は不要。 |
