# Phase 0 Vercel代替案検討

**作成日**: 2025年11月27日  
**目的**: Vercelの運用問題を踏まえた代替案の検討  
**背景**: Vercelの自動デプロイが機能せず、CLIログインも複雑で時間を浪費

---

## 現状の問題点

### Vercelの運用上の問題

1. **自動デプロイが機能しない**
   - GitHubへのプッシュ後、自動デプロイがトリガーされない
   - 手動デプロイが必要

2. **CLIログインが複雑**
   - ブラウザ認証が必要
   - デバイス認証コードの入力が必要
   - 時間がかかる

3. **デプロイプロセスが不明確**
   - どのコミットがデプロイされているか分かりにくい
   - デプロイログが分かりにくい

4. **時間の浪費**
   - Google Analytics設定だけで数時間を費やしている
   - 本来の開発に集中できない

---

## 代替案の検討

### オプション1: GitHub Pages（推奨）

**メリット**:
- ✅ **完全無料**
- ✅ **自動デプロイ**: GitHubへのプッシュで自動デプロイ
- ✅ **シンプル**: 設定が簡単
- ✅ **カスタムドメイン対応**: `yadopera.com` を設定可能
- ✅ **SSL証明書**: 自動で設定される
- ✅ **GitHubと統合**: 既存のリポジトリで完結

**デメリット**:
- ⚠️ 静的サイトのみ（現在のランディングページは静的なので問題なし）
- ⚠️ ビルド時間がやや長い（1-2分）

**設定手順**:
1. GitHubリポジトリのSettings → Pages
2. Source: `main` ブランチ、`/landing` フォルダを選択
3. カスタムドメイン: `yadopera.com` を設定
4. 完了（自動デプロイ開始）

**所要時間**: 約5分

---

### オプション2: Netlify

**メリット**:
- ✅ **無料プランあり**
- ✅ **自動デプロイ**: GitHub連携で自動デプロイ
- ✅ **カスタムドメイン対応**
- ✅ **SSL証明書**: 自動で設定される
- ✅ **フォーム機能**: 内蔵（Formspree不要の可能性）

**デメリット**:
- ⚠️ 無料プランに制限あり（月100GB帯域幅）
- ⚠️ 設定がやや複雑

**設定手順**:
1. Netlifyアカウント作成
2. GitHubリポジトリを連携
3. Build settings: Root directory: `landing`
4. カスタムドメイン: `yadopera.com` を設定
5. 完了

**所要時間**: 約10分

---

### オプション3: Cloudflare Pages

**メリット**:
- ✅ **完全無料**
- ✅ **高速**: CloudflareのCDN
- ✅ **自動デプロイ**: GitHub連携
- ✅ **カスタムドメイン対応**
- ✅ **SSL証明書**: 自動で設定される

**デメリット**:
- ⚠️ 設定がやや複雑
- ⚠️ ドキュメントが少ない

**設定手順**:
1. Cloudflareアカウント作成
2. Pages → Create a project
3. GitHubリポジトリを連携
4. Build settings: Root directory: `landing`
5. カスタムドメイン: `yadopera.com` を設定
6. 完了

**所要時間**: 約10分

---

### オプション4: Render.com（既にアカウントあり）

**メリット**:
- ✅ **既にアカウント準備済み**
- ✅ **自動デプロイ**: GitHub連携
- ✅ **カスタムドメイン対応**
- ✅ **SSL証明書**: 自動で設定される

**デメリット**:
- ⚠️ 静的サイトホスティングは有料プランが必要な可能性
- ⚠️ 本番アプリ用に使う予定（混在させたくない可能性）

**設定手順**:
1. Render.comダッシュボード
2. New → Static Site
3. GitHubリポジトリを連携
4. Root directory: `landing`
5. カスタムドメイン: `yadopera.com` を設定
6. 完了

**所要時間**: 約10分

---

### オプション5: Vercelを継続（現状維持）

**メリット**:
- ✅ 既に設定済み
- ✅ カスタムドメイン設定済み

**デメリット**:
- ❌ 自動デプロイが機能しない
- ❌ CLIログインが複雑
- ❌ 時間を浪費

**改善策**:
- Vercelダッシュボードから手動デプロイを習慣化
- または、GitHub Actionsで自動デプロイを設定

---

## 推奨案

### 最優先: GitHub Pages（オプション1）

**理由**:
1. **最もシンプル**: 設定が簡単で、時間を浪費しない
2. **完全無料**: コストがかからない
3. **自動デプロイ**: GitHubへのプッシュで自動デプロイ
4. **既存のリポジトリで完結**: 新しいサービスを覚える必要がない
5. **カスタムドメイン対応**: `yadopera.com` を設定可能

**移行手順**:
1. GitHubリポジトリのSettings → Pages
2. Source: `main` ブランチ、`/landing` フォルダを選択
3. カスタムドメイン: `yadopera.com` を設定（DNS設定は既に完了しているので、ドメインを追加するだけ）
4. 完了（約5分）

**注意**: VercelからGitHub Pagesに移行する場合、DNS設定を変更する必要があります。

## 時間の浪費を避けるための提案

### 現状の時間浪費

- Google Analytics設定: 数時間
- Vercelのデプロイ問題: 数時間
- **合計**: 半日以上を浪費

### 本来の目的

- **Phase 0**: 準備期間（1週間）
- **Phase 1**: MVP開発（4週間）
- **時間は限られている**

### 推奨アクション

1. **即座にGitHub Pagesに移行**（約5分）
   - 設定が簡単
   - 自動デプロイが機能する
   - 時間を浪費しない

2. **Vercelは後で検討**
   - 本番アプリ（Phase 1以降）で使用する予定
   - ランディングページはGitHub Pagesで十分

3. **Google Analytics設定を完了**
   - GitHub Pages移行後、すぐに設定可能
   - 自動デプロイが機能するので、設定が反映される

## 結論

### 推奨: GitHub Pagesに移行

**理由**:
1. **時間を浪費しない**: 設定が簡単で、すぐに完了
2. **自動デプロイが機能する**: GitHubへのプッシュで自動デプロイ
3. **完全無料**: コストがかからない
4. **既存のリポジトリで完結**: 新しいサービスを覚える必要がない

**移行時間**: 約5分

**効果**:
- 時間の浪費を避けられる
- 本来の開発に集中できる
- Google Analytics設定もすぐに完了できる
