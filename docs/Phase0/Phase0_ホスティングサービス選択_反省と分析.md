# Phase 0 ホスティングサービス選択 反省と分析

**作成日**: 2025年11月27日  
**目的**: なぜ最初からGitHub Pagesを推奨しなかったのか、その理由と反省  
**背景**: Vercelの運用問題を踏まえた選択プロセスの見直し

---

## 質問: なぜ最初からGitHub Pagesを推奨しなかったのか？

### 実際の選択プロセス

**当初の選択**: Vercel

**選択理由（推測）**:
1. **一般的な認知度**: VercelはNext.jsやReactアプリでよく使われている
2. **ドキュメント**: アーキテクチャ設計書でVercelが言及されていた可能性
3. **自動デプロイ**: GitHub連携で自動デプロイが機能すると思っていた
4. **カスタムドメイン**: 簡単に設定できると思っていた

**実際の問題**:
- 自動デプロイが機能しなかった
- CLIログインが複雑
- 時間を浪費した

---

## GitHub Pagesのデメリット（実際の評価）

### デメリット1: ビルド時間がやや長い

**評価**: ⚠️ **軽微**

- GitHub Pages: 1-2分
- Vercel: 30秒-1分

**影響**: 静的HTMLなので、ビルド時間の差はほとんど問題にならない

### デメリット2: 静的サイトのみ

**評価**: ✅ **問題なし**

- ランディングページは静的HTMLなので、問題なし
- サーバーサイドの処理は不要

### デメリット3: カスタムドメイン設定がやや複雑

**評価**: ⚠️ **軽微**

- GitHub Pages: DNS設定が必要（CNAMEレコード）
- Vercel: DNS設定が必要（AレコードまたはCNAMEレコード）

**実際**: どちらもDNS設定が必要なので、差はほとんどない

### デメリット4: 機能が限定的

**評価**: ✅ **問題なし**

- ランディングページに必要な機能はすべて揃っている
- フォーム送信: Formspreeを使用（Vercelでも同じ）
- Google Analytics: どちらでも同じように設定可能

---

## 実際のデメリット評価

### 結論: **実質的なデメリットはほとんどない**

**理由**:
1. **静的HTML**: ランディングページは静的HTMLなので、GitHub Pagesで十分
2. **自動デプロイ**: GitHub Pagesも自動デプロイが機能する
3. **カスタムドメイン**: どちらも設定可能
4. **コスト**: どちらも無料
5. **設定の簡単さ**: GitHub Pagesの方が簡単

---

## なぜ最初から推奨しなかったのか？

### 理由1: 選択プロセスの問題

**問題点**:
- 各サービスの比較検討が不十分だった
- 実際の運用を考慮していなかった
- 「一般的に使われている」という理由だけで選択した可能性

**反省**:
- 実際の運用を考慮すべきだった
- 各サービスのメリット・デメリットを比較すべきだった
- 時間の浪費を避けるべきだった

### 理由2: ドキュメントの影響

**問題点**:
- アーキテクチャ設計書でVercelが言及されていた可能性
- 既存のドキュメントに従った可能性

**反省**:
- ドキュメントを盲目的に従うのではなく、実際の運用を考慮すべきだった
- ドキュメントを更新すべきだった

### 理由3: 経験不足

**問題点**:
- Vercelの実際の運用経験が不足していた
- GitHub Pagesの利点を理解していなかった

**反省**:
- 実際の運用を考慮すべきだった
- より簡単な方法を選択すべきだった

---

## 正しい選択プロセス

### ステップ1: 要件の明確化

**ランディングページの要件**:
- ✅ 静的HTML
- ✅ カスタムドメイン（`yadopera.com`）
- ✅ 自動デプロイ（GitHubプッシュで自動反映）
- ✅ 無料
- ✅ 設定が簡単

### ステップ2: 各サービスの比較

| サービス | 設定時間 | 自動デプロイ | カスタムドメイン | コスト | 推奨度 |
|---------|---------|------------|----------------|--------|--------|
| **GitHub Pages** | 5分 | ✅ | ✅ | 無料 | ⭐⭐⭐⭐⭐ |
| Vercel | 30分+ | ❌（問題あり） | ✅ | 無料 | ⭐ |
| Netlify | 10分 | ✅ | ✅ | 無料 | ⭐⭐⭐⭐ |
| Cloudflare Pages | 10分 | ✅ | ✅ | 無料 | ⭐⭐⭐⭐ |

### ステップ3: 実際の運用を考慮

**考慮すべき点**:
- 設定の簡単さ
- 自動デプロイの確実性
- 時間の浪費を避ける
- トラブルシューティングの容易さ

### ステップ4: 選択

**正しい選択**: **GitHub Pages**

**理由**:
1. 設定が最も簡単（5分）
2. 自動デプロイが確実に機能する
3. 既存のリポジトリで完結
4. 時間を浪費しない

## 反省と改善

### 反省点

1. **選択プロセスの問題**
   - 各サービスの比較検討が不十分だった
   - 実際の運用を考慮していなかった

2. **時間の浪費**
   - Vercelの運用問題で数時間を浪費
   - 本来の開発に集中できなかった

3. **ドキュメントの影響**
   - 既存のドキュメントに盲目的に従った
   - 実際の運用を考慮していなかった

### 改善策

1. **選択プロセスの改善**
   - 各サービスの比較検討を徹底する
   - 実際の運用を考慮する
   - 時間の浪費を避ける

2. **ドキュメントの更新**
   - 実際の運用経験を反映する
   - より簡単な方法を推奨する

3. **経験の蓄積**
   - 実際の運用を経験する
   - 問題点を記録する
