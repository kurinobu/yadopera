# リードゲットオプション（決済なしクーポン）実装計画書

**作成日**: 2026年2月21日  
**対象**: ゲストのメールアドレス獲得を目的とするクーポン発行機能（支払いを伴わない）  
**準拠**: 要約定義書 v0.3.18・アーキテクチャ設計書 v0.3.7・大原則

---

## 1. 目的と位置づけ

- **目的**: 追加決済を伴わずにゲストのメールアドレスを取得し、宿泊施設がOTA外の顧客接点（リード）を獲得する。
- **位置づけ**:
  - 決済機能（payment_option_plan：レイトチェックアウト／延泊）とは**完全に独立**した機能。
  - PoC段階でも導入しやすく、心理的・法務的ハードルが低い。
  - YadOPERAを「FAQチャット」から「集客・再訪導線ツール」へ進化させる中核機能の第一歩。

本実装は `lead_no_payment_plan.md` の「決済なしリード獲得（クーポン取得機能）」を、コードベースの現状に合わせて具体化した実装計画である。

---

## 2. 準拠する大原則（要約定義書 0. 大原則）

実装は以下を最優先で遵守する。

| 原則 | 本機能での適用 |
|------|----------------|
| 根本解決 > 暫定解決 | クーポン設定・リード保存・メール送信を正規のDB・API・サービスで実装する。暫定のローカル保存や手動運用に依存しない。 |
| シンプル構造 > 複雑構造 | 固定フッターは1コンポーネントに集約し、後続オプション（延長・延泊）も同じ並びで拡張する。 |
| 統一・同一化 > 特殊独自 | 既存の施設公開API・認証済み管理API・メール送信（Brevo）・Alembicマイグレーションのパターンに従う。 |
| 具体的 > 一般 | エンドポイント名・テーブル名・コンポーネント配置・多言語キーを本計画書で明示する。 |
| 拙速 < 安全確実 | 実装後はDocker環境でテストし、ステージングで動作確認してから本番を検討する。 |
| Docker環境必須 | すべての修正・テストは docker-compose で実行する。 |

## 3. アクションフロー整理

### 3.1 ゲスト側フロー

1. **ゲストがゲストページにアクセス**  
   - QRコードまたは `https://yadopera.com/f/{facility_id}` 等で言語選択 → Welcome / Chat のいずれかに遷移。
2. **固定フッターのCTAをタップ**  
   - ボタンは**デフォルトで「オトクなクーポン」**とする。「次回公式サイトから予約で ◯％オフ」は文言が長くボタンに収まらないため、ボタン表記には使わない。割引率・利用条件はエントリー画面や送付メールで伝える。
3. **エントリー画面へ遷移**  
   - ゲスト名（任意または必須のいずれか、仕様で固定）とメールアドレスを入力してエントリー。
4. **クーポン送付**  
   - 指定したメールアドレスにクーポン内容（施設名・割引率・利用条件・有効期限）を記載したメールが送信される。
5. **完了メッセージ表示**  
   - 「クーポン送付完了」等のメッセージを表示。
6. **次回予約・利用**  
   - ゲストは次回、宿泊施設の公式サイトから予約し、現地で支払い時にクーポン提示により割引を受ける（**YadOPERAは決済に関与しない**）。

### 3.2 宿泊施設側フロー

1. **クーポン設定（管理画面）**  
   - クーポン有効/無効（ON/OFF）  
   - 割引率（%）  
   - クーポン文言（任意編集可）  
   - 有効期限（例：発行日から6ヶ月）
2. **リードの確認・活用**  
   - 管理画面でエントリー一覧を確認し、必要に応じてCSVダウンロード。顧客名簿として利用。
3. **支払い時の割引**  
   - 現地でゲストがクーポン提示した場合、施設側が支払い時に割引して決済する。**YadOPERAは決済に関与しない。**

## 4. UI/UX方針（固定フッター・横スライド）

- **固定フッター**: ゲスト向け画面（言語選択・Welcome・Chat）で、画面下部に**固定フッター**を設け、その中にクーポンCTAを配置する。**ボタン表記はデフォルトで「オトクなクーポン」**とし、割引率などの詳細はエントリー画面・送付メールで伝える（長文言はボタンに収まらないため使用しない）。
- **拡張性**: 後に「レイトチェックアウト」「延泊」等のオプションを同じ並びに追加する。横幅が不足する場合は**横スライド（水平スクロール）**で対応する。
- **配置**: 既存のChat画面の「メッセージ入力（固定フッター）」はそのまま維持し、その上段、またはゲストレイアウト全体で共通の「オプション用固定フッター」を設ける形が妥当。  
  - 案A: `GuestLayout` に固定フッターを追加し、全ゲストページで表示する。  
  - 案B: Welcome / Chat 等の各ページで「オプション用フッター」を同じデザインで埋め込む。  
  - **推奨**: 案A（レイアウトで一括管理し、施設・言語に応じて表示内容を切り替える）。
- **表示条件**: 施設でクーポンが有効なときのみ、固定フッターにクーポンCTAを表示する。無効の場合は非表示または後続オプションのみ表示。

## 5. コードベース調査結果と現状

### 5.1 ゲスト側

- **ルート**: `frontend/src/router/guest.ts` で `/f/:facilityId`（言語選択）、`/f/:facilityId/welcome`、`/f/:facilityId/chat` が定義されている。
- **レイアウト**: `GuestLayout.vue` が `layout: 'guest'` でスロットのみ表示。現状、共通フッターはない。
- **施設情報取得**: `facilityApi.getFacility(slug, location, language)` → `GET /api/v1/facility/{slug}`。  
  - レスポンス: `facility`（id, name, slug, email, phone, check_in_time, check_out_time, wifi_ssid, plan_type, available_languages）、`top_questions`。
- **Chat.vue**: ヘッダー固定・メッセージリスト・**メッセージ入力の固定フッター**（`MessageInput`）・エラー表示の構成。ここにクーポンCTAを「上段の固定バー」として追加するか、レイアウト側でまとめるかで検討。

### 5.2 バックエンド

- **施設モデル**: `backend/app/models/facility.py` の `Facility`。現状、クーポン関連カラムはない。
- **施設公開API**: `backend/app/api/v1/facility.py` の `GET /facility/{slug}`。`FacilityService.get_facility_public_info` を呼び、`FacilityPublicResponse` を返している。クーポン設定を「有効時のみ公開情報に含める」形で拡張する。
- **施設設定（管理画面）**: `backend/app/api/v1/admin/facility.py` の `GET/PUT /admin/facility/settings`。ここにクーポン設定の取得・更新を追加する。
- **メール送信**: `backend/app/services/email_service.py` で **Brevo** を使用。認証用メールの送信実装がある。**クーポン送付メール**を新規メソッドで追加する（Brevo の利用と設定は後述 §7.2 参照）。

### 5.3 データベース

- **マイグレーション**: `backend/alembic/versions/` に 001〜014 等が存在。新規テーブル・カラムは Alembic で追加する。
- **既存テーブル**: facilities, users, conversations, messages, faqs, faq_translations, guest_feedback, escalations 等。**リード（ゲスト名・メール・施設ID・エントリー日時）**を保存する新規テーブルと、施設ごとの**クーポン設定**を保存するカラム（または別テーブル）が必要。

### 5.4 管理画面

- 施設設定画面でクーポンON/OFF・割引率・文言・有効期限を編集可能にする。
- リード一覧・CSVダウンロード用の管理APIと画面（または既存ダッシュボードのサブセクション）を追加する。
- **ダッシュボード「その他の統計」欄**: 宿泊施設管理者用ダッシュボードの「その他の統計」に**「クーポン発行（メールアドレス取得数）」**を表示する。ユーザーがリード獲得効果を一目で確認でき、評価向上につながる。

## 6. システム仕様（最小構成）

### 6.1 データベース

- **facilities テーブル拡張（クーポン設定）**  
  - `coupon_enabled` (Boolean, default False)  
  - `coupon_discount_percent` (Integer, nullable, 例: 5〜20)  
  - `coupon_description` (Text, nullable, 施設が編集する文言)  
  - `coupon_validity_months` (Integer, nullable, 発行日から何ヶ月有効か)
- **新規テーブル: guest_leads**  
  - `id` (PK)  
  - `facility_id` (FK → facilities)  
  - `guest_name` (String, nullable または not null のいずれかに仕様で統一)  
  - `email` (String, not null)  
  - `created_at` (Timestamp)  
  - 必要に応じて `coupon_sent_at`, `language` 等を追加可能。

### 6.2 バックエンドAPI

- **公開API（ゲスト用）**  
  - `GET /api/v1/facility/{slug}` のレスポンスに、クーポンが有効な場合のみ `coupon` オブジェクトを含める。  
    - 例: `coupon: { enabled: true, discount_percent: 10, description: "...", validity_months: 6 }`  
  - `POST /api/v1/lead/entry`（または `POST /api/v1/facility/{slug}/lead`）  
    - Body: `{ "guest_name": "...", "email": "..." }`  
    - 施設IDは slug または path から解決。  
    - 同一施設・同一email の重複登録方針（上書き・拒否・許可のいずれか）を決める。
- **管理API（認証必須）**  
  - `GET /api/v1/admin/facility/settings` のレスポンスにクーポン設定を追加。  
  - `PUT /api/v1/admin/facility/settings` のリクエストでクーポン設定を更新可能にする。  
  - `GET /api/v1/admin/leads` でリード一覧取得（ページネーション推奨）。  
  - `GET /api/v1/admin/leads/export` または `GET /api/v1/admin/leads?format=csv` でCSVダウンロード。  
  - **ダッシュボード**: `GET /api/v1/admin/dashboard` のレスポンスに **`coupon_lead_count`**（クーポン発行数＝メールアドレス取得数）を追加する。表示は「その他の統計」欄で行う（累計または請求期間内のいずれか、仕様で固定）。
- **メール送信（Brevo）**  
  - クーポン送付メールは**既存の Brevo 利用**とする。`EmailService` に `send_coupon_email(...)` を追加し、リード登録処理内で保存後に呼び出す。  
  - 送信失敗時はログとリトライ方針を決める（要約定義書の「拙速 < 安全確実」に従い、失敗時はエラー返却または再送可能にする）。  
  - Brevo の設定・環境変数は §7.2 を参照。

### 6.3 フロントエンド

- **ゲスト**  
  - **GuestLayout**: 固定フッター用の新コンポーネント（例: `GuestOptionFooter.vue`）を配置。施設情報から `coupon.enabled` を見てクーポンCTAを表示。**ボタン表記はデフォルトで「オトクなクーポン」**とする。  
  - クーポンCTAクリック時: エントリー用のモーダルまたは専用ルート（例: `/f/:facilityId/coupon`）で「ゲスト名」「メールアドレス」入力フォームを表示。  
  - 送信後: 完了メッセージ表示。  
  - 固定フッター内のボタンが複数になる場合は、横スライド可能なコンテナで並べる。
- **管理画面**  
  - 施設設定画面に「クーポン設定」セクションを追加（ON/OFF、割引率、文言、有効期限）。  
  - リード一覧・CSVダウンロード用の画面または既存ダッシュボードのタブを追加。  
  - **ダッシュボード「その他の統計」**: 既存の「総質問数」「自動応答率」「平均信頼度」「未解決質問」に加え、**「クーポン発行（メールアドレス取得数）」**を `StatsCard` で表示する。値は `GET /api/v1/admin/dashboard` の `coupon_lead_count` を表示。ユーザー評価向上を目的とする。

### 6.4 多言語

- ゲスト向けラベル・ボタン・完了メッセージは、既存の言語選択（`lang`）に合わせて表示する。  
- 施設が編集する「クーポン文言」は、現状は単一言語でよい（必要に応じて後から多言語化）。

## 7. 法務・運用上の留意点／メール送信（Brevo）

### 7.1 法務・運用上の留意点（lead_no_payment_plan と同一）

- **利用目的の明示**: 「次回予約のご案内のため」等をエントリー画面に表示する。
- **プライバシーポリシー**: エントリー画面にプライバシーポリシーへのリンクを表示する。
- **YadOPERAは決済に関与しない**: クーポン利用可否の最終判断は宿泊施設側。割引は現地決済で施設が行う。
- **個人情報**: リードは施設IDと紐付けて保存し、管理画面は当該施設の管理者のみアクセス可能とする（既存の施設紐づき認証を流用）。

### 7.2 メール送信（Brevo）の利用と設定

クーポン送付メールは**既存の Brevo（Sendinblue）** を利用する。認証用メールと同じ送信基盤のため、**追加の Brevo 契約やトランザクションメール種別の新規作成は不要**。既存の環境変数と送信元設定で動作する。

**必要な環境変数（backend `.env`）**

| 変数名 | 説明 | 例 |
|--------|------|-----|
| `BREVO_API_KEY` | Brevo API キー（必須・本番では未設定時起動エラー） | `xkeysib-...` |
| `BREVO_SENDER_EMAIL` | 送信元メールアドレス（任意・未設定時は `noreply@yadopera.com`） | `noreply@yadopera.com` |
| `BREVO_SENDER_NAME` | 送信元表示名（任意・未設定時は `YadOPERA`） | `YadOPERA` |

**Brevo 側の設定（既存の認証メールで済んでいれば追加不要）**

- **API キー**: [Brevo 管理画面](https://app.brevo.com/settings/keys/api) で API Key を発行し、`BREVO_API_KEY` に設定する。
- **送信元ドメイン**: 送信元メール（`BREVO_SENDER_EMAIL`）のドメインについて、Brevo の「送信ドメイン認証」が完了していること。未認証だと迷惑メール扱いや送信制限の可能性がある。
- **テンプレート**: クーポン送付はコード内で HTML 本文を組み立てて送る想定（認証メールと同様）。Brevo のテンプレート機能を使う場合は、テンプレートID用の環境変数追加とコード対応を別途検討する。

**動作**

- 本番で `BREVO_API_KEY` が未設定の場合、アプリ起動時に `ValueError` で起動失敗する（要約定義書・既存 config と同じ）。
- 開発環境で未設定の場合は警告ログのみで起動し、クーポン送信時に `EmailService` 初期化で `ValueError` となる。

## 8. 実装ステップ（推奨順）

**ステップ着手前に必ず実施すること：Brevo 設定の確認**

- Brevo は**宿泊施設管理者の新規登録時メールアドレス確認**で既に利用しており、本番（Render）では `BREVO_API_KEY` が設定済みです。
- 実装・テスト中に Brevo 設定不足でエラーにならないよう、**各ステップを開始する前に**以下を確認してください。
  - **ローカル・Docker 環境**: `backend/.env` に `BREVO_API_KEY` が設定されていること。未設定だとクーポン送信やメール送信を伴うテストで `EmailService` 初期化時に失敗します。
  - **ステージング／本番（Render）**: 環境変数に `BREVO_API_KEY` が設定されていること（本番では未設定時はアプリ起動に失敗します）。
- 設定方法は §7.2 を参照。既存の認証メール送信と同じキー・送信元で問題ありません。
