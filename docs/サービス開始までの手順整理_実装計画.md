## 1. 目的と完了定義

### 1.1 目的

「サービス開始までの手順の整理」を、実行可能なランブックとして整備する。  
単なる概念整理ではなく、**誰が・いつ・何を・どう検証して次へ進むか**を明確にする。

### 1.2 完了定義（Definition of Done）

以下が揃った時点で本タスクを完了とする。

1. `docs` に本番ローンチ手順書（Runbook）を作成し、チェックボックスで進行管理できる。
2. ステージング -> 本番の移行順序（コード・DB・環境変数・決済・監視）が固定化されている。
3. Go/No-Go 判定基準が明文化され、緊急ロールバック手順が定義されている。
4. 総括・要約定義書に参照が追加され、残存課題から実行フェーズへ移行できる。

## 2. 調査結果サマリー（現況）

### 2.1 ブランチ・デプロイ運用

- 現在の運用実態:
  - `develop` = ステージング（`origin/develop` と一致）
  - `main` = 本番
  - `feature/*` = 開発ブランチ
- `render.yaml` は **ステージング設定のみ定義**（`yadopera-backend-staging`, `yadopera-frontend-staging`）。
- ステージングの実行コミット確認手段として `GET /__debug_env` が実装済み（`RENDER_GIT_COMMIT` を返却）。

### 2.2 Docker・マイグレーション運用

- `docker-compose.yml` で `postgres`/`redis`/`backend`/`frontend` が揃っており、ローカル再現性は確保されている。
- `backend/alembic/versions` は `001` から `022` まで存在し、Stripe含む最新スキーマ変更は履歴化済み。
- 本番相当の起動で必要な `alembic upgrade head` は設計書・既存運用で一貫している。

### 2.3 ローンチ観点の未整備点（更新後）

- 本番開始Runbookの初版は作成済み（`docs/サービス開始までの手順_Runbook.md`）。今後は運用実態に合わせた具体値の追記と実施記録の蓄積が必要。
- `render.yaml` に本番サービス定義がないため、**本番設定を手順として別管理**する必要がある。
- 旧手順書には Vercel 前提の記述が残っており、現行方針（Vercel不使用・GitHub Pages/Render運用）と不整合の箇所がある。

## 3. 大原則への適用

| 大原則 | 本計画での適用 |
|--------|----------------|
| 根本解決 > 暫定解決 | 単発デプロイメモではなく、事前準備からロールバックまで一気通貫の手順書を作る。 |
| シンプル構造 > 複雑構造 | ブランチ運用は `feature/* -> develop -> main` を維持し、例外運用を増やさない。 |
| 統一・同一化 > 特殊独自 | 検証は Docker と `__debug_env` に統一。DB反映は Alembic に統一。 |
| 具体的 > 一般 | 作業順、実行コマンド、確認API、責任者、完了条件を明示する。 |
| 拙速 < 安全確実 | Go/No-Go基準、凍結時間、ロールバック条件を先に定義してから実施する。 |
| Docker環境必須 | すべての修正・検証は Docker で行い、ローカル直接実行のみの確認は完了条件に含めない。 |

## 6. サービス開始までのロードマップ（実作業の順番）

> **この節が「何から始めるか」の答えです。**  
> 旧来の **§7「実装ステップ計画」内の Step 0〜3** は、**手順書ドキュメントと総括・要約への参照を揃える作業** に限定されたステップでした。**本番SaaS・Stripe本番・LP/公式の本番公開**の実作業手順ではありません。混乱のもとになっていた点をここで明文化します。  
> **LP/公式サイト（`landing/`）の実装・本番公開は、§6.0 全テスト完了ゲート通過後**（§6.2 順9）。  
> **Stripe 本番キー（`STRIPE_*` live）を本番環境に入れるのも §6.0 通過後**（§6.2 順7）。

### 6.0 全テスト完了ゲート（次フェーズ進行条件）

**目的**: Stripe 本番キー投入・本番 LP 実装・本番への LP 反映など、**誤りが金銭・顧客影響につながる操作の直前**に、テストの抜けをなくす。

**適用**: 少なくとも次に進む前に §6.0 を満たすこと。

- **Stripe 本番**の `STRIPE_*`（live）を **Render 本番（または本番相当）の環境変数に設定する**こと。
- **`landing/` の実装**（新規ページ・改修・本番向け文案確定を伴う作業）および **`main` / GitHub Pages への本番反映**。

**ゲート内容（すべて満たすこと。未満なら上記には進まない）**

1. **自動化できるテストはすべて実行し、合格させる**  
   - リポジトリで定義されている **pytest / スクリプト / CI / Docker 上の検証コマンド**など、**人手なしで再現できる範囲は漏れなく実行**する。  
   - 対象一覧は **`§14`（ドラフト）を正としてチェックリスト化**し、リリース直前に Runbook 実施記録または `docs/` に証跡（ログ・日付・実施者）を残す。  
   - **自動テストの記録例**: `docs/20260323_6.0品質ゲート_実行記録.md`（§6.5・手動回帰・Owner 確認は含まない場合があるので、§6.0 全体の完了判定と混同しないこと）。

2. **テスト用データの生成が必要なテスト、外部連携を伴うテスト**  
   - Stripe **test mode**、Brevo、OpenAI、Render ステージング API 等を含む。  
   - **Render の設定値、パスコード、`DATABASE_URL`、シークレット**がないと実行できないものは、**Owner（利用者）と Dev がペアで**実施する（秘密値はチャット等に平文で残さない）。

3. **完了条件**  
   - 上記 1・2 の **未実行・不合格・スキップが残っていない**こと。  
   - Owner が「全テスト完了」を確認したことを **文書または Runbook チェック**に記録する。  
   - **1 件でも未了がある場合は、Stripe live 投入・LP 実装・本番 LP 反映のいずれにも進まない。**
   - **ステージングで実施済みの記録のみでは完了扱いにしない。** 本番での動作確認が必須と定義した項目（例: 開発者管理画面、FAQ CSV 一括登録の方法D）は、**本番実施記録**が揃うまで §6.0 未完了とする。

**補足**

- §6.2 **順2（品質ゲート）** は、Docker 回帰・ステージングでの **§6.5** 完走とあわせ、**§6.0 の中核**として位置づける。順2 完了＝§6.0 の主要部分完了とみなすが、**リポジトリに存在するその他の自動テスト**も 1 に含め、順2 のチェックリストに列挙する。  
- **本番環境での検証**が必要な場合は、**§6.2 順6** のとおり本番 DB に**検証用データのみ**投入してよい（ステージングからの一括レコード移行は行わない）。

#### 6.1.1 Render：本番サービス立ち上げは計画に含まれるか

**含まれますが、粒度は「ロードマップ＋確認項目」までです。**

| 観点 | 本計画・Runbook での位置づけ |
|------|------------------------------|

| **まだ無いもの** | Render ダッシュボードの **画面操作のスクリーンショット付き手順**や、**ステージングを Blueprint 複製するか**の公式手順の丸写しは、**本書には未記載**（必要なら `docs/Deployment/` 配下に **Render 本番サービス作成手順.md** を別紙で追加するのがよい）。 |

**本番立ち上げ時の最低チェックリスト（実施内容は計画に含意される）**

1. **Backend（本番）**: Web Service 新規 → 同一リポジトリ・**ブランチ `main`**・`backend/Dockerfile`・`preDeployCommand: alembic upgrade head`（`render.yaml` と揃える）。本番 `DATABASE_URL` / `REDIS_URL`、本番 `CORS_ORIGINS`（本番フロント URL）、`ENVIRONMENT=production`、Stripe・メール等は **本番値**。
2. **Frontend（本番）**: Static Site または現行と同型の設定 → **ビルド**は staging と同様、`VITE_API_BASE_URL` は **本番 Backend の公開 URL**。
3. **カスタムドメイン・HTTPS**: API / フロントの DNS・証明書（Render 管理）を順3〜5で決めた URL に合わせる。
4. **任意**: 将来 `render.yaml` に本番サービスを追記し Blueprint で管理するかは運用方針（現状は **未コミット**でも可）。

→ **「本番用 Render の新規サービス作成と環境設定」は、§6.2 の順3〜5 と Runbook §1.2・§2.3 でカバーする対象だが、ダッシュボードの具体手順は別紙追記が推奨**、と整理できる。

## 8. Go / No-Go 基準（手順書に必須記載）

### Go 条件
