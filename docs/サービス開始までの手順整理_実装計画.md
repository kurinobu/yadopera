## 1. 目的と完了定義

### 1.1 目的

「サービス開始までの手順の整理」を、実行可能なランブックとして整備する。

### 1.2 完了定義（Definition of Done）

以下が揃った時点で本タスクを完了とする。

1. `docs` に本番ローンチ手順書（Runbook）を作成し、チェックボックスで進行管理できる。
2. ステージング -> 本番の移行順序（コード・DB・環境変数・決済・監視）が固定化されている。
3. Go/No-Go 判定基準が明文化され、緊急ロールバック手順が定義されている。
4. 総括・要約定義書に参照が追加され、残存課題から実行フェーズへ移行できる。

## 2. 調査結果サマリー（現況）

### 2.1 ブランチ・デプロイ運用

- 現在の運用実態:
  - `develop` = ステージング（`origin/develop` と一致）
  - `main` = 本番
  - `feature/*` = 開発ブランチ
- `render.yaml` は **ステージング設定のみ定義**（`yadopera-backend-staging`, `yadopera-frontend-staging`）。
- ステージングの実行コミット確認手段として `GET /__debug_env` が実装済み（`RENDER_GIT_COMMIT` を返却）。

### 2.2 Docker・マイグレーション運用

- `docker-compose.yml` で `postgres`/`redis`/`backend`/`frontend` が揃っており、ローカル再現性は確保されている。
- `backend/alembic/versions` は `001` から `022` まで存在し、Stripe含む最新スキーマ変更は履歴化済み。
- 本番相当の起動で必要な `alembic upgrade head` は設計書・既存運用で一貫している。

### 2.3 ローンチ観点の未整備点（更新後）

- 本番開始Runbookの初版は作成済み（`docs/サービス開始までの手順_Runbook.md`）。今後は運用実態に合わせた具体値の追記と実施記録の蓄積が必要。
- `render.yaml` に本番サービス定義がないため、**本番設定を手順として別管理**する必要がある。
- 旧手順書には Vercel 前提の記述が残っており、現行方針（Vercel不使用・GitHub Pages/Render運用）と不整合の箇所がある。

## 3. 大原則への適用

| 大原則 | 本計画での適用 |
|--------|----------------|
| 根本解決 > 暫定解決 | 単発デプロイメモではなく、事前準備からロールバックまで一気通貫の手順書を作る。 |
| シンプル構造 > 複雑構造 | ブランチ運用は `feature/* -> develop -> main` を維持し、例外運用を増やさない。 |
| 統一・同一化 > 特殊独自 | 検証は Docker と `__debug_env` に統一。DB反映は Alembic に統一。 |
| 具体的 > 一般 | 作業順、実行コマンド、確認API、責任者、完了条件を明示する。 |
| 拙速 < 安全確実 | Go/No-Go基準、凍結時間、ロールバック条件を先に定義してから実施する。 |
| Docker環境必須 | すべての修正・検証は Docker で行い、ローカル直接実行のみの確認は完了条件に含めない。 |

## 8. Go / No-Go 基準（手順書に必須記載）

### Go 条件

- `main` 反映後の本番で health check 正常（API/フロント）。
- マイグレーション適用後の主要機能（認証/チャット/FAQ/請求）が正常。
- 決済導線（プラン変更/請求履歴/領収書）が致命障害なく動作。

### No-Go 条件

- `alembic` 適用失敗、または schema 不整合。
- ログイン不能、主要API 5xx 多発、Webhooks 連続失敗。
- 課金や契約状態の不整合が解消不能。

## 12. 実装着手時の進め方

### 12.1 ステージングでの PoC 実行と本計画の関係

**可能です。** むしろ総括上 Phase 3（PoC）は **`develop` → ステージング** が主戦場であり、本計画の **§6.2 順2（品質ゲート）** は **Docker＋ステージング（Stripe test mode）** で完走させる想定です。

| 区分 | PoC・ステージングとの関係 |
|------|---------------------------|
| **順1 準備** | 総括の残件・ブロッカーを PoC とあわせて整理できる。 |
| **順2 品質ゲート** | **ステージング上で実施**（§6.5・Runbook §1.4）。PoC 施設・利用シナリオのフィードバックをここに取り込める。 |
| **順3 本番インフラ設計** | コードを本番に出さず、並行して「本番用の設計メモ・コスト・URL を決める」作業だけ進められる。 |
| **順4〜8** | **本番環境・本番 DB・Stripe live 等**。PoC が継続中でも**準備の文書化**はできるが、**本番への実デプロイ・live 切替**は、Owner 判断・ブロッカー解消・Go/No-Go のタイミングで切り出す。 |
| **順9 LP/公式本番** | 計画どおり **品質ゲートと SaaS 本番 Go の後**。PoC だけ先に進めていても、**LP 本番公開はブロックしてよい**。 |

**注意**: Runbook の「T-14日」などは **本番リリース当日**を原点とした目安。PoC 中は **staging で Runbook §3 相当の手順をリハーサル**し、本番日が決まったら同じチェックを本番 URL で実行する、と捉える。

### 12.2 AI アシスタント（チャット等）への依頼の出し方

実装・運用手順の実行や Runbook 追記を AI に依頼する際は、**範囲・現状・完了条件**を揃えると成功しやすい。

#### 依頼に含めるべき骨子

| 項目 | 説明 |
|------|------|
| **目的** | 例: 本番ローンチ作業の一部、品質ゲートのみ、Runbook の具体値追記のみ、など |
| **スコープ** | **§6.2 の順番**（例: 順1〜3）または **Runbook の §**（例: §1.4 のみ）で区切る。「全部」より区切りを推奨。 |
| **現状** | Render 本番の有無、本番 DB/Redis、Stripe は test のみか live か、`main`/`develop` の差分の把握状況 |
| **完了条件** | 例: Runbook に本番 URL が埋まった、§6.5 の証跡が `docs/` に残った、など |
| **触ってよい範囲** | コード / `render.yaml` / `docs` のみ、本番ダッシュボードは手元で操作し AI には秘密を渡さない、など |

#### コピー用テンプレート

```text
【目的】サービス本番ローンチ作業のうち、以下を実施したい。

【スコープ】
- 実装計画 `docs/サービス開始までの手順整理_実装計画.md` の §6.2 の順【　】〜順【　】
  （または Runbook `docs/サービス開始までの手順_Runbook.md` の §【　】のみ）

【現状】
- Render 本番：【未作成 / 作成済み。メモ：…】
- 本番 DB・Redis：【未 / あり】
- Stripe：【test のみ / live 準備あり】
- main と develop：【差分未確認 / 概要把握済み】

【やってほしいこと】
- （例：Runbook 追記、環境変数ドラフト、staging での §6.5 再実行と報告ドキュメント化、など）

【完了条件】
- （例：該当チェックリストが実行可能な具体値で埋まっている、など）

【制約】
- Docker 必須・Vercel 禁止・本番 Stripe live の無承認切替は禁止、など
```

#### 短文依頼の例

- 「`docs/サービス開始までの手順整理_実装計画.md` §6.2 **順1** に沿い、本番ブロッカーを一覧し、`docs/サービス開始までの手順_Runbook.md` §1.1 に追記してください（コード変更は不要）。」
- 「**順2** と Runbook §1.4。Docker と**ステージング**で回帰し、§6.5 を証跡付きで埋めてください。秘匿情報はチャットに貼りません。」
- 「**順3** のみ。Render 本番 B/F・DB・Redis・ドメインの設計を1枚にまとめ、Runbook §1.2 に貼れる形にしてください。」
